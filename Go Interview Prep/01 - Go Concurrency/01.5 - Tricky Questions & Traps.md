# Tricky Questions & Traps

Связанные темы: [[01.1 - GMP Model & Scheduler]] | [[01.2 - Channels & Select]] | [[01.3 - Sync Primitives]]

---

> [!question]- 1. Будет ли deadlock? Объясни.
> ```go
> ch := make(chan int, 4)
> go func() {
>     ch <- 1
>     ch <- 2
>     ch <- 3
>     ch <- 4
>     ch <- 5
>     close(ch)
> }()
> for num := range ch {
>     fmt.Println(num)
> }
> ```

> [!warning]- Ловушка!
> Большинство говорят: "Deadlock на 5-м элементе — буфер 4, а элементов 5."
> **Неправильно!**

> [!example]- Правильный ответ
> Программа успешно выведет 1, 2, 3, 4, 5. **Нет deadlock.**
>
> **Почему**: producer и consumer работают в **разных горутинах**.
> Пока producer пишет 5-й элемент, main goroutine уже читает из канала.
> Concurrent consumption не позволяет буферу заполниться полностью.
>
> **Deadlock будет** если убрать `go func()`:
> ```go
> ch := make(chan int, 4)
> ch <- 1
> ch <- 2
> ch <- 3
> ch <- 4
> ch <- 5 // DEADLOCK: та же горутина, некому читать
> ```

> [!question]- 2. Что выведет этот код? Почему?
> ```go
> for i := 0; i < 5; i++ {
>     go func() {
>         fmt.Println(i)
>     }()
> }
> time.Sleep(time.Second)
> ```

> [!warning]- Ловушка: closure capture

> [!example]- Правильный ответ
> Скорее всего выведет: `5 5 5 5 5` (но порядок не гарантирован).
>
> **Причина**: closure захватывает **переменную** `i`, а не её **значение**.
> К моменту выполнения горутин, цикл завершился и `i == 5`.
>
> **Исправление 1: передать как аргумент**
> ```go
> for i := 0; i < 5; i++ {
>     go func(n int) {
>         fmt.Println(n)
>     }(i) // копия i передаётся в горутину
> }
> ```
>
> **Исправление 2 (Go 1.22+): loop variable scoping**
> С Go 1.22 каждая итерация создаёт новую переменную `i`.
> Этот код уже правильно выведет 0-4 (в случайном порядке).

> [!question]- 3. Что произойдёт? Найди баг.
> ```go
> func main() {
>     var wg sync.WaitGroup
>     for i := 0; i < 3; i++ {
>         go func() {
>             wg.Add(1)
>             defer wg.Done()
>             fmt.Println("working")
>         }()
>     }
>     wg.Wait()
>     fmt.Println("all done")
> }
> ```

> [!example]- Ответ
> **Race condition**: `wg.Add(1)` вызывается внутри горутины.
> `main` может дойти до `wg.Wait()` раньше чем все горутины вызовут `Add()`.
> Результат: `Wait()` вернётся когда счётчик ещё 0 → "all done" напечатается раньше "working".
>
> **Правильно:**
> ```go
> for i := 0; i < 3; i++ {
>     wg.Add(1) // ДО запуска горутины
>     go func() {
>         defer wg.Done()
>         fmt.Println("working")
>     }()
> }
> wg.Wait()
> ```

> [!question]- 4. Есть ли goroutine leak в этом коде?
> ```go
> func search(query string) string {
>     results := make(chan string)
>     go func() { results <- searchGoogle(query) }()
>     go func() { results <- searchBing(query) }()
>     go func() { results <- searchDDG(query) }()
>     return <-results // берём первый результат
> }
> ```

> [!example]- Ответ
> **Да, goroutine leak.** Мы читаем только 1 результат, а запустили 3 горутины.
> 2 оставшиеся горутины навсегда заблокированы на `results <- ...` (unbuffered channel).
>
> **Исправления:**
>
> **Вариант 1: buffered channel**
> ```go
> results := make(chan string, 3) // буфер на всех
> ```
>
> **Вариант 2: context cancellation**
> ```go
> func search(ctx context.Context, query string) string {
>     ctx, cancel := context.WithCancel(ctx)
>     defer cancel() // отменит оставшиеся горутины
>     results := make(chan string, 1)
>     search := func(fn func(string) string) {
>         go func() {
>             select {
>             case results <- fn(query):
>             case <-ctx.Done():
>             }
>         }()
>     }
>     search(searchGoogle)
>     search(searchBing)
>     search(searchDDG)
>     return <-results
> }
> ```

> [!question]- 5. Что выведет? Будет ли race condition?
> ```go
> func main() {
>     m := make(map[string]int)
>     var wg sync.WaitGroup
>     for i := 0; i < 10; i++ {
>         wg.Add(1)
>         go func(n int) {
>             defer wg.Done()
>             m[fmt.Sprint(n)] = n
>         }(i)
>     }
>     wg.Wait()
>     fmt.Println(m)
> }
> ```

> [!example]- Ответ
> **Fatal error: concurrent map writes.**
>
> Go map **не** thread-safe. Concurrent write вызывает runtime panic (не просто data race, а crash).
>
> С Go 1.6 runtime детектит concurrent map access и фатально крашит программу.
>
> **Решения:**
> - `sync.Mutex` перед каждым доступом к map
> - `sync.RWMutex` если reads >> writes
> - `sync.Map` для cache-like паттернов

> [!question]- 6. Что произойдёт при запуске?
> ```go
> func main() {
>     ch := make(chan int)
>     close(ch)
>     close(ch)
> }
> ```

> [!example]- Ответ
> **panic: close of closed channel**
>
> Повторное закрытие канала — runtime panic. Go не проверяет "уже закрыт?".
>
> Защита:
> ```go
> type SafeChan struct {
>     ch     chan int
>     closed bool
>     mu     sync.Mutex
> }
>
> func (s *SafeChan) Close() {
>     s.mu.Lock()
>     defer s.mu.Unlock()
>     if !s.closed {
>         close(s.ch)
>         s.closed = true
>     }
> }
> ```
> Или `sync.Once` (см. [[01.2 - Channels & Select]]).

> [!question]- 7. Сколько горутин будет утечка? Объясни каждую.
> ```go
> func main() {
>     ch1 := make(chan int)
>     ch2 := make(chan int)
>
>     go func() { ch1 <- 1 }() // горутина A
>     go func() { ch2 <- 2 }() // горутина B
>     go func() {               // горутина C
>         select {
>         case v := <-ch1:
>             fmt.Println(v)
>         case v := <-ch2:
>             fmt.Println(v)
>         }
>     }()
>
>     time.Sleep(time.Second)
> }
> ```

> [!example]- Ответ
> **1 горутина утечёт.**
>
> Горутина C прочитает из **одного** канала (рандомно из ch1 или ch2).
> Допустим C прочитала из ch1:
> - Горутина A — разблокирована (ch1 прочитан), завершается
> - Горутина C — завершается после select
> - Горутина B — **навсегда заблокирована** на `ch2 <- 2` (unbuffered, нет reader)
>
> Утечка: **1 горутина** (та, чей канал не был прочитан).

> [!question]- 8. Что выведет? Порядок defer и горутин.
> ```go
> func main() {
>     fmt.Println("start")
>     defer fmt.Println("deferred")
>     go fmt.Println("goroutine")
>     fmt.Println("end")
> }
> ```

> [!example]- Ответ
> Гарантированный вывод:
> ```
> start
> end
> deferred
> ```
>
> "goroutine" **может** не напечататься вообще! Когда main завершается, все горутины убиваются.
> `defer` выполняется **до** выхода из main, но горутина может не успеть.
>
> Порядок: normal code → deferred (LIFO) → main exit → все горутины killed.

> [!question]- 9. Этот код безопасен? Объясни.
> ```go
> var done bool
>
> func setup() {
>     go func() {
>         // do initialization
>         done = true
>     }()
> }
>
> func main() {
>     setup()
>     for !done { /* spin wait */ }
>     fmt.Println("initialized")
> }
> ```

> [!example]- Ответ
> **Нет, data race.** `done` читается и пишется без синхронизации.
>
> Более того: компилятор может **оптимизировать** `for !done {}` в бесконечный цикл,
> потому что без happens-before гарантии нет обязательства видеть изменение `done`.
>
> **Правильно:**
> ```go
> var done atomic.Bool
>
> func setup() {
>     go func() {
>         // do initialization
>         done.Store(true)
>     }()
> }
>
> func main() {
>     setup()
>     for !done.Load() {
>         runtime.Gosched() // yield CPU
>     }
>     fmt.Println("initialized")
> }
> ```
> Или лучше — используй канал или `sync.WaitGroup`.

> [!question]- 10. Что произойдёт? Select с nil channel.
> ```go
> func main() {
>     var ch chan int // nil channel
>     select {
>     case v := <-ch:
>         fmt.Println(v)
>     case ch <- 1:
>         fmt.Println("sent")
>     }
> }
> ```

> [!example]- Ответ
> **fatal error: all goroutines are asleep - deadlock!**
>
> Оба case работают с nil channel:
> - `<-ch` на nil — блокирует навсегда
> - `ch <- 1` на nil — блокирует навсегда
>
> Оба case неготовы → select блокируется навсегда → runtime детектит deadlock.
>
> **Если добавить `default`** — select не заблокируется, выполнит default.

> [!question]- 11. Найди проблему: producer-consumer с несколькими producers.
> ```go
> func main() {
>     ch := make(chan int)
>
>     // 3 producers
>     for i := 0; i < 3; i++ {
>         go func(id int) {
>             for j := 0; j < 5; j++ {
>                 ch <- id*10 + j
>             }
>             close(ch) // каждый producer закрывает!
>         }(i)
>     }
>
>     for v := range ch {
>         fmt.Println(v)
>     }
> }
> ```

> [!example]- Ответ
> **panic: close of closed channel** — первый producer закрывает, второй пытается закрыть опять.
> Плюс: **panic: send on closed channel** — оставшиеся producers пишут в закрытый канал.
>
> **Правильно**: один координатор закрывает канал после всех producers:
> ```go
> var wg sync.WaitGroup
> for i := 0; i < 3; i++ {
>     wg.Add(1)
>     go func(id int) {
>         defer wg.Done()
>         for j := 0; j < 5; j++ {
>             ch <- id*10 + j
>         }
>     }(i)
> }
> go func() {
>     wg.Wait()
>     close(ch) // закрывает ОДИН раз после ВСЕХ
> }()
> ```

> [!question]- 12. Почему этот mutex не работает?
> ```go
> type Counter struct {
>     mu    sync.Mutex
>     count int
> }
>
> func (c Counter) Increment() {
>     c.mu.Lock()
>     defer c.mu.Unlock()
>     c.count++
> }
> ```

> [!example]- Ответ
> **Value receiver** копирует struct при каждом вызове. Каждый вызов `Increment()` работает со **своей копией** Mutex и count.
>
> Результат: нет реальной синхронизации, count всегда 0 для оригинала.
>
> **Правильно**: pointer receiver:
> ```go
> func (c *Counter) Increment() {
>     c.mu.Lock()
>     defer c.mu.Unlock()
>     c.count++
> }
> ```
>
> **Правило: если struct содержит Mutex — всегда используй pointer receiver.**
> `go vet` поймает эту ошибку: "passes lock by value".
