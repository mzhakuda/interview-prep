# GMP Model & Scheduler

Связанные темы: [[01.2 - Channels & Select]] | [[01.3 - Sync Primitives]] | [[07.1 - GC & Memory Model]]

---

> [!question]- 1. Объясни GMP модель Go шедулера. Что происходит когда горутина блокируется на syscall?
>
> **G** (Goroutine) — лёгкий поток, управляется Go runtime, начальный стек ~2KB (растёт динамически до 1GB)
> **M** (Machine) — реальный OS поток, выполняет код горутин
> **P** (Processor) — логический процессор, содержит local run queue. Количество = `GOMAXPROCS`
>
> Когда G блокируется на syscall:
> 1. M отсоединяется от P (hand-off)
> 2. P подхватывает другой свободный M (или runtime создаёт новый)
> 3. Заблокированная G остаётся привязана к M (без P)
> 4. После разблокировки G пытается найти свободный P
> 5. Если P нет — G уходит в **global run queue**
>
> Ключевая идея: P никогда не простаивает из-за blocked syscall.

> [!warning]- Частая ошибка на интервью
> Путают "параллелизм" и "конкурентность":
> - **Конкурентность** — структура программы (несколько задач могут прогрессировать)
> - **Параллелизм** — одновременное выполнение (требует GOMAXPROCS > 1 и несколько ядер CPU)
> Go гарантирует конкурентность, параллелизм — только при наличии ресурсов.

> [!example]- Код: проверка runtime параметров
> ```go
> package main
>
> import (
>     "fmt"
>     "runtime"
> )
>
> func main() {
>     fmt.Println("GOMAXPROCS:", runtime.GOMAXPROCS(0)) // текущее значение
>     fmt.Println("NumCPU:", runtime.NumCPU())           // доступные ядра
>     fmt.Println("NumGoroutine:", runtime.NumGoroutine()) // живые горутины
>
>     runtime.GOMAXPROCS(4) // установить 4 логических процессора
> }
> ```

> [!question]- 2. Как работает work stealing в Go шедулере?
>
> Каждый P имеет свою **local run queue** (LRQ, до 256 горутин). Есть также **global run queue** (GRQ).
>
> Когда LRQ пуста, P пытается украсть работу в порядке:
> 1. Проверить **global run queue** (каждый 61-й тик, чтобы не голодать GRQ)
> 2. Проверить **network poller** (готовые I/O горутины)
> 3. **Украсть половину** LRQ у случайного другого P
>
> Это обеспечивает равномерную загрузку без центрального диспетчера.

> [!hint]- Подсказка: зачем каждый 61-й тик?
> Простое число 61 выбрано чтобы избежать резонанса с паттернами доступа.
> Без проверки GRQ горутины там могут голодать бесконечно.

> [!question]- 3. Чем preemptive scheduling в Go 1.14+ отличается от cooperative в более ранних версиях?
>
> **До Go 1.14 (cooperative)**:
> - Горутина уступает управление только на function calls, channel ops, syscalls
> - Tight loop `for { i++ }` блокирует весь P навсегда
> - Другие горутины на этом P голодают
>
> **С Go 1.14 (preemptive)**:
> - Runtime использует **OS signals** (SIGURG на Unix) для прерывания
> - Compiler вставляет **safe points** в код
> - Горутина может быть прервана асинхронно в любом safe point
> - Tight loops больше не блокируют P

> [!example]- Код: демонстрация проблемы до Go 1.14
> ```go
> // До Go 1.14: эта горутина заблокировала бы весь P
> func main() {
>     runtime.GOMAXPROCS(1)
>
>     go func() {
>         // tight loop — нет function calls, нет yield points
>         for {
>             // другие горутины на этом P не выполняются
>         }
>     }()
>
>     // С Go 1.14 — main горутина всё ещё получит время
>     time.Sleep(time.Second)
>     fmt.Println("main done") // До 1.14 — не напечатается
> }
> ```

> [!question]- 4. Что такое spinning threads в Go runtime? Зачем они нужны?
>
> **Spinning M** — OS поток, который активно ищет работу вместо засыпания.
>
> Runtime поддерживает не более `GOMAXPROCS` spinning threads.
> Они быстро подхватывают новые горутины без overhead пробуждения OS потока.
>
> Trade-off: тратят CPU cycles на spinning, но уменьшают latency.
> Эффективно при бурстовой нагрузке, когда горутины появляются часто.

> [!question]- 5. Объясни как GOMAXPROCS влияет на производительность. Какое значение оптимально?
>
> `GOMAXPROCS` задаёт количество P (логических процессоров):
> - **По умолчанию** = `runtime.NumCPU()` (все ядра)
> - Больше P → больше параллелизма, но больше contention на shared ресурсах
> - Меньше P → меньше contention, но меньше throughput
>
> **Рекомендации:**
> - CPU-bound задачи: `GOMAXPROCS = NumCPU` (default)
> - I/O-bound задачи: можно больше, но обычно default достаточно
> - В контейнерах: `runtime.NumCPU()` видит все ядра хоста! Используй `go.uber.org/automaxprocs` для корректного значения в Docker/K8s

> [!warning]- Ловушка с контейнерами
> В Docker/K8s `runtime.NumCPU()` возвращает ядра **хоста**, а не контейнера.
> Если контейнер ограничен 2 CPU, а хост имеет 64 — GOMAXPROCS будет 64.
> Результат: excessive context switching, деградация производительности.
> Решение: `go.uber.org/automaxprocs` автоматически читает cgroup лимиты.

> [!question]- 6. Что происходит при создании новой горутины? Опиши lifecycle полностью.
>
> 1. **Создание**: `go func()` создаёт новый G (Goroutine struct)
> 2. **Аллокация стека**: начальный стек 2KB (с Go 1.4 — growable stacks)
> 3. **Помещение в очередь**: G добавляется в LRQ текущего P
>    - Если LRQ полна — половина перемещается в GRQ
> 4. **Планирование**: когда P свободен, он берёт G из LRQ (или через work stealing)
> 5. **Выполнение**: M выполняет G до завершения, yield, или preemption
> 6. **Завершение**: G возвращается в пул свободных G (переиспользование)
>
> Стоимость создания горутины: ~2-3 микросекунды (vs ~1ms для OS thread).

> [!question]- 7. Как Go runtime решает когда создать новый OS thread (M)?
>
> Runtime создаёт новый M когда:
> 1. Горутина заблокирована на syscall, а все M заняты — нужен новый для P
> 2. P есть свободный, но нет idle M для него
>
> **Лимит**: по умолчанию до 10000 M (`runtime/debug.SetMaxThreads()`)
>
> M **не** уничтожаются после использования — уходят в idle pool.
> Это означает что программа с большим количеством блокирующих syscalls может накопить тысячи OS потоков.

> [!hint]- Подсказка: как отследить?
> `runtime.NumGoroutine()` — количество горутин
> `runtime/pprof` → threadcreate profile — количество созданных OS потоков

> [!question]- 8. Объясни netpoller в Go. Как он интегрирован с шедулером?
>
> **Netpoller** — runtime компонент для асинхронного I/O:
> - Использует `epoll` (Linux), `kqueue` (macOS), `IOCP` (Windows)
> - Горутина вызывает `net.Read()` → runtime не блокирует M
> - Вместо этого: G паркуется, M продолжает выполнять другие G
> - Когда I/O готов: netpoller будит G, ставит обратно в run queue
>
> Это ключевое отличие от blocking syscalls: network I/O **не** отсоединяет M от P.
> Поэтому Go обрабатывает тысячи concurrent connections без тысяч OS потоков.

> [!warning]- Важное отличие
> **Network I/O** → netpoller → M не отсоединяется от P (эффективно)
> **File I/O / CGO** → blocking syscall → M отсоединяется от P (дорого)
> Это почему file I/O в Go не так эффективен как network I/O.

> [!question]- 9. Что такое goroutine-local storage? Почему в Go нет thread-local storage?
>
> Go **намеренно** не имеет TLS/goroutine-local storage:
> - Горутины дешёвые → их тысячи/миллионы → TLS не масштабируется
> - Encourage explicit passing через аргументы и `context.Context`
> - TLS создаёт неявные зависимости, усложняет тестирование
>
> Если нужен request-scoped state: используй `context.WithValue()`.
> Если нужен global mutable state: используй `sync.Map` или `sync.Mutex`.

> [!question]- 10. Как отладить проблемы с Go шедулером? Какие инструменты есть?
>
> **Envvar `GODEBUG=schedtrace=1000`** — печатает состояние шедулера каждые 1000ms:
> ```
> SCHED 1000ms: gomaxprocs=4 idleprocs=2 threads=6 spinningthreads=1 idlethreads=1 runqueue=0 [3 0 1 0]
> ```
> - `idleprocs`: P без работы
> - `runqueue`: длина global queue
> - `[3 0 1 0]`: длины local queues каждого P
>
> **`GODEBUG=scheddetail=1`** — детальный дамп каждой G, M, P
>
> **`runtime/trace`** — полная трейс шедулера (go tool trace):
> ```go
> f, _ := os.Create("trace.out")
> trace.Start(f)
> defer trace.Stop()
> ```
>
> **pprof goroutine profile** — стейт всех горутин:
> ```go
> import _ "net/http/pprof"
> go http.ListenAndServe(":6060", nil)
> // GET /debug/pprof/goroutine?debug=2
> ```
