# Channels & Select

Связанные темы: [[01.1 - GMP Model & Scheduler]] | [[01.3 - Sync Primitives]] | [[01.4 - Patterns (WorkerPool, FanIn, Pipeline)]]

---

> [!question]- 1. Как устроен channel внутри? Опиши внутреннюю структуру hchan.
>
> Channel — это `runtime.hchan` struct:
> ```
> type hchan struct {
>     qcount   uint           // кол-во элементов в буфере
>     dataqsiz uint           // размер буфера (capacity)
>     buf      unsafe.Pointer // ring buffer для данных
>     elemsize uint16         // размер одного элемента
>     closed   uint32         // флаг: канал закрыт
>     sendx    uint           // индекс записи в ring buffer
>     recvx    uint           // индекс чтения из ring buffer
>     recvq    waitq          // очередь заблокированных receivers
>     sendq    waitq          // очередь заблокированных senders
>     lock     mutex          // мьютекс для защиты всей структуры
> }
> ```
>
> **Ключевое**: каждая операция с каналом захватывает lock. Канал — thread-safe by design.
> Ring buffer используется для buffered channels. Unbuffered — buf = nil.

> [!question]- 2. Разница buffered vs unbuffered channel. Когда использовать какой?
>
> **Unbuffered** (`make(chan int)`):
> - Sender блокируется пока receiver не готов (и наоборот)
> - Синхронная передача — **гарантия доставки**
> - Используй: для сигнализации, synchronization points
>
> **Buffered** (`make(chan int, N)`):
> - Sender блокируется только когда буфер полный
> - Receiver блокируется только когда буфер пустой
> - Используй: для decoupling producer/consumer, burst absorption
>
> **Правило**: если не знаешь какой — начни с unbuffered. Buffered добавляй только когда нужна производительность.

> [!warning]- Частая ошибка
> Buffered channel **не** гарантирует что receiver получил значение.
> `ch <- value` возвращается сразу если есть место в буфере — receiver может ещё не прочитать.
> Для гарантии доставки: unbuffered или acknowledgment channel.

> [!question]- 3. Что произойдёт при операциях с nil channel?
>
> | Операция | nil channel | closed channel |
> |----------|-------------|----------------|
> | `ch <- v` | **блокирует навсегда** | **panic** |
> | `<-ch` | **блокирует навсегда** | возвращает zero value |
> | `close(ch)` | **panic** | **panic** |
> | `v, ok := <-ch` | блокирует навсегда | zero value, `ok=false` |
>
> **Полезность nil channel**: в `select` nil case никогда не срабатывает.
> Это позволяет динамически включать/отключать ветки select.

> [!example]- Код: отключение ветки через nil
> ```go
> func merge(ch1, ch2 <-chan int) <-chan int {
>     out := make(chan int)
>     go func() {
>         defer close(out)
>         for ch1 != nil || ch2 != nil {
>             select {
>             case v, ok := <-ch1:
>                 if !ok { ch1 = nil; continue }
>                 out <- v
>             case v, ok := <-ch2:
>                 if !ok { ch2 = nil; continue }
>                 out <- v
>             }
>         }
>     }()
>     return out
> }
> ```

> [!question]- 4. Как работает select? Что если несколько case готовы одновременно?
>
> `select` блокируется пока хотя бы один case не будет готов.
>
> **Если несколько case готовы**: Go выбирает **случайный** (uniform random).
> Это не баг — это дизайн: предотвращает starvation.
>
> **default case**: делает select неблокирующим.
>
> **Empty select** `select {}` — блокирует навсегда (полезно для long-running сервисов).

> [!example]- Код: non-blocking send
> ```go
> select {
> case ch <- value:
>     // отправлено
> default:
>     // канал полный или нет receiver — drop value
>     log.Println("message dropped")
> }
> ```

> [!question]- 5. Объясни directional channels. Зачем нужны `chan<-` и `<-chan`?
>
> **Send-only** `chan<- int` — можно только записывать
> **Receive-only** `<-chan int` — можно только читать
>
> Используются в **сигнатурах функций** для type safety:
> ```go
> func producer(out chan<- int) { out <- 42 }
> func consumer(in <-chan int)  { v := <-in }
> ```
>
> Bi-directional `chan int` автоматически конвертируется в directional при передаче.
> Обратная конверсия запрещена — это **compile-time** гарантия.

> [!question]- 6. Когда и как правильно закрывать каналы? Кто должен закрывать?
>
> **Правило: закрывает только sender, никогда receiver.**
>
> Закрытие сигнализирует receivers что больше данных не будет.
> `for range ch` автоматически завершается при закрытии.
>
> **Опасности:**
> - Send на closed channel → **panic**
> - Двойной close → **panic**
> - Close nil channel → **panic**
>
> **Паттерн с несколькими senders:**
> Используй `sync.Once` для безопасного однократного закрытия или отдельный done channel.

> [!example]- Код: безопасное закрытие с sync.Once
> ```go
> type SafeChannel struct {
>     ch   chan int
>     once sync.Once
> }
>
> func (sc *SafeChannel) Close() {
>     sc.once.Do(func() { close(sc.ch) })
> }
> ```

> [!question]- 7. Как реализовать timeout для channel операции?
>
> **Способ 1: time.After**
> ```go
> select {
> case v := <-ch:
>     process(v)
> case <-time.After(5 * time.Second):
>     log.Println("timeout")
> }
> ```
>
> **Способ 2: context.WithTimeout** (предпочтительнее)
> ```go
> ctx, cancel := context.WithTimeout(ctx, 5*time.Second)
> defer cancel()
> select {
> case v := <-ch:
>     process(v)
> case <-ctx.Done():
>     log.Println("timeout:", ctx.Err())
> }
> ```

> [!warning]- Утечка с time.After в цикле
> ```go
> // ПЛОХО: time.After создаёт таймер на каждую итерацию — memory leak
> for {
>     select {
>     case v := <-ch:
>         process(v)
>     case <-time.After(time.Second):
>         // каждая итерация аллоцирует новый timer
>     }
> }
>
> // ПРАВИЛЬНО: переиспользуй timer
> timer := time.NewTimer(time.Second)
> defer timer.Stop()
> for {
>     timer.Reset(time.Second)
>     select {
>     case v := <-ch:
>         process(v)
>     case <-timer.C:
>         // ok
>     }
> }
> ```

> [!question]- 8. Что такое channel axioms? Перечисли все.
>
> 1. Send на **nil** channel — блокирует навсегда
> 2. Receive из **nil** channel — блокирует навсегда
> 3. Send на **closed** channel — panic
> 4. Receive из **closed** channel — возвращает zero value (немедленно)
> 5. Close **nil** channel — panic
> 6. Close уже **closed** channel — panic
>
> Мнемоника: "nil блокирует, closed паникует на send".

> [!question]- 9. Как определить размер буфера для buffered channel?
>
> Нет универсальной формулы. Руководство:
> - **0 (unbuffered)**: synchronization, handoff
> - **1**: простой сигнал, semaphore
> - **N (known)**: batch processing, bounded work queue
> - **N (estimated)**: `producer_rate × acceptable_latency`
>
> **Антипаттерн**: огромный буфер "на всякий случай" — маскирует проблемы с backpressure.
> Если буфер регулярно заполняется — проблема не в размере буфера, а в consumer.

> [!question]- 10. Как сделать fan-out/fan-in через channels? Набросай архитектуру.
>
> **Fan-out**: одна горутина → несколько workers
> ```go
> jobs := make(chan Job)
> for i := 0; i < numWorkers; i++ {
>     go worker(jobs) // все читают из одного канала
> }
> ```
>
> **Fan-in**: несколько каналов → один выход
> ```go
> func fanIn(channels ...<-chan int) <-chan int {
>     out := make(chan int)
>     var wg sync.WaitGroup
>     for _, ch := range channels {
>         wg.Add(1)
>         go func(c <-chan int) {
>             defer wg.Done()
>             for v := range c { out <- v }
>         }(ch)
>     }
>     go func() { wg.Wait(); close(out) }()
>     return out
> }
> ```

> [!hint]- Подсказка для интервью
> Интервьюер ожидает что ты упомянешь `sync.WaitGroup` для координации закрытия output channel.
> Без WaitGroup — либо goroutine leak, либо panic при close.
