# Sync Primitives

Связанные темы: [[01.1 - GMP Model & Scheduler]] | [[01.2 - Channels & Select]] | [[01.4 - Patterns (WorkerPool, FanIn, Pipeline)]]

---

> [!question]- 1. Когда использовать channel, а когда mutex? Дай чёткие критерии.
>
> **Channels** — для передачи ownership данных между горутинами:
> - Pipeline обработки данных
> - Fan-out/fan-in
> - Сигнализация (done, cancel)
> - Когда данные "путешествуют" от producer к consumer
>
> **Mutex** — для защиты shared state (in-place):
> - Cache / map с concurrent access
> - Счётчики, метрики
> - Конфигурация, которую читают многие
> - Когда данные "сидят на месте" и к ним обращаются многие
>
> **Правило Rob Pike**: "Don't communicate by sharing memory; share memory by communicating."
> Но это не абсолют — для простого shared state mutex проще и быстрее.

> [!question]- 2. sync.Mutex vs sync.RWMutex — когда что использовать?
>
> **`sync.Mutex`**: exclusive lock. Один goroutine читает ИЛИ пишет.
>
> **`sync.RWMutex`**: multiple readers OR one writer.
> - `RLock()` / `RUnlock()` — shared read lock
> - `Lock()` / `Unlock()` — exclusive write lock
>
> **Используй RWMutex когда:**
> - Read операций значительно больше чем write
> - Read не мутирует данные
> - Reads длительные (justify overhead)
>
> **Не используй RWMutex когда:**
> - Write частые — RWMutex дороже простого Mutex из-за overhead координации
> - Критическая секция очень короткая — overhead RWMutex не окупается

> [!warning]- Ловушка: recursive locking
> ```go
> var mu sync.Mutex
> mu.Lock()
> mu.Lock() // DEADLOCK! Go mutex не reentrant
> ```
> Go мьютексы **не рекурсивные** (не reentrant). Это by design — recursive locking часто признак плохой архитектуры.

> [!question]- 3. Как работает sync.WaitGroup? Какие ошибки часто допускают?
>
> `WaitGroup` — счётчик ожидания завершения N горутин:
> - `Add(n)` — увеличить счётчик
> - `Done()` — уменьшить на 1 (алиас `Add(-1)`)
> - `Wait()` — блокировать пока счётчик не станет 0
>
> **Типичные ошибки:**
>
> 1. **`Add()` внутри горутины** — race condition:
> ```go
> // ПЛОХО: main может дойти до Wait() до Add()
> for i := 0; i < 10; i++ {
>     go func() {
>         wg.Add(1) // race!
>         defer wg.Done()
>     }()
> }
> wg.Wait()
>
> // ПРАВИЛЬНО: Add() до запуска горутины
> for i := 0; i < 10; i++ {
>     wg.Add(1)
>     go func() {
>         defer wg.Done()
>     }()
> }
> wg.Wait()
> ```
>
> 2. **Забыть `Done()`** → `Wait()` висит навсегда. Используй `defer wg.Done()`.
>
> 3. **Переиспользование до завершения Wait()** → undefined behavior.

> [!question]- 4. Объясни sync.Once. Где используется в реальных проектах?
>
> `sync.Once` гарантирует что функция вызовется **ровно один раз**, даже из множества горутин.
>
> ```go
> var once sync.Once
> var instance *Config
>
> func GetConfig() *Config {
>     once.Do(func() {
>         instance = loadConfig() // вызовется ровно 1 раз
>     })
>     return instance
> }
> ```
>
> **Реальные use cases:**
> - Singleton (lazy initialization)
> - Database connection pool init
> - One-time resource loading
> - Graceful shutdown signal
>
> **Важно**: если `Do()` паникует — считается выполненным. Повторный вызов `Do()` **не** перезапустит функцию.

> [!question]- 5. sync/atomic vs sync.Mutex — когда что?
>
> **`sync/atomic`** — lock-free операции для простых типов:
> - `atomic.AddInt64(&counter, 1)` — атомарный инкремент
> - `atomic.LoadInt64(&val)` / `atomic.StoreInt64(&val, n)`
> - `atomic.CompareAndSwapInt64(&val, old, new)` (CAS)
> - Go 1.19+: `atomic.Int64{}` — удобная обёртка
>
> **Когда atomic:**
> - Простой счётчик, флаг
> - Одна переменная, не связанная с другими
> - Maximum performance critical path
>
> **Когда Mutex:**
> - Несколько связанных переменных (invariants)
> - Сложная логика в критической секции
> - Map, slice, struct с несколькими полями

> [!example]- Код: atomic vs mutex бенчмарк
> ```go
> // Atomic — ~5ns per operation
> var counter atomic.Int64
> counter.Add(1)
>
> // Mutex — ~20-30ns per operation
> var mu sync.Mutex
> var count int64
> mu.Lock()
> count++
> mu.Unlock()
> ```

> [!question]- 6. Что такое sync.Map и когда его использовать вместо обычной map + mutex?
>
> `sync.Map` — concurrent map без external locking.
>
> **Оптимизирован для двух паттернов:**
> 1. Key написан один раз, читается много раз (cache)
> 2. Множество горутин работают с disjoint наборами ключей
>
> **Не используй sync.Map когда:**
> - Частые записи по одним и тем же ключам
> - Нужен `len()` — sync.Map не поддерживает (нужен `Range()`)
> - Нужна типобезопасность (sync.Map использует `any`)
>
> **Обычная map + RWMutex** лучше в большинстве случаев.
> sync.Map — для специфических hot-path оптимизаций.

> [!question]- 7. Объясни sync.Pool. Как он взаимодействует с GC?
>
> `sync.Pool` — пул переиспользуемых объектов для снижения GC pressure:
>
> ```go
> var bufPool = sync.Pool{
>     New: func() any { return make([]byte, 4096) },
> }
>
> buf := bufPool.Get().([]byte) // получить из пула или создать
> defer bufPool.Put(buf)        // вернуть в пул
> ```
>
> **Взаимодействие с GC:**
> - Pool очищается при **каждом GC цикле** (victim cache с Go 1.13)
> - Объекты в Pool **не** считаются live — GC может их собрать
> - Не храни в Pool объекты которые нельзя потерять
>
> **Use cases:** буферы, временные объекты, encoder/decoder structs
> **Антипаттерн:** Pool для long-lived объектов (connections — используй connection pool)

> [!question]- 8. Что такое sync.Cond? Когда нужен и когда нет?
>
> `sync.Cond` — conditional variable для ожидания условия:
>
> ```go
> cond := sync.NewCond(&sync.Mutex{})
>
> // Waiter
> cond.L.Lock()
> for !condition() {
>     cond.Wait() // atomically: unlock + sleep, then re-lock
> }
> // condition is true
> cond.L.Unlock()
>
> // Signaler
> cond.L.Lock()
> setCondition(true)
> cond.Signal()    // разбудить одного
> // cond.Broadcast() // разбудить всех
> cond.L.Unlock()
> ```
>
> **Когда нужен:** множество горутин ждут одного условия (broadcast)
> **Когда не нужен:** в 99% случаев channel проще и идиоматичнее
>
> `sync.Cond` — low-level примитив, редко используется в application code.

> [!question]- 9. Как обнаружить data race в Go? Что делает race detector?
>
> **Race detector**: `go run -race ./...` или `go test -race ./...`
>
> Работает на основе **ThreadSanitizer** (TSan):
> - Инструментирует каждый memory access
> - Отслеживает happens-before отношения
> - Обнаруживает unsynchronized concurrent access
>
> **Overhead**: ~5-10x CPU, ~5-10x RAM — только для dev/test.
>
> **Что выводит:**
> ```
> WARNING: DATA RACE
> Read at 0x00c000014088 by goroutine 7:
>   main.increment()
> Previous write at 0x00c000014088 by goroutine 6:
>   main.increment()
> Goroutine 7 (running) created at:
>   main.main()
> ```
>
> **Best practice:** запускай `-race` в CI на каждый PR.

> [!question]- 10. Что такое happens-before в Go memory model?
>
> **Happens-before** — гарантия видимости записи одной горутиной для чтения другой.
>
> Go memory model определяет happens-before для:
> 1. **Goroutine creation**: `go f()` happens-before начало `f()`
> 2. **Channel send**: send happens-before receive завершается
> 3. **Channel close**: close happens-before receive возвращает zero
> 4. **Mutex unlock**: `Unlock()` happens-before следующий `Lock()`
> 5. **sync.Once**: `Do(f)` happens-before return of any `Do()`
>
> Без happens-before гарантии: compiler и CPU могут reorder operations.
> Это значит что `x = 1; done = true` в одной горутине **не гарантирует** что другая увидит `x = 1` когда увидит `done = true` — без синхронизации.

> [!question]- 11. Что такое `context.Context`? Зачем нужен?
>
> `context.Context` — интерфейс для передачи трёх вещей по цепочке вызовов:
> 1. **Cancellation** — сигнал отмены (parent отменяет → все children отменяются)
> 2. **Deadline/Timeout** — автоматическая отмена по времени
> 3. **Values** — request-scoped данные (trace ID, user ID)
>
> ```go
> // Отмена вручную
> ctx, cancel := context.WithCancel(context.Background())
> defer cancel() // всегда cancel() чтобы не утечь горутины
>
> // Таймаут
> ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
> defer cancel()
>
> // Проверка отмены в горутине
> select {
> case <-ctx.Done():
>     return ctx.Err() // context.Canceled или context.DeadlineExceeded
> default:
>     // продолжаем работу
> }
> ```
>
> **Правила:**
> - Context — **первый аргумент** функции: `func DoWork(ctx context.Context, ...)`
> - Не хранить в struct — передавать через аргументы
> - Никогда не передавать `nil` — используй `context.Background()` или `context.TODO()`
> - Отмена распространяется **только вниз** по дереву: parent → children

> [!question]- 12. Что хранить в context? Ограничения и best practices.
>
> **`context.WithValue`** создаёт новый контекст с key-value парой.
>
> **Что МОЖНО хранить (request-scoped):**
> - Request/Trace ID: `trace_id`, `request_id`
> - Auth данные уже извлечённые из токена: `user_id`, `roles`
> - Locale / tenant ID
>
> **Что НЕ хранить:**
> - Опциональные параметры функции → используй явные аргументы
> - DB connections, HTTP clients → dependency injection
> - Большие объекты / mutable state — context не для тяжёлых данных
>
> **Typed key pattern (защита от коллизий между пакетами):**
> ```go
> type contextKey string
> const userIDKey contextKey = "user_id"
>
> // Set
> ctx = context.WithValue(ctx, userIDKey, "u-123")
>
> // Get
> if id, ok := ctx.Value(userIDKey).(string); ok {
>     // использовать id
> }
> ```
>
> **Правило:** если данные нужны для корректной работы функции → аргумент. Если "декоративные" (трейсинг, логирование) → context.

> [!question]- 13. Какова временная сложность `context.Value()`? Почему?
>
> **O(n)** — линейный поиск по цепочке контекстов.
>
> **Почему:** каждый `context.WithValue()` создаёт новый узел поверх родителя:
> ```
> Background → WithValue(k1,v1) → WithValue(k2,v2) → WithValue(k3,v3) → ctx
> ```
>
> При `ctx.Value(k1)`:
> 1. Текущий узел: k3 ≠ k1 → parent.Value(k1)
> 2. Следующий узел: k2 ≠ k1 → parent.Value(k1)
> 3. Следующий узел: k1 == k1 → возвращаем v1
>
> **Практические следствия:**
> - Не вызывай `ctx.Value()` в hot loop — используй локальную переменную
> - Не добавляй много значений через WithValue — каждый вызов = heap аллокация
> - Для performance-critical кода: извлеки один раз в начале функции
>
> ```go
> // Плохо: O(n) на каждой итерации
> for _, item := range items {
>     userID := ctx.Value(userIDKey).(string)
>     process(userID, item)
> }
>
> // Хорошо: O(n) один раз
> userID := ctx.Value(userIDKey).(string)
> for _, item := range items {
>     process(userID, item)
> }
> ```

> [!hint]- Почему не map внутри context?
> Map потребует synchronization. Context immutable by design → linked list дешевле для малого числа значений. К тому же linked list позволяет derivation tree: каждый дочерний context наследует родительские значения без копирования.

> [!question]- 14. Что такое cache contention и false sharing? Как влияют на Go concurrency?
>
> **Cache contention** — множество ядер одновременно пишут в данные в одной **CPU cache line** (64 байта). Каждая запись инвалидирует линию у остальных ядер → перечитывание из L3/RAM → throughput падает.
>
> ```go
> // Contention: все ядра бьются за одну cache line
> var counter int64
> // 100 горутин делают atomic.AddInt64(&counter, 1) → медленно из-за cache bouncing
> ```
>
> **False sharing** — два *независимых* поля попали в одну cache line. Писатели мешают друг другу, хотя данные логически не связаны:
>
> ```go
> // Плохо — a и b рядом в памяти → одна cache line
> type Bad struct {
>     a int64  // горутина 1 пишет
>     b int64  // горутина 2 пишет → инвалидирует cache line горутины 1
> }
>
> // Fix — padding разносит поля на разные cache lines
> type Good struct {
>     a int64
>     _ [56]byte  // padding до 64 байт (размер cache line)
>     b int64
> }
> ```
>
> **Решения:**
> - **Sharding**: вместо одного counter — массив per-P счётчиков, суммировать при чтении
> - **Padding**: разносить hot fields на разные cache lines
> - **Reduce write frequency**: батчить обновления
>
> **Как обнаружить:** `perf stat` (Linux) — смотреть `cache-misses`, `cache-references`. В Go: benchmark с разным GOMAXPROCS — если скалирование плохое при увеличении P → вероятно contention.

> [!question]- 15. Почему в Go нет reentrant mutex? Что такое "отсутствие владельца"?
>
> Go `sync.Mutex` **не отслеживает, какая горутина его захватила** (нет owner tracking):
>
> - **Любая горутина** может вызвать `Unlock()`, не только та, что делала `Lock()`
> - **Нет reentrant/recursive lock** — повторный `Lock()` из той же горутины → **deadlock**
> - **Нет `IsLockedByMe()`** — нельзя проверить "мой ли это лок"
>
> ```go
> var mu sync.Mutex
>
> func A() {
>     mu.Lock()
>     B()         // DEADLOCK — B тоже хочет mu.Lock()
>     mu.Unlock()
> }
>
> func B() {
>     mu.Lock()   // навсегда заблокирована
>     defer mu.Unlock()
> }
> ```
>
> **Fix — выделить inner-функцию без лока:**
> ```go
> func A() {
>     mu.Lock()
>     doB()       // внутренняя логика без Lock
>     mu.Unlock()
> }
>
> func B() {
>     mu.Lock()
>     doB()
>     mu.Unlock()
> }
>
> func doB() { /* логика без лока */ }
> ```
>
> **Почему так сделано (Russ Cox):** reentrant mutex скрывает проблемы дизайна — если функция вызывается и с локом, и без → инварианты не защищены. Go предпочитает явное разделение locked/unlocked кода.

> [!question]- 16. Что такое singleflight? Когда и как использовать?
>
> **Проблема:** cache miss по ключу → 1000 горутин одновременно идут в БД за одним и тем же → **thundering herd / cache stampede**.
>
> **Решение:** `golang.org/x/sync/singleflight` — выполняет **один запрос**, остальные ждут его результат.
>
> ```go
> import "golang.org/x/sync/singleflight"
>
> var group singleflight.Group
>
> func GetUser(id string) (*User, error) {
>     v, err, shared := group.Do("user:"+id, func() (any, error) {
>         return db.QueryUser(id)  // вызовется ОДИН раз
>     })
>     // shared=true → результат разделён между callers
>     _ = shared
>     return v.(*User), err
> }
> ```
>
> **Как работает внутри:**
> 1. `group.Do(key, fn)` — проверяет `map[key]*call`
> 2. Если ключ есть → горутина подписывается на `call.wg.Wait()` и ждёт
> 3. Если ключа нет → создаёт `call`, запускает `fn()`, по завершении `wg.Done()`
> 4. Все ожидающие получают тот же `(result, err)`
> 5. Ключ удаляется из map → следующий вызов запустит `fn()` заново
>
> **API:**
> - `group.Do(key, fn)` — блокирующий, все ждут результат
> - `group.DoChan(key, fn)` — возвращает `<-chan Result`, можно `select` с timeout
> - `group.Forget(key)` — принудительно сбросить (следующий вызов не будет ждать текущий)
>
> **Ловушка:** если `fn` возвращает ошибку → **все** ожидающие получат эту ошибку. Для retry-логики нужно `Forget()`:
> ```go
> v, err, _ := group.Do(key, func() (any, error) {
>     result, err := fetchFromDB()
>     if err != nil {
>         group.Forget(key)  // следующий вызов попробует заново
>     }
>     return result, err
> })
> ```
>
> **Где применяется:** кэш (защита от stampede), API gateway (дедупликация), DNS resolver, config reload
