# Patterns (WorkerPool, FanIn, Pipeline)

Связанные темы: [[01.2 - Channels & Select]] | [[01.3 - Sync Primitives]] | [[06.1 - Concurrency Tasks]]

> [!question]- 1. Реализуй паттерн Generator. Объясни зачем возвращать `<-chan`.
>
> **Generator** — функция, которая создаёт горутину и возвращает read-only channel:
>
> ```go
> func generate(nums ...int) <-chan int {
>     out := make(chan int)
>     go func() {
>         defer close(out)
>         for _, n := range nums {
>             out <- n
>         }
>     }()
>     return out
> }
> ```
>
> **Зачем `<-chan`**: caller не может случайно записать или закрыть канал.
> Generator владеет каналом и ответственен за его закрытие.
>
> **Проблема**: если caller перестал читать — горутина повисает навсегда.
> **Решение**: принимай `ctx context.Context` и слушай `ctx.Done()`.

> [!question]- 2. Реализуй Worker Pool с graceful shutdown через context.
>
> ```go
> func WorkerPool(ctx context.Context, numWorkers int, jobs <-chan Job) <-chan Result {
>     results := make(chan Result, numWorkers)
>     var wg sync.WaitGroup
>
>     for i := 0; i < numWorkers; i++ {
>         wg.Add(1)
>         go func(id int) {
>             defer wg.Done()
>             for {
>                 select {
>                 case job, ok := <-jobs:
>                     if !ok {
>                         return // канал закрыт
>                     }
>                     results <- process(job)
>                 case <-ctx.Done():
>                     return // graceful shutdown
>                 }
>             }
>         }(i)
>     }
>
>     go func() {
>         wg.Wait()
>         close(results)
>     }()
>
>     return results
> }
> ```

> [!warning]- Типичные ошибки в Worker Pool
> 1. **`wg.Add(1)` после `go func()`** — race condition, `Wait()` может вернуться раньше
> 2. **`close(results)` до `wg.Wait()`** — panic: send on closed channel
> 3. **Не слушать `ctx.Done()`** — утечка горутин при shutdown
> 4. **Unbounded goroutines** — без pool, `for job := range jobs { go process(job) }` создаёт неограниченное количество горутин

> [!question]- 3. Объясни Fan-out / Fan-in. Когда это полезно?
>
> **Fan-out**: раздаём работу нескольким горутинам для параллельной обработки
> - Полезно когда: CPU-bound обработка, множественные I/O requests
>
> **Fan-in**: объединяем результаты нескольких горутин в один канал
> - Полезно когда: нужно собрать результаты из параллельных обработчиков
>
> ```go
> // Fan-out: запускаем N обработчиков
> workers := make([]<-chan Result, numWorkers)
> for i := 0; i < numWorkers; i++ {
>     workers[i] = processAsync(jobs) // каждый возвращает свой канал
> }
>
> // Fan-in: сливаем все каналы в один
> merged := fanIn(workers...)
> ```
>
> **Реальный пример**: HTTP scraper — fan-out по URL, fan-in результатов.

> [!question]- 4. Реализуй Pipeline из 3 стадий. Как обеспечить backpressure?
>
> **Pipeline** — цепочка стадий, каждая читает из предыдущего канала:
>
> ```go
> func pipeline(ctx context.Context, input <-chan int) <-chan string {
>     // Stage 1: filter
>     filtered := filter(ctx, input, func(n int) bool { return n%2 == 0 })
>
>     // Stage 2: transform
>     transformed := transform(ctx, filtered, func(n int) int { return n * n })
>
>     // Stage 3: format
>     formatted := format(ctx, transformed, func(n int) string {
>         return fmt.Sprintf("result: %d", n)
>     })
>
>     return formatted
> }
>
> func filter(ctx context.Context, in <-chan int, fn func(int) bool) <-chan int {
>     out := make(chan int)
>     go func() {
>         defer close(out)
>         for v := range in {
>             select {
>             case <-ctx.Done():
>                 return
>             default:
>                 if fn(v) { out <- v }
>             }
>         }
>     }()
>     return out
> }
> ```
>
> **Backpressure обеспечивается автоматически**: unbuffered channels блокируют producer когда consumer медленный. Добавь buffered channels для decoupling.

> [!question]- 5. Что такое Semaphore pattern в Go? Реализуй через channel.
>
> **Semaphore** ограничивает количество одновременных операций:
>
> ```go
> type Semaphore struct {
>     ch chan struct{}
> }
>
> func NewSemaphore(max int) *Semaphore {
>     return &Semaphore{ch: make(chan struct{}, max)}
> }
>
> func (s *Semaphore) Acquire(ctx context.Context) error {
>     select {
>     case s.ch <- struct{}{}:
>         return nil
>     case <-ctx.Done():
>         return ctx.Err()
>     }
> }
>
> func (s *Semaphore) Release() {
>     <-s.ch
> }
> ```
>
> **Альтернатива**: `golang.org/x/sync/semaphore` — weighted semaphore.
> **Use case**: ограничить concurrent HTTP requests, DB connections, file handles.

> [!question]- 6. Реализуй Rate Limiter через token bucket.
>
> ```go
> func rateLimiter(ctx context.Context, rate time.Duration) <-chan struct{} {
>     tokens := make(chan struct{}, 1)
>     go func() {
>         defer close(tokens)
>         ticker := time.NewTicker(rate)
>         defer ticker.Stop()
>         for {
>             select {
>             case <-ticker.C:
>                 select {
>                 case tokens <- struct{}{}:
>                 default: // bucket full, drop token
>                 }
>             case <-ctx.Done():
>                 return
>             }
>         }
>     }()
>     return tokens
> }
>
> // Использование:
> limiter := rateLimiter(ctx, 100*time.Millisecond) // 10 req/sec
> for req := range requests {
>     <-limiter // ждём токен
>     process(req)
> }
> ```
>
> В production: `golang.org/x/time/rate` (token bucket с burst).

> [!question]- 7. Как реализовать Or-Done channel? Зачем он нужен?
>
> **Or-Done** — обёртка для чтения из канала с поддержкой cancellation:
>
> ```go
> func orDone(ctx context.Context, ch <-chan int) <-chan int {
>     out := make(chan int)
>     go func() {
>         defer close(out)
>         for {
>             select {
>             case <-ctx.Done():
>                 return
>             case v, ok := <-ch:
>                 if !ok { return }
>                 select {
>                 case out <- v:
>                 case <-ctx.Done():
>                     return
>                 }
>             }
>         }
>     }()
>     return out
> }
> ```
>
> **Зачем**: без orDone, `for v := range ch` не реагирует на context cancellation.
> С orDone: `for v := range orDone(ctx, ch)` — clean shutdown.

> [!question]- 8. Объясни паттерн errgroup. Чем лучше голого WaitGroup?
>
> `golang.org/x/sync/errgroup` — WaitGroup + error propagation + context cancellation:
>
> ```go
> g, ctx := errgroup.WithContext(ctx)
>
> for _, url := range urls {
>     url := url // capture loop variable
>     g.Go(func() error {
>         resp, err := http.Get(url)
>         if err != nil { return err }
>         defer resp.Body.Close()
>         return processResponse(resp)
>     })
> }
>
> if err := g.Wait(); err != nil {
>     log.Fatal("failed:", err)
> }
> ```
>
> **Преимущества перед WaitGroup:**
> - Автоматическая propagation первой ошибки
> - Cancellation context при ошибке (все остальные горутины получают signal)
> - `SetLimit(n)` — встроенный semaphore для ограничения concurrency
> - Не нужен ручной `Add()/Done()`

> [!question]- 9. Как реализовать graceful shutdown сервера?
>
> ```go
> func main() {
>     srv := &http.Server{Addr: ":8080", Handler: mux}
>
>     // Запуск в горутине
>     go func() {
>         if err := srv.ListenAndServe(); err != http.ErrServerClosed {
>             log.Fatal(err)
>         }
>     }()
>
>     // Ожидание сигнала
>     quit := make(chan os.Signal, 1)
>     signal.Notify(quit, syscall.SIGINT, syscall.SIGTERM)
>     <-quit
>
>     // Graceful shutdown с таймаутом
>     ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
>     defer cancel()
>     if err := srv.Shutdown(ctx); err != nil {
>         log.Fatal("forced shutdown:", err)
>     }
>     log.Println("server stopped")
> }
> ```

> [!hint]- Что спросит интервьюер дальше
> "А как ты shutdown горутины которые обрабатывают background jobs?"
> Ответ: передавай context от main через всю цепочку. Каждая горутина слушает `ctx.Done()`.

> [!question]- 10. Сравни sync.Mutex, channel и atomic для разных задач.
>
> | Задача | Лучший инструмент | Почему |
> |--------|-------------------|--------|
> | Простой счётчик | `atomic.Int64` | Fastest, lock-free |
> | Shared map | `sync.RWMutex` + map | Типобезопасно, понятно |
> | Pipeline данных | channels | Идиоматично, composable |
> | One-time init | `sync.Once` | Гарантия ровно одного вызова |
> | Bounded concurrency | buffered channel / semaphore | Простой backpressure |
> | Wait for N goroutines | `sync.WaitGroup` / `errgroup` | errgroup если нужны ошибки |
> | Broadcast signal | `close(ch)` | Все receivers разблокируются |
> | Request-scoped cancel | `context.Context` | Propagation через call chain |
