# Rate Limiting

Связанные темы: [[03.1 - API Gateway Patterns]] | [[03.2 - Circuit Breaker & Resilience]] | [[01.4 - Patterns (WorkerPool, FanIn, Pipeline)]]

---

> [!question]- 1. Объясни алгоритмы rate limiting: Token Bucket, Leaky Bucket, Sliding Window.
>
> **Token Bucket:**
> - Bucket заполняется токенами с фиксированной скоростью
> - Каждый запрос потребляет 1 токен
> - Нет токенов → reject (429)
> - Допускает **burst** (если bucket полный)
>
> **Leaky Bucket:**
> - Запросы попадают в queue (bucket)
> - Обрабатываются с фиксированной скоростью (leak rate)
> - Queue полная → reject
> - Smooth output rate, **нет burst**
>
> **Fixed Window:**
> - Делим время на окна (1 минута)
> - Считаем запросы в текущем окне
> - Проблема: burst на границе окон (99 в конце + 100 в начале = 199 за 2 секунды)
>
> **Sliding Window Log:**
> - Храним timestamp каждого запроса
> - Считаем запросы в скользящем окне
> - Точный, но memory-heavy
>
> **Sliding Window Counter:**
> - Комбинация fixed window + weighted overlap
> - `count = prev_window × overlap% + current_window`
> - Хороший баланс точности и памяти

> [!question]- 2. Реализуй Token Bucket на Go.
>
> ```go
> type TokenBucket struct {
>     mu         sync.Mutex
>     tokens     float64
>     maxTokens  float64
>     refillRate float64 // tokens per second
>     lastRefill time.Time
> }
>
> func NewTokenBucket(rate float64, burst int) *TokenBucket {
>     return &TokenBucket{
>         tokens:     float64(burst),
>         maxTokens:  float64(burst),
>         refillRate: rate,
>         lastRefill: time.Now(),
>     }
> }
>
> func (tb *TokenBucket) Allow() bool {
>     tb.mu.Lock()
>     defer tb.mu.Unlock()
>
>     now := time.Now()
>     elapsed := now.Sub(tb.lastRefill).Seconds()
>     tb.tokens = math.Min(tb.maxTokens, tb.tokens+elapsed*tb.refillRate)
>     tb.lastRefill = now
>
>     if tb.tokens >= 1 {
>         tb.tokens--
>         return true
>     }
>     return false
> }
> ```
>
> В production: `golang.org/x/time/rate` (standard library quality).

> [!question]- 3. Distributed Rate Limiting — как реализовать с Redis?
>
> **Проблема:** несколько инстансов API → нужен shared counter.
>
> **Redis Sliding Window (Lua script для атомарности):**
> ```lua
> -- KEYS[1] = rate limit key
> -- ARGV[1] = window size (seconds)
> -- ARGV[2] = max requests
> -- ARGV[3] = current timestamp
>
> local key = KEYS[1]
> local window = tonumber(ARGV[1])
> local limit = tonumber(ARGV[2])
> local now = tonumber(ARGV[3])
>
> -- Remove expired entries
> redis.call('ZREMRANGEBYSCORE', key, 0, now - window)
>
> -- Count current entries
> local count = redis.call('ZCARD', key)
>
> if count < limit then
>     redis.call('ZADD', key, now, now .. math.random())
>     redis.call('EXPIRE', key, window)
>     return 1  -- allowed
> else
>     return 0  -- denied
> end
> ```
>
> **Альтернатива:** Redis `INCR` + `EXPIRE` (fixed window, проще, менее точно).

> [!question]- 4. Какие HTTP headers нужно возвращать для rate limiting?
>
> ```
> HTTP/1.1 200 OK
> X-RateLimit-Limit: 100        # макс запросов в окне
> X-RateLimit-Remaining: 42     # осталось запросов
> X-RateLimit-Reset: 1640995200 # Unix timestamp сброса окна
>
> HTTP/1.1 429 Too Many Requests
> Retry-After: 30               # секунд до retry
> X-RateLimit-Limit: 100
> X-RateLimit-Remaining: 0
> ```
>
> **Best practices:**
> - Всегда возвращай `Retry-After` при 429
> - Различай rate limits per endpoint, per user, per API key
> - Документируй limits в API docs

> [!question]- 5. Rate Limiting middleware на Go для HTTP сервера.
>
> ```go
> func rateLimitMiddleware(limiter *rate.Limiter) func(http.Handler) http.Handler {
>     return func(next http.Handler) http.Handler {
>         return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
>             if !limiter.Allow() {
>                 w.Header().Set("Retry-After", "1")
>                 http.Error(w, "rate limit exceeded", http.StatusTooManyRequests)
>                 return
>             }
>             next.ServeHTTP(w, r)
>         })
>     }
> }
>
> // Per-user rate limiting
> type UserRateLimiter struct {
>     mu       sync.Mutex
>     limiters map[string]*rate.Limiter
> }
>
> func (u *UserRateLimiter) GetLimiter(userID string) *rate.Limiter {
>     u.mu.Lock()
>     defer u.mu.Unlock()
>     if l, ok := u.limiters[userID]; ok {
>         return l
>     }
>     l := rate.NewLimiter(rate.Limit(10), 20) // 10 req/sec, burst 20
>     u.limiters[userID] = l
>     return l
> }
> ```

> [!warning]- Memory leak с per-user limiter
> Map `limiters` растёт неограниченно. Нужна периодическая очистка неактивных пользователей.
> Или используй LRU cache с TTL вместо plain map.

> [!question]- 6. Как выбрать rate limit значения? На что ориентироваться?
>
> **Факторы:**
> 1. **Backend capacity:** сколько RPS обрабатывает сервис стабильно
> 2. **SLA:** какой latency допустим при пиковой нагрузке
> 3. **Fair usage:** не позволить одному клиенту занять все ресурсы
> 4. **Cost:** если upstream API имеет стоимость per-call
>
> **Типичные тиры:**
> ```
> Free:       60 req/min, burst 10
> Basic:      600 req/min, burst 50
> Pro:        6000 req/min, burst 200
> Enterprise: custom
> ```
>
> **Разные limits для разных endpoints:**
> - `POST /auth/login`: 5/min per IP (brute force protection)
> - `GET /api/users`: 100/min per user
> - `POST /api/upload`: 10/min per user

> [!question]- 7. Adaptive Rate Limiting — что это и зачем?
>
> **Static rate limiting:** фиксированный limit (100 req/sec) — не адаптируется к нагрузке.
>
> **Adaptive rate limiting:** limit меняется в зависимости от состояния системы:
> - CPU utilization > 80% → снижаем limit
> - Error rate > 5% → снижаем limit
> - Response time > SLA → снижаем limit
> - Всё хорошо → повышаем limit (до максимума)
>
> **Netflix Concurrency Limits:**
> - Не фиксированный rate, а **max concurrent requests**
> - Автоматически подстраивается через TCP-like congestion control (AIMD)
> - `github.com/platinummonkey/go-concurrency-limits`
