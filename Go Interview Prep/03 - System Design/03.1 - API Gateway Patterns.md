# API Gateway Patterns

> [!question]- 1. API Gateway vs Load Balancer vs Reverse Proxy — в чём разница?
>
> | | Load Balancer | Reverse Proxy | API Gateway |
> |---|---------------|---------------|-------------|
> | Уровень | L4 (TCP) или L7 (HTTP) | L7 (HTTP) | L7 (Application) |
> | Задача | Распределить нагрузку | Скрыть backend, кэш, SSL | Полный lifecycle API |
> | Routing | Round-robin, least-conn | URL-based | Path, header, method |
> | Auth | Нет | Basic | OAuth, JWT, API keys |
> | Rate limiting | Нет | Basic | Advanced (per-user, per-endpoint) |
> | Transform | Нет | Header rewrite | Request/Response transformation |
> | Примеры | AWS ALB/NLB, HAProxy | Nginx, Envoy | Kong, Envoy + control plane |
>
> **На интервью:** API Gateway = Reverse Proxy + Auth + Rate Limiting + Transformations + Observability.

> [!question]- 2. Какие паттерны реализует API Gateway?
>
> **Request Aggregation** — объединение нескольких backend вызовов в один клиентский:
> - Client: `GET /user-dashboard`
> - Gateway: параллельно `GET /users/1` + `GET /orders?user=1` + `GET /notifications?user=1`
> - Response: merged JSON
>
> **Protocol Translation** — REST ↔ gRPC:
> - Client отправляет REST JSON
> - Gateway конвертирует в gRPC Protobuf для internal services
>
> **Authentication/Authorization** — централизованно на gateway:
> - JWT validation
> - OAuth2 token introspection
> - API key management
>
> **Caching** — response cache на gateway уровне:
> - GET запросы с одинаковыми параметрами
> - TTL-based invalidation
>
> **Request/Response Transformation** — header injection, body mapping, versioning.

> [!question]- 3. Как ты будешь проектировать API Gateway с нуля на Go?
>
> **Компоненты:**
> ```
> Client → TLS Termination → Rate Limiter → Auth Middleware
>       → Router → Circuit Breaker → Load Balancer → Backend Service
>       → Response Transformer → Client
> ```
>
> **Ключевые решения:**
> 1. **Routing**: `http.ServeMux` + dynamic config (etcd / consul)
> 2. **Middleware chain**: `func(http.Handler) http.Handler`
> 3. **Service discovery**: DNS-based или registry (Consul, etcd)
> 4. **Health checks**: active (ping) + passive (track errors)
> 5. **Config**: hot reload без restart
>
> **Trade-offs:**
> - Build vs Buy: Kong/Envoy уже решили 90% проблем
> - Custom gateway: полный контроль, но maintenance burden
> - Sidecar (Envoy): per-pod proxy, zero code change в сервисах

> [!question]- 4. Microservices vs Monolith — trade-offs для рефакторинга.
>
> | | Monolith | Microservices |
> |---|----------|---------------|
> | Deploy | Одним куском | Независимо |
> | Scaling | Всё целиком | Per-service |
> | Latency | In-process calls | Network calls |
> | Data | Shared DB | Database per service |
> | Complexity | Code complexity | Operational complexity |
> | Testing | E2E проще | Contract testing |
> | Team | Shared codebase | Team ownership |
>
> **Когда рефакторить в microservices:**
> - Разные части системы scale по-разному
> - Разные команды работают над разными частями
> - Deploy одной части блокирует другие
>
> **Strangler Fig Pattern** для миграции:
> 1. Новую функциональность пишем как microservice
> 2. Gateway роутит traffic: старое → monolith, новое → service
> 3. Постепенно выносим функции из monolith
> 4. Monolith уменьшается до нуля

> [!question]- 5. Как обеспечить idempotency в API?
>
> **Idempotency** — повторный запрос даёт тот же результат.
>
> **Idempotency Key:**
> ```
> POST /payments
> Idempotency-Key: abc-123-def
>
> → Backend проверяет: обрабатывал ли уже abc-123-def?
>   → Да: вернуть cached response
>   → Нет: обработать, сохранить результат с ключом, вернуть
> ```
>
> **Реализация в Go:**
> ```go
> func idempotencyMiddleware(next http.Handler) http.Handler {
>     return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
>         key := r.Header.Get("Idempotency-Key")
>         if key == "" {
>             next.ServeHTTP(w, r)
>             return
>         }
>         // Check cache (Redis)
>         if cached, ok := cache.Get(key); ok {
>             w.Write(cached)
>             return
>         }
>         // Process and cache result
>         rec := httptest.NewRecorder()
>         next.ServeHTTP(rec, r)
>         cache.Set(key, rec.Body.Bytes(), 24*time.Hour)
>         // Copy response
>         for k, v := range rec.Header() { w.Header()[k] = v }
>         w.WriteHeader(rec.Code)
>         w.Write(rec.Body.Bytes())
>     })
> }
> ```
>
> **Naturally idempotent**: GET, PUT, DELETE.
> **Not idempotent**: POST — нужен Idempotency-Key.

> [!question]- 6. Kafka vs RabbitMQ — когда какой?
>
> | | Kafka | RabbitMQ |
> |---|-------|----------|
> | Модель | Log-based (append-only) | Queue-based (push) |
> | Ordering | Per-partition guarantee | Per-queue (best-effort) |
> | Retention | Хранит сообщения (days/weeks) | Удаляет после consume |
> | Replay | Можно перечитать с offset | Нельзя (уже удалено) |
> | Throughput | Millions msg/sec | Thousands msg/sec |
> | Latency | Выше (batching) | Ниже (push-based) |
> | Use case | Event sourcing, log aggregation | Task queues, RPC |
>
> **Kafka когда:**
> - Event sourcing / CQRS
> - Stream processing (Kafka Streams)
> - Audit log (нужен replay)
> - High throughput (analytics events)
>
> **RabbitMQ когда:**
> - Task queue (background jobs)
> - Request-reply pattern
> - Complex routing (topic exchange)
> - Low-latency messaging

> [!question]- 7. SSO — как работает? Объясни OAuth 2.0 + OIDC flow.
>
> **OAuth 2.0 Authorization Code Flow:**
> ```
> 1. User clicks "Login with Google"
> 2. App redirects to Google: /authorize?client_id=X&redirect_uri=Y&scope=openid+email
> 3. User authenticates with Google
> 4. Google redirects back: Y?code=AUTH_CODE
> 5. App exchanges code for tokens: POST /token (code + client_secret)
> 6. Google returns: { access_token, id_token, refresh_token }
> 7. App validates id_token (JWT) → gets user info
> ```
>
> **OIDC** = OAuth 2.0 + Identity layer (id_token с user claims).
>
> **JWT (id_token) содержит:**
> - `sub` — unique user ID
> - `email` — verified email
> - `exp` — expiration time
> - `iss` — issuer (Google, Auth0, etc.)

> [!question]- 8. Как организовать API versioning?
>
> **Подходы:**
> | Подход | Пример | Плюсы | Минусы |
> |--------|--------|-------|--------|
> | URL path | `/v1/users` | Простой, явный | URL coupling |
> | Header | `Accept: application/vnd.api+json;version=2` | Clean URL | Сложнее debug |
> | Query param | `/users?version=2` | Простой, cacheable | Загрязняет params |
>
> **Рекомендация для API Gateway рефакторинга:**
> - URL path (`/v1/`, `/v2/`) — самый распространённый, cache-friendly
> - Gateway роутит по version: `/v1/*` → old service, `/v2/*` → new service
> - Deprecation: header `Sunset: date` + `Deprecation: true`

> [!question]- 9. gRPC vs REST — trade-offs для микросервисов.
>
> | | REST | gRPC |
> |---|------|------|
> | Protocol | HTTP/1.1 (обычно) | HTTP/2 |
> | Format | JSON (text) | Protobuf (binary) |
> | Contract | OpenAPI (optional) | .proto (required) |
> | Streaming | WebSocket, SSE | Bidirectional native |
> | Codegen | Optional (OpenAPI) | Required (protoc) |
> | Browser | Native | Нужен gRPC-Web proxy |
> | Performance | Ниже (text parsing) | Выше (~10x faster serialization) |
>
> **Паттерн для API Gateway:**
> - External (clients) → REST/JSON (universal)
> - Internal (service-to-service) → gRPC (performance)
> - Gateway: REST → gRPC translation (transcoding)

> [!question]- 10. Observability — три столпа. Как реализовать?
>
> **1. Logging** (structured):
> ```go
> slog.Info("request processed",
>     "method", r.Method,
>     "path", r.URL.Path,
>     "status", status,
>     "duration_ms", duration.Milliseconds(),
>     "trace_id", traceID,
> )
> ```
>
> **2. Metrics** (Prometheus):
> ```go
> httpRequestsTotal := prometheus.NewCounterVec(
>     prometheus.CounterOpts{Name: "http_requests_total"},
>     []string{"method", "path", "status"},
> )
> httpRequestDuration := prometheus.NewHistogramVec(
>     prometheus.HistogramOpts{Name: "http_request_duration_seconds"},
>     []string{"method", "path"},
> )
> ```
>
> **3. Tracing** (OpenTelemetry):
> - Каждый запрос получает `trace_id`
> - Передаётся через все сервисы в headers
> - Визуализация в Jaeger/Tempo
>
> **Ключевое для интервью:** все три связаны через `trace_id` — лог содержит trace_id, trace содержит метрики, метрики ведут к traces.

> [!question]- 11. gRPC и HTTP/2 — в чём разница? gRPC это не то же самое что HTTP/2?
>
> **gRPC — это фреймворк поверх HTTP/2, не альтернатива ему.**
>
> ```
> gRPC
> └── HTTP/2 (transport)
>     └── TLS
>         └── TCP
> ```
>
> **HTTP/2** даёт:
> - Мультиплексирование (много запросов по одному TCP соединению)
> - Header compression (HPACK)
> - Binary framing (в отличие от text-based HTTP/1.1)
> - Server push
>
> **gRPC добавляет поверх HTTP/2:**
> - Protobuf сериализацию (contract-first: `.proto` файлы)
> - Codegen (protoc генерирует client/server stubs)
> - 4 типа RPC: Unary, Server-streaming, Client-streaming, Bidirectional
> - Interceptors (аналог middleware для gRPC)
> - Свои Status codes (OK, CANCELLED, UNAVAILABLE, NOT_FOUND...)
> - Встроенные: deadlines, health checking, load balancing
>
> **Ключевые выводы:**
> - REST обычно ходит по HTTP/1.1, но может и по HTTP/2 (без gRPC фич)
> - gRPC **требует** HTTP/2 по спецификации — без него не работает
> - Можно использовать HTTP/2 напрямую без gRPC, но тогда Protobuf, codegen и streaming типы нужно реализовывать самостоятельно
>
> **Паттерн для микросервисов:**
> ```
> External clients → REST/JSON over HTTP/1.1 (universal, browser-friendly)
> Internal services → gRPC/Protobuf over HTTP/2 (performance, strong contracts)
> API Gateway: REST → gRPC transcoding (grpc-gateway / Envoy)
> ```

> [!question]- 12. HTTP методы — назначение и идемпотентность.
>
> | Метод | Назначение | Идемпотентен? | Тело запроса |
> |-------|-----------|---------------|--------------|
> | **GET** | Получить ресурс | Да | Нет |
> | **POST** | Создать ресурс / отправить данные | Нет | Да |
> | **PUT** | Заменить ресурс целиком | Да | Да |
> | **PATCH** | Частично обновить ресурс | Нет (обычно) | Да |
> | **DELETE** | Удалить ресурс | Да | Нет (редко) |
> | **HEAD** | Как GET, но без тела ответа | Да | Нет |
> | **OPTIONS** | Узнать поддерживаемые методы | Да | Нет |
>
> **Идемпотентность** — повторный запрос даёт тот же результат на сервере:
> - `DELETE /users/1` → удалён (повторно: 404, но ресурс не изменился дважды)
> - `PUT /users/1 {"name":"Alice"}` → всегда один и тот же результат
> - `POST /users` → каждый раз создаёт нового пользователя
>
> **PUT vs PATCH:**
> ```
> PUT  /users/1 {"name":"Alice","age":30}  — полная замена (нужны ВСЕ поля)
> PATCH /users/1 {"name":"Alice"}          — частичное обновление (только изменения)
> ```
>
> **HEAD** используется для:
> - Проверки существования ресурса без загрузки тела
> - Получения Content-Length перед скачиванием
> - Cache validation (ETag без body)
>
> **OPTIONS** — браузер отправляет preflight OPTIONS запрос перед cross-origin POST/PUT/DELETE (CORS).
