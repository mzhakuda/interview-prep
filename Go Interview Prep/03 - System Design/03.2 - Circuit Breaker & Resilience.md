> [!question]- 1. Объясни Circuit Breaker паттерн. Нарисуй state machine.
>
> ```
> ┌──────────┐  failure threshold   ┌──────────┐
> │  CLOSED  │ ──────────────────→  │   OPEN   │
> │ (normal) │                      │(fail fast)│
> └──────────┘                      └──────────┘
>      ↑                                  │
>      │ success                    timeout│
>      │                                  ↓
>      │                           ┌───────────┐
>      └─────────────────────────  │ HALF-OPEN │
>                success           │  (probe)  │
>                                  └───────────┘
>                                       │
>                                 failure│
>                                       ↓
>                                  ┌──────────┐
>                                  │   OPEN   │
>                                  └──────────┘
> ```
>
> **CLOSED** — нормальная работа, запросы проходят. Считаются failures.
> **OPEN** — сервис недоступен, запросы **сразу отклоняются** (fail fast). Таймер.
> **HALF-OPEN** — пропускаем ограниченное число probe запросов. Если ок → CLOSED. Если fail → OPEN.
>
> **Параметры:** failure threshold (5), timeout (30s), half-open max requests (3).

> [!example]- Код: Circuit Breaker на Go (упрощённый)
> ```go
> type CircuitBreaker struct {
>     mu          sync.Mutex
>     state       string // closed, open, half-open
>     failures    int
>     threshold   int
>     timeout     time.Duration
>     lastFailure time.Time
> }
>
> func (cb *CircuitBreaker) Execute(fn func() error) error {
>     cb.mu.Lock()
>     if cb.state == "open" {
>         if time.Since(cb.lastFailure) > cb.timeout {
>             cb.state = "half-open"
>         } else {
>             cb.mu.Unlock()
>             return ErrCircuitOpen
>         }
>     }
>     cb.mu.Unlock()
>
>     err := fn()
>
>     cb.mu.Lock()
>     defer cb.mu.Unlock()
>     if err != nil {
>         cb.failures++
>         cb.lastFailure = time.Now()
>         if cb.failures >= cb.threshold {
>             cb.state = "open"
>         }
>         return err
>     }
>     cb.failures = 0
>     cb.state = "closed"
>     return nil
> }
> ```
> В production: `sony/gobreaker` или `afex/hystrix-go`.

> [!question]- 2. Retry с Exponential Backoff + Jitter — зачем jitter?
>
> **Exponential Backoff:** delay = base × 2^attempt (1s, 2s, 4s, 8s...)
>
> **Проблема без jitter:** если 1000 клиентов получили ошибку одновременно,
> все retry в одни и те же моменты → **thundering herd** → сервис снова падает.
>
> **Jitter** добавляет случайность: delay = random(0, base × 2^attempt)
>
> ```go
> func retryWithBackoff(ctx context.Context, maxRetries int, fn func() error) error {
>     for i := 0; i < maxRetries; i++ {
>         err := fn()
>         if err == nil { return nil }
>         if !isRetryable(err) { return err }
>
>         // Exponential backoff with full jitter
>         maxDelay := time.Duration(math.Pow(2, float64(i))) * time.Second
>         delay := time.Duration(rand.Int63n(int64(maxDelay)))
>
>         select {
>         case <-time.After(delay):
>         case <-ctx.Done():
>             return ctx.Err()
>         }
>     }
>     return fmt.Errorf("max retries exceeded")
> }
> ```

> [!warning]- Что retry-ить, а что нет
> **Retry:** 503, 429, timeout, connection reset (transient)
> **Не retry:** 400, 401, 403, 404, 409 (client error — retry не поможет)
> **Осторожно:** POST запросы — только если idempotent

> [!question]- 3. Bulkhead паттерн — как изолировать failures?
>
> **Bulkhead** — изоляция ресурсов между компонентами (как переборки в корабле).
>
> **Реализации:**
> 1. **Thread pool isolation:** отдельный пул потоков/горутин для каждого downstream
> 2. **Semaphore isolation:** ограничение concurrent calls к каждому сервису
> 3. **Connection pool isolation:** отдельный пул соединений per service
>
> ```go
> // Semaphore bulkhead
> type Bulkhead struct {
>     sem chan struct{}
> }
>
> func NewBulkhead(maxConcurrent int) *Bulkhead {
>     return &Bulkhead{sem: make(chan struct{}, maxConcurrent)}
> }
>
> func (b *Bulkhead) Execute(ctx context.Context, fn func() error) error {
>     select {
>     case b.sem <- struct{}{}:
>         defer func() { <-b.sem }()
>         return fn()
>     case <-ctx.Done():
>         return ctx.Err()
>     }
> }
> ```
>
> **Без bulkhead:** один медленный downstream → все горутины заняты ожиданием → весь сервис лежит.

> [!question]- 4. Timeout стратегии — какие бывают?
>
> | Тип | Где | Пример |
> |-----|-----|--------|
> | Connection timeout | TCP handshake | 5s |
> | Read/Write timeout | Ожидание ответа | 10-30s |
> | Request timeout | Общий на весь запрос | 30s |
> | Context timeout | Go context.WithTimeout | Propagates through chain |
>
> **Каскадные timeouts:**
> ```
> Client (30s) → Gateway (25s) → Service A (20s) → Service B (15s)
> ```
> **Правило:** каждый следующий timeout должен быть **меньше** предыдущего.
> Иначе: Gateway timeout = 30s, Service = 60s → Gateway отменяет, Service продолжает впустую.

> [!question]- 5. Graceful Degradation — как деградировать красиво?
>
> **Стратегии:**
> 1. **Fallback response:** Cache, default value, stale data
> 2. **Feature flags:** отключить тяжёлые features под нагрузкой
> 3. **Load shedding:** отклонять low-priority запросы
> 4. **Partial response:** вернуть что есть, пометить missing parts
>
> ```go
> func getUserProfile(ctx context.Context, id string) (*Profile, error) {
>     profile, err := userService.Get(ctx, id)
>     if err != nil {
>         // Fallback: cached version
>         cached, cacheErr := cache.Get(ctx, "profile:"+id)
>         if cacheErr == nil {
>             return cached, nil // stale but available
>         }
>         return &Profile{ID: id, Name: "Unknown"}, nil // degraded
>     }
>     cache.Set(ctx, "profile:"+id, profile, time.Hour)
>     return profile, nil
> }
> ```

> [!question]- 6. Health checks — liveness vs readiness. Разница?
>
> **Liveness** — "процесс жив?":
> - Ответ: 200 OK если процесс работает
> - Провал → **restart** контейнера (K8s restartPolicy)
> - Не проверяй зависимости! Если DB down, restart сервиса не поможет
>
> **Readiness** — "готов принимать трафик?":
> - Ответ: 200 OK если все зависимости доступны
> - Провал → убрать из load balancer (не restart)
> - Проверяй: DB connection, cache, critical dependencies
>
> ```go
> // Liveness: /health/live
> func liveness(w http.ResponseWriter, r *http.Request) {
>     w.WriteHeader(http.StatusOK)
> }
>
> // Readiness: /health/ready
> func readiness(w http.ResponseWriter, r *http.Request) {
>     if err := db.PingContext(r.Context()); err != nil {
>         w.WriteHeader(http.StatusServiceUnavailable)
>         return
>     }
>     w.WriteHeader(http.StatusOK)
> }
> ```

> [!question]- 7. Как предотвратить cascading failures в microservices?
>
> **Комбинация паттернов:**
> 1. **Circuit Breaker** — fail fast для недоступных сервисов
> 2. **Timeout** — не висеть бесконечно
> 3. **Retry + Backoff** — для transient failures
> 4. **Bulkhead** — изолировать ресурсы per-downstream
> 5. **Rate Limiting** — защита от перегрузки
> 6. **Load Shedding** — отклонение при пиковой нагрузке
> 7. **Async communication** — message queue для non-critical paths
>
> **Порядок в middleware chain:**
> ```
> Rate Limiter → Timeout → Circuit Breaker → Bulkhead → Retry → HTTP Call
> ```

> [!question]- 8. Что такое back-pressure? Как реализовать?
>
> **Back-pressure** — механизм замедления producer когда consumer не справляется.
>
> **Реализации:**
> 1. **Bounded channels:** producer блокируется когда buffer full
> 2. **Rate limiting:** ограничить incoming requests
> 3. **HTTP 429:** Too Many Requests (retry-after header)
> 4. **TCP flow control:** автоматический на transport level
> 5. **Reactive Streams:** pull-based model (consumer запрашивает N items)
>
> **Без back-pressure:** OOM, queue overflow, cascading failure.
> **С back-pressure:** система замедляется, но остаётся стабильной.
