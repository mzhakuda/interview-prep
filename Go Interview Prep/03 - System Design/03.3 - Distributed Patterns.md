# Distributed Patterns

> [!question]- 1. Saga pattern — как реализовать distributed transaction?
>
> **Проблема:** в микросервисах нет shared DB → нет ACID транзакций между сервисами.
>
> **Saga** = последовательность local transactions. Каждый шаг имеет compensating action.
>
> **Choreography (события):**
> ```
> Order Service → OrderCreated event
>   → Payment Service обрабатывает → PaymentCompleted event
>     → Inventory Service обрабатывает → InventoryReserved event
>       → Order Service: mark as confirmed
>
> Если Payment fails:
>   → PaymentFailed event → Order Service: cancel order (compensate)
> ```
>
> **Orchestration (центральный координатор):**
> ```
> Saga Orchestrator:
>   1. Call Order.Create() → ok
>   2. Call Payment.Charge() → ok
>   3. Call Inventory.Reserve() → FAIL
>   4. Compensate: Payment.Refund()
>   5. Compensate: Order.Cancel()
> ```
>
> **Trade-offs:**
> - Choreography: децентрализован, но сложно дебажить (event chains)
> - Orchestration: единая точка управления, но single point of failure

> [!question]- 2. Outbox Pattern — как надёжно публиковать события?
>
> **Проблема:** `UPDATE db + publish event` — не атомарно. Crash между ними = потеря события.
>
> **Outbox pattern:**
> 1. В одной DB транзакции: UPDATE бизнес-данных + INSERT в outbox таблицу
> 2. Отдельный процесс (CDC / poller) читает outbox → публикует в message broker
> 3. После publish → помечает как отправленное
>
> ```sql
> BEGIN;
> UPDATE orders SET status = 'confirmed' WHERE id = 1;
> INSERT INTO outbox (aggregate_id, event_type, payload)
>     VALUES (1, 'OrderConfirmed', '{"orderId": 1}');
> COMMIT;
> ```
>
> **CDC (Change Data Capture):** Debezium читает WAL PostgreSQL → отправляет в Kafka.
> Это надёжнее polling: at-least-once delivery без нагрузки на DB.

> [!question]- 3. Event Sourcing vs Traditional CRUD — когда что?
>
> **CRUD**: хранит текущее состояние. `UPDATE accounts SET balance = 150`.
>
> **Event Sourcing**: хранит все события. State = replay events.
> ```
> AccountCreated(balance=100)
> MoneyDeposited(amount=50)
> MoneyWithdrawn(amount=20)
> → Current state: balance = 130
> ```
>
> **Преимущества ES:**
> - Полный audit trail (кто, когда, что)
> - Time travel (восстановить состояние на любой момент)
> - Естественный fit для event-driven architectures
>
> **Проблемы ES:**
> - Complexity (eventual consistency, CQRS usually needed)
> - Event versioning (schema evolution)
> - Replay performance (snapshots needed for long streams)
>
> **Используй ES когда:** audit trail critical, temporal queries нужны, domain events = первичная модель.
> **Используй CRUD когда:** simple domain, team inexperienced with ES, не нужна history.

> [!question]- 4. CQRS — зачем разделять reads и writes?
>
> **CQRS** (Command Query Responsibility Segregation):
> - **Command model** — оптимизирован для writes (normalized, consistency)
> - **Query model** — оптимизирован для reads (denormalized, performance)
>
> ```
> Write: POST /orders → Command Handler → Write DB (PostgreSQL)
>                                        → Events → Projection → Read DB (Elasticsearch)
>
> Read:  GET /orders?search=... → Query Handler → Read DB (Elasticsearch)
> ```
>
> **Когда CQRS:**
> - Read и Write нагрузки сильно различаются (100:1)
> - Разные requirements для reads (full-text search, aggregations)
> - Event Sourcing (CQRS почти обязателен)
>
> **Проблема:** eventual consistency между write и read models.

> [!question]- 5. Consistent Hashing — зачем и как работает?
>
> **Проблема:** `hash(key) % N` — при добавлении/удалении node **все** ключи перераспределяются.
>
> **Consistent Hashing:**
> - Nodes и keys хешируются на кольцо (0 → 2^32)
> - Key идёт к ближайшему node по часовой стрелке
> - При добавлении/удалении node — перемещается только ~1/N ключей
>
> **Virtual nodes:** каждый physical node = множество точек на кольце.
> Обеспечивает равномерное распределение (без virtual nodes один node может получить непропорционально много ключей).
>
> **Use cases:** distributed cache (Redis Cluster), partitioned databases, load balancers.

> [!question]- 6. CAP теорема — что реально означает?
>
> **C**onsistency — все nodes видят одни данные одновременно
> **A**vailability — каждый запрос получает ответ (не обязательно свежий)
> **P**artition tolerance — система работает при network partition
>
> **Реальность:** partition ВСЕГДА возможен → выбор между C и A:
> - **CP** (Consistency + Partition tolerance): HBase, MongoDB (strong consistency mode)
> - **AP** (Availability + Partition tolerance): Cassandra, DynamoDB
>
> **Нюанс для интервью:** CAP — это не "выбери 2 из 3". Это: "при partition, ты выбираешь C или A?"
> В отсутствие partition система может быть и C, и A одновременно.

> [!question]- 7. Distributed caching — стратегии инвалидации.
>
> | Стратегия | Описание | Trade-off |
> |-----------|----------|-----------|
> | **Cache-Aside** | App читает cache, если miss → DB → update cache | Stale data возможен |
> | **Write-Through** | Каждый write → cache + DB одновременно | Consistent, но медленный write |
> | **Write-Behind** | Write → cache, async flush → DB | Fast write, risk of data loss |
> | **TTL-based** | Cache expires after N seconds | Simple, but stale window |
> | **Event-based** | DB change → event → invalidate cache | Consistent, complex setup |
>
> **Cache-Aside pattern (самый распространённый):**
> ```go
> func GetUser(ctx context.Context, id string) (*User, error) {
>     // 1. Try cache
>     if cached, err := redis.Get(ctx, "user:"+id); err == nil {
>         return deserialize(cached)
>     }
>     // 2. Cache miss → DB
>     user, err := db.GetUser(ctx, id)
>     if err != nil { return nil, err }
>     // 3. Update cache
>     redis.Set(ctx, "user:"+id, serialize(user), time.Hour)
>     return user, nil
> }
> ```

> [!question]- 8. Leader Election — зачем и как реализовать?
>
> **Зачем:** один процесс должен выполнять задачу (cron job, rebalancing).
>
> **Механизмы:**
> 1. **PostgreSQL advisory lock:** `SELECT pg_try_advisory_lock(1)` — leader = who got lock
> 2. **Redis SETNX:** `SET leader:cron $id NX EX 30` — с TTL для lease renewal
> 3. **etcd / Consul:** native leader election API
> 4. **Kubernetes Lease:** coordination.k8s.io/v1 Lease objects
>
> **Проблема split-brain:** два процесса считают себя leader:
> - Fencing tokens: leader получает monotonic token, operations с old token отклоняются
> - Lease renewal: leader периодически обновляет lease, при пропуске — новые выборы

> [!question]- 9. Что такое eventual consistency? Как жить с ней?
>
> **Eventual consistency:** если нет новых writes, все replicas **со временем** придут к одному состоянию.
>
> **Практика:**
> - Read-after-write: пользователь создал пост → refresh → не видит (replica stale)
> - Решение: читай из master после write (session consistency)
>
> **Паттерны для работы с EC:**
> 1. **Read-your-writes:** направляй reads на master после write (таймер или cookie)
> 2. **Causal consistency:** отслеживай "happens-before" через vector clocks
> 3. **Conflict resolution:** last-writer-wins, merge functions, CRDTs
> 4. **Compensation:** если обнаружен конфликт → compensating action

> [!question]- 10. Как спроектировать distributed job scheduler?
>
> **Требования:** at-least-once execution, distributed, scalable, observable.
>
> **Архитектура:**
> ```
> Job Table (PostgreSQL):
>   id | type | payload | status | run_at | locked_by | locked_until
>
> Workers (N pods):
>   1. SELECT ... WHERE status='pending' AND run_at <= now()
>      FOR UPDATE SKIP LOCKED LIMIT 10
>   2. UPDATE SET status='running', locked_by=worker_id, locked_until=now()+timeout
>   3. Execute job
>   4. UPDATE SET status='completed'
>
> Если worker dies: locked_until expires → другой worker подберёт
> ```
>
> **FOR UPDATE SKIP LOCKED** — идеален: workers не конкурируют за одну задачу, нет deadlocks.
> **Observability:** metrics per job type (duration, success rate, queue depth).
