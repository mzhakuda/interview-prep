# Hexagonal Architecture

Связанные темы: [[05.1 - Clean Architecture]] | [[03.1 - API Gateway Patterns]]

---

> [!question]- 1. Объясни Hexagonal Architecture (Ports & Adapters). Как она работает?
>
> **Hexagonal Architecture** (Alistair Cockburn, 2005) делит приложение на:
>
> ```
>              ┌──────────────────────────┐
>  HTTP ──→ [Port] ──→ │                        │
>  gRPC ──→ [Port] ──→ │     Application Core   │ ──→ [Port] ──→ PostgreSQL
>  CLI  ──→ [Port] ──→ │   (Domain + Use Cases)  │ ──→ [Port] ──→ Redis
>              │                        │ ──→ [Port] ──→ Email Service
>              └──────────────────────────┘
>        Driving (Primary)              Driven (Secondary)
>        Adapters                       Adapters
> ```
>
> **Port** — интерфейс (контракт). Определяется внутри core.
> **Adapter** — конкретная реализация порта (HTTP handler, PostgreSQL repo).
>
> **Driving (Primary) Adapters:** вызывают application core (HTTP, gRPC, CLI, tests)
> **Driven (Secondary) Adapters:** вызываются application core (DB, cache, email, queue)

> [!question]- 2. Hexagonal vs Clean Architecture — в чём разница?
>
> | | Hexagonal | Clean Architecture |
> |---|-----------|-------------------|
> | Автор | Alistair Cockburn (2005) | Robert Martin (2012) |
> | Слои | 2: Core + Adapters | 4: Entity, UseCase, Adapter, Framework |
> | Визуализация | Гексагон с портами | Концентрические круги |
> | Направление | Inside-out (core определяет порты) | Outside-in (dependency rule) |
> | Naming | Ports & Adapters | Layers & Boundaries |
>
> **На практике почти одно и то же:**
> - Оба изолируют бизнес-логику от инфраструктуры
> - Оба используют Dependency Inversion
> - Оба определяют интерфейсы в core, реализации снаружи
>
> **Hexagonal проще** — меньше слоёв, легче начать.
> **Clean Architecture точнее** — разделяет Entities от Use Cases.

> [!question]- 3. Реализуй Hexagonal Architecture для Order Service на Go.
>
> ```go
> // === PORTS (interfaces in core) ===
>
> // Primary port (driving):
> type OrderService interface {
>     CreateOrder(ctx context.Context, req CreateOrderRequest) (*Order, error)
>     GetOrder(ctx context.Context, id string) (*Order, error)
> }
>
> // Secondary ports (driven):
> type OrderRepository interface {
>     Save(ctx context.Context, order *Order) error
>     FindByID(ctx context.Context, id string) (*Order, error)
> }
>
> type PaymentGateway interface {
>     Charge(ctx context.Context, amount Money) (*PaymentResult, error)
> }
>
> type EventPublisher interface {
>     Publish(ctx context.Context, event DomainEvent) error
> }
>
> // === CORE (application logic) ===
>
> type orderService struct {
>     repo     OrderRepository
>     payment  PaymentGateway
>     events   EventPublisher
> }
>
> func (s *orderService) CreateOrder(ctx context.Context, req CreateOrderRequest) (*Order, error) {
>     order := NewOrder(req)  // domain logic
>     if err := s.payment.Charge(ctx, order.Total()); err != nil {
>         return nil, fmt.Errorf("payment failed: %w", err)
>     }
>     if err := s.repo.Save(ctx, order); err != nil {
>         return nil, err
>     }
>     s.events.Publish(ctx, OrderCreatedEvent{OrderID: order.ID})
>     return order, nil
> }
>
> // === ADAPTERS ===
>
> // Primary adapter (HTTP):
> type HTTPOrderHandler struct { svc OrderService }
>
> // Secondary adapter (PostgreSQL):
> type PostgresOrderRepo struct { db *pgxpool.Pool }
>
> // Secondary adapter (Stripe):
> type StripePaymentGateway struct { client *stripe.Client }
> ```

> [!question]- 4. Как Dependency Injection работает в Go без фреймворков?
>
> **Manual DI в main.go (wire up):**
> ```go
> func main() {
>     // Infrastructure
>     db := connectDB()
>     redisClient := connectRedis()
>     stripeClient := stripe.NewClient(os.Getenv("STRIPE_KEY"))
>
>     // Adapters (secondary)
>     orderRepo := postgres.NewOrderRepo(db)
>     cache := redis.NewCache(redisClient)
>     payment := stripe.NewPaymentGateway(stripeClient)
>     publisher := kafka.NewEventPublisher(kafkaConfig)
>
>     // Core
>     orderSvc := service.NewOrderService(orderRepo, payment, publisher)
>
>     // Adapters (primary)
>     handler := http.NewOrderHandler(orderSvc)
>
>     // Start server
>     mux := http.NewServeMux()
>     mux.Handle("/orders", handler)
>     http.ListenAndServe(":8080", mux)
> }
> ```
>
> **Преимущества manual DI:**
> - Нет магии, всё явно
> - Compile-time проверка зависимостей
> - Легко понять что от чего зависит
>
> **Если DI граф большой:** `google/wire` (compile-time DI generation).

> [!question]- 5. Когда Hexagonal — overkill? Альтернативы?
>
> **Overkill когда:**
> - Simple CRUD API (5-10 endpoints, no complex logic)
> - Prototype / MVP (нужна скорость, не архитектура)
> - Solo developer, маленький проект
>
> **Альтернативы:**
>
> **Flat structure** — для маленьких проектов:
> ```
> main.go
> handlers.go
> models.go
> db.go
> ```
>
> **Package-by-feature** — для средних проектов:
> ```
> users/
>   handler.go
>   service.go
>   repository.go
>   model.go
> orders/
>   handler.go
>   service.go
>   ...
> ```
>
> **Правило:** начинай с simple → рефактори в hexagonal когда domain complexity растёт.
> "Make it work, make it right, make it fast."
