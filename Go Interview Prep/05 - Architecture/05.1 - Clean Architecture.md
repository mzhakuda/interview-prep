# Clean Architecture

Связанные темы: [[05.2 - Hexagonal Architecture]] | [[03.1 - API Gateway Patterns]]

---

> [!question]- 1. Объясни Clean Architecture. Какие слои и как они зависят друг от друга?
>
> **Dependency Rule:** зависимости направлены **только внутрь** (от внешних слоёв к внутренним).
>
> ```
> ┌─────────────────────────────────────┐
> │  Frameworks & Drivers (HTTP, DB)    │  ← внешний
> │  ┌─────────────────────────────┐    │
> │  │  Interface Adapters         │    │  ← controllers, presenters, gateways
> │  │  ┌─────────────────────┐   │    │
> │  │  │  Use Cases           │   │    │  ← application business rules
> │  │  │  ┌───────────────┐  │   │    │
> │  │  │  │  Entities      │  │   │    │  ← enterprise business rules (core)
> │  │  │  └───────────────┘  │   │    │
> │  │  └─────────────────────┘   │    │
> │  └─────────────────────────────┘    │
> └─────────────────────────────────────┘
> ```
>
> **Entities** — бизнес-объекты, не зависят ни от чего
> **Use Cases** — application-specific бизнес-правила
> **Interface Adapters** — конвертируют данные между use cases и внешним миром
> **Frameworks** — конкретные технологии (PostgreSQL, HTTP, gRPC)

> [!question]- 2. Как Clean Architecture выглядит в Go проекте?
>
> ```
> project/
> ├── cmd/
> │   └── server/main.go          # entry point, wire dependencies
> ├── internal/
> │   ├── domain/                  # Entities
> │   │   ├── user.go             # User struct, business rules
> │   │   └── order.go
> │   ├── usecase/                 # Use Cases
> │   │   ├── user_service.go     # interfaces + implementation
> │   │   └── order_service.go
> │   ├── adapter/                 # Interface Adapters
> │   │   ├── http/               # HTTP handlers
> │   │   │   └── user_handler.go
> │   │   ├── grpc/               # gRPC handlers
> │   │   └── repository/         # DB implementations
> │   │       └── postgres_user.go
> │   └── infrastructure/         # Frameworks & Drivers
> │       ├── database/           # DB connection, migrations
> │       └── config/             # Config loading
> └── pkg/                        # Shared utilities
> ```
>
> **Ключевое:** `domain/` и `usecase/` не импортируют ничего из `adapter/` или `infrastructure/`.
> Use cases определяют **интерфейсы** (ports), adapters их **реализуют**.

> [!example]- Код: Dependency Inversion в Go
> ```go
> // domain/user.go
> type User struct {
>     ID    string
>     Email string
>     Name  string
> }
>
> // usecase/user_service.go — интерфейс (port)
> type UserRepository interface {
>     GetByID(ctx context.Context, id string) (*domain.User, error)
>     Save(ctx context.Context, user *domain.User) error
> }
>
> type UserService struct {
>     repo UserRepository // зависит от интерфейса, не от реализации
> }
>
> func (s *UserService) GetUser(ctx context.Context, id string) (*domain.User, error) {
>     return s.repo.GetByID(ctx, id)
> }
>
> // adapter/repository/postgres_user.go — реализация (adapter)
> type PostgresUserRepo struct {
>     db *sql.DB
> }
>
> func (r *PostgresUserRepo) GetByID(ctx context.Context, id string) (*domain.User, error) {
>     // SQL query...
> }
> ```

> [!question]- 3. Плюсы и минусы Clean Architecture?
>
> **Плюсы:**
> - Testability: бизнес-логика тестируется без DB, HTTP
> - Flexibility: замена PostgreSQL на MongoDB — только adapter
> - Maintainability: чёткие границы, легко ориентироваться
> - Independence: бизнес-логика не зависит от фреймворков
>
> **Минусы:**
> - Boilerplate: много файлов, interfaces, struct conversions
> - Over-engineering для простых CRUD приложений
> - Mapping overhead: конвертация между слоями (domain ↔ DTO ↔ DB model)
> - Learning curve: команда должна понимать принцип
>
> **Когда использовать:**
> - Долгоживущий проект с complex domain logic
> - Несколько точек входа (HTTP + gRPC + CLI)
> - Команда > 3 человек
>
> **Когда НЕ использовать:**
> - Simple CRUD / MVP / прототип
> - Одна точка входа, простой domain

> [!question]- 4. Как тестировать в Clean Architecture?
>
> **Unit tests (use cases):** mock репозитории через интерфейсы:
> ```go
> type mockUserRepo struct {
>     users map[string]*domain.User
> }
>
> func (m *mockUserRepo) GetByID(_ context.Context, id string) (*domain.User, error) {
>     if u, ok := m.users[id]; ok { return u, nil }
>     return nil, ErrNotFound
> }
>
> func TestGetUser(t *testing.T) {
>     repo := &mockUserRepo{users: map[string]*domain.User{
>         "1": {ID: "1", Name: "John"},
>     }}
>     svc := &UserService{repo: repo}
>     user, err := svc.GetUser(context.Background(), "1")
>     assert.NoError(t, err)
>     assert.Equal(t, "John", user.Name)
> }
> ```
>
> **Integration tests (adapters):** реальная DB через testcontainers.
> **E2E tests:** полный stack, HTTP client → server → DB.

> [!question]- 5. SOLID в контексте Go — как применяются?
>
> **S** — Single Responsibility: один struct = одна ответственность
> **O** — Open/Closed: расширяй через интерфейсы, не меняя существующий код
> **L** — Liskov Substitution: любая реализация интерфейса должна быть заменяема
> **I** — Interface Segregation: маленькие интерфейсы (1-3 метода), не монолитные
> **D** — Dependency Inversion: зависи от интерфейсов, не от конкретных типов
>
> **Go идиома:** "Accept interfaces, return structs."
> ```go
> // Хорошо: принимаем io.Reader (маленький интерфейс)
> func ProcessData(r io.Reader) error { ... }
>
> // Плохо: принимаем конкретный тип
> func ProcessData(f *os.File) error { ... }
> ```
>
> **Interface Segregation в Go:**
> ```go
> // Плохо: жирный интерфейс
> type UserStore interface {
>     Get(id string) (*User, error)
>     Save(user *User) error
>     Delete(id string) error
>     List() ([]*User, error)
>     Search(query string) ([]*User, error)
> }
>
> // Хорошо: разделённые интерфейсы
> type UserReader interface { Get(id string) (*User, error) }
> type UserWriter interface { Save(user *User) error }
> ```
