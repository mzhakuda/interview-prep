# MVCC & Transactions

Связанные темы: [[02.2 - Indexes Deep Dive]] | [[02.3 - Locking & Isolation Levels]] | [[02.4 - Performance & Connection Pooling]]

---

> [!question]- 1. Объясни MVCC в PostgreSQL. Зачем он нужен и как работает?
>
> **MVCC** (Multi-Version Concurrency Control) — основной механизм конкурентности PostgreSQL.
>
> Принцип: при UPDATE/DELETE строка **не перезаписывается**. Создаётся **новая версия** (tuple).
>
> Каждый tuple имеет скрытые системные колонки:
> - `xmin` — ID транзакции, которая создала tuple
> - `xmax` — ID транзакции, которая удалила/обновила tuple (0 если живой)
>
> **Читатели не блокируют писателей. Писатели не блокируют читателей.**
>
> Каждая транзакция видит **snapshot** данных на момент своего начала.
> Это позволяет множественным транзакциям работать параллельно без блокировок на чтение.

> [!hint]- Подсказка для интервью
> Ключевая фраза: "PostgreSQL хранит несколько версий одной строки одновременно.
> Каждая транзакция видит свой consistent snapshot данных."

> [!question]- 2. Что такое dead tuples? Почему они появляются и чем опасны?
>
> **Dead tuples** — старые версии строк, которые больше никому не видны.
>
> Появляются при:
> - `UPDATE` — старая версия стала dead, новая — live
> - `DELETE` — строка помечена как dead (xmax != 0)
>
> **Опасности:**
> - **Table bloat** — таблица растёт, Seq Scan замедляется
> - **Index bloat** — индексы ссылаются на dead tuples
> - **Transaction ID wraparound** — если не vacuum, через 2 млрд транзакций данные "исчезнут"
>
> **Мониторинг:**
> ```sql
> SELECT relname, n_dead_tup, n_live_tup,
>        round(n_dead_tup::numeric / greatest(n_live_tup, 1) * 100, 2) as dead_pct
> FROM pg_stat_user_tables
> ORDER BY n_dead_tup DESC;
> ```

> [!question]- 3. Объясни VACUUM и VACUUM FULL. Когда использовать какой?
>
> **VACUUM (обычный)**:
> - Помечает место от dead tuples как reusable
> - Обновляет **visibility map** (ускоряет index-only scans)
> - Обновляет **freeze map** (предотвращает TX ID wraparound)
> - **Не возвращает** место ОС — таблица не уменьшается
> - Не блокирует reads/writes (concurrent-safe)
>
> **VACUUM FULL**:
> - Перезаписывает таблицу полностью — компактно
> - **Возвращает** место ОС
> - Требует **ACCESS EXCLUSIVE LOCK** — таблица недоступна
> - Очень медленный, требует место для копии таблицы
>
> **Когда VACUUM FULL:** только при extreme bloat (>50% dead), в maintenance window.
> **В 99% случаев:** обычный VACUUM через autovacuum достаточно.

> [!warning]- Частая ошибка
> "Давайте сделаем VACUUM FULL на production таблице в 100GB."
> Это заблокирует таблицу на часы. Вместо этого: `pg_repack` — online rebuild без lock.

> [!question]- 4. Как настроить autovacuum? Какие параметры критичны?
>
> **autovacuum** автоматически запускает VACUUM когда dead tuples накопились.
>
> Критичные параметры:
> ```
> autovacuum_vacuum_threshold = 50        # мин. dead tuples для trigger
> autovacuum_vacuum_scale_factor = 0.2    # + 20% от размера таблицы
> autovacuum_analyze_threshold = 50       # trigger для ANALYZE
> autovacuum_analyze_scale_factor = 0.1   # + 10% от размера таблицы
> autovacuum_naptime = 1min               # интервал проверки
> ```
>
> **Формула trigger:** `threshold + scale_factor × table_rows`
>
> **Для больших таблиц:** `scale_factor` нужно уменьшить.
> Таблица в 100М строк: trigger при 20М dead tuples — слишком поздно.
> ```sql
> ALTER TABLE big_table SET (autovacuum_vacuum_scale_factor = 0.01);
> ```

> [!question]- 5. Что такое Transaction ID Wraparound? Почему это катастрофа?
>
> PostgreSQL использует 32-bit Transaction ID (XID). 2^32 ≈ 4.3 млрд.
>
> XID space — **circular**: после 2 млрд транзакций от текущей, старые XID выглядят как "в будущем".
> Это значит: данные с "будущим" xmin **невидимы** — как будто не существуют.
>
> **Защита:** VACUUM замораживает (freeze) старые XID, заменяя на `FrozenTransactionId`.
>
> **Если autovacuum не справляется:**
> PostgreSQL переходит в **shutdown mode** при приближении к wraparound.
> Система отказывается принимать новые транзакции. Single-user mode для emergency VACUUM.
>
> **Мониторинг:**
> ```sql
> SELECT datname,
>        age(datfrozenxid) as xid_age,
>        2000000000 - age(datfrozenxid) as remaining
> FROM pg_database;
> -- remaining < 10M = красная тревога
> ```

> [!question]- 6. Как работают транзакции в PostgreSQL? Объясни ACID.
>
> **A**tomicity — транзакция либо целиком, либо откат (WAL + undo через MVCC)
> **C**onsistency — constraints проверяются в конце (или deferred)
> **I**solation — MVCC обеспечивает snapshot isolation
> **D**urability — WAL гарантирует выживание после crash
>
> **Важные команды:**
> ```sql
> BEGIN;                        -- начать транзакцию
> SAVEPOINT sp1;                -- savepoint внутри транзакции
> ROLLBACK TO SAVEPOINT sp1;    -- откат к savepoint (не вся транзакция)
> COMMIT;                       -- зафиксировать
> ```
>
> **READ ONLY транзакция:**
> ```sql
> BEGIN READ ONLY;
> -- не может выполнять INSERT/UPDATE/DELETE
> -- оптимизирует snapshot management
> ```

> [!question]- 7. Что такое WAL (Write-Ahead Log)? Зачем он нужен?
>
> **WAL** — лог всех изменений, записывается **до** применения к data files.
>
> **Зачем:**
> 1. **Durability** — после crash, PostgreSQL реплеит WAL для восстановления
> 2. **Репликация** — streaming replication отправляет WAL на replica
> 3. **Point-in-time recovery** — восстановление до любого момента из архивных WAL
>
> **Процесс записи:**
> 1. Изменение записывается в WAL buffer
> 2. `COMMIT` → WAL buffer flush на диск (`fsync`)
> 3. Затем (позже) dirty pages из shared buffers записываются в data files (checkpoint)
>
> **Параметры:**
> - `wal_level` = replica (default) / logical
> - `max_wal_size` = 1GB — trigger checkpoint
> - `synchronous_commit` = on / off (trade durability for speed)

> [!warning]- Подвох на интервью
> "Можно ли потерять данные при `synchronous_commit = off`?"
> **Да**: при crash между commit и flush на диск.
> Но транзакция **не** будет частично применена — она просто исчезнет.

> [!question]- 8. JSON vs JSONB — в чём разница? Когда что использовать?
>
> | | JSON | JSONB |
> |---|------|-------|
> | Хранение | текст as-is | бинарный формат |
> | Скорость записи | быстрее (нет парсинга) | медленнее (парсинг при INSERT) |
> | Скорость чтения | медленнее (парсинг при SELECT) | быстрее (уже распарсен) |
> | Индексирование | нет | GIN индексы (@>, ?, ?&) |
> | Дубликаты ключей | сохраняет | удаляет (последний wins) |
> | Порядок ключей | сохраняет | не сохраняет |
>
> **Правило: используй JSONB в 99% случаев.**
> JSON только если критично сохранение exact formatting или порядка ключей.
>
> ```sql
> CREATE INDEX idx_data ON events USING GIN (data);
> SELECT * FROM events WHERE data @> '{"type": "click"}';
> ```

> [!question]- 9. Что такое HOT update (Heap-Only Tuple)? Когда он работает?
>
> **HOT update** — оптимизация: новая версия строки хранится на той же heap page.
> Индексы **не обновляются** — pointer из старого tuple ведёт к новому.
>
> **Условия для HOT:**
> 1. Новый tuple помещается на **ту же page** (достаточно места)
> 2. **Ни один индексированный столбец** не изменился
>
> **Преимущества:**
> - Нет обновления индексов → быстрее
> - Меньше index bloat
> - Меньше WAL записей
>
> **Мониторинг:**
> ```sql
> SELECT relname, n_tup_upd, n_tup_hot_upd,
>        round(n_tup_hot_upd::numeric / greatest(n_tup_upd, 1) * 100) as hot_pct
> FROM pg_stat_user_tables
> WHERE n_tup_upd > 0;
> -- hot_pct > 90% = хорошо
> ```

> [!question]- 10. Объясни pg_stat_statements. Как использовать для оптимизации?
>
> `pg_stat_statements` — extension, собирает статистику по всем SQL запросам:
>
> ```sql
> CREATE EXTENSION IF NOT EXISTS pg_stat_statements;
>
> SELECT query,
>        calls,
>        total_exec_time / 1000 as total_sec,
>        mean_exec_time as avg_ms,
>        rows
> FROM pg_stat_statements
> ORDER BY total_exec_time DESC
> LIMIT 10;
> ```
>
> **Что смотреть:**
> - `total_exec_time` — самые "дорогие" запросы суммарно
> - `mean_exec_time` — самые медленные по среднему
> - `calls` — самые частые запросы
> - `rows / calls` — среднее rows per call (слишком много = missing index?)
>
> **Best practice:** установи на каждый production сервер. Reset статистики периодически.
