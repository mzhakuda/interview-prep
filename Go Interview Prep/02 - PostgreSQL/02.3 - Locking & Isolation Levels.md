# Locking & Isolation Levels

Связанные темы: [[02.1 - MVCC & Transactions]] | [[02.2 - Indexes Deep Dive]] | [[02.4 - Performance & Connection Pooling]]

---

> [!question]- 1. Какие уровни изоляции есть в PostgreSQL? Чем отличаются от стандарта SQL?
>
> | Уровень | Dirty Read | Non-Repeatable | Phantom Read | Serialization Anomaly |
> |---------|-----------|----------------|--------------|----------------------|
> | READ UNCOMMITTED | невозможен* | возможен | возможен | возможен |
> | **READ COMMITTED** | невозможен | возможен | возможен | возможен |
> | **REPEATABLE READ** | невозможен | невозможен | невозможен** | возможен |
> | **SERIALIZABLE** | невозможен | невозможен | невозможен | невозможен |
>
> **\*** PostgreSQL не реализует READ UNCOMMITTED — работает как READ COMMITTED.
> **\*\*** PostgreSQL REPEATABLE READ строже SQL стандарта: phantom reads тоже невозможны (snapshot isolation).
>
> **Default**: READ COMMITTED.

---

> [!question]- 2. Как работает READ COMMITTED? Какие проблемы может создать?
>
> Каждый **оператор** (не транзакция!) видит свой snapshot.
>
> ```sql
> BEGIN;
> SELECT * FROM accounts WHERE id = 1;  -- видит баланс 100
> -- другая транзакция: UPDATE accounts SET balance = 50 WHERE id = 1; COMMIT;
> SELECT * FROM accounts WHERE id = 1;  -- видит баланс 50 (новый snapshot!)
> COMMIT;
> ```
>
> **Проблемы:**
> - **Non-repeatable read**: повторный SELECT возвращает другие данные
> - **Lost update**: две транзакции читают, обе обновляют — одна перезаписывает другую
>
> **Lost update пример:**
> ```sql
> -- TX1: SELECT balance FROM accounts WHERE id=1; → 100
> -- TX2: SELECT balance FROM accounts WHERE id=1; → 100
> -- TX1: UPDATE accounts SET balance = 100 + 10 WHERE id=1; → 110
> -- TX2: UPDATE accounts SET balance = 100 + 20 WHERE id=1; → 120 (TX1 потеряна!)
> ```

---

> [!question]- 3. Как работает REPEATABLE READ? Когда получишь ошибку сериализации?
>
> Вся транзакция видит **один snapshot** (на момент первого non-transaction-control statement).
>
> Если две REPEATABLE READ транзакции изменяют **ту же строку**:
> - Первая коммитит успешно
> - Вторая получает: `ERROR: could not serialize access due to concurrent update`
>
> **Обработка:** retry loop:
> ```go
> for retries := 0; retries < 3; retries++ {
>     err := doTransaction(ctx)
>     if err == nil { return nil }
>     if isSerializationError(err) { continue }
>     return err
> }
> ```

> [!question]- 4. SELECT FOR UPDATE vs SELECT FOR SHARE — разница и use cases.
>
> **SELECT FOR UPDATE** — exclusive row lock:
> - Другие `FOR UPDATE` и `UPDATE/DELETE` **блокируются**
> - Другие обычные `SELECT` **не блокируются** (MVCC!)
> - Use case: "я собираюсь обновить эту строку"
>
> **SELECT FOR SHARE** — shared row lock:
> - Другие `FOR SHARE` **разрешены**
> - `FOR UPDATE` и `UPDATE/DELETE` **блокируются**
> - Use case: "убедиться что строка не изменится пока я читаю связанные данные"
>
> **Варианты:**
> - `FOR UPDATE NOWAIT` — ошибка вместо ожидания
> - `FOR UPDATE SKIP LOCKED` — пропустить заблокированные строки (очереди!)
>
> ```sql
> -- Job queue pattern:
> SELECT * FROM jobs WHERE status = 'pending'
> ORDER BY created_at
> LIMIT 1
> FOR UPDATE SKIP LOCKED;
> ```

> [!hint]- SKIP LOCKED — мощный паттерн
> Идеален для job queues в PostgreSQL. Каждый worker берёт незаблокированную задачу,
> без deadlock, без ожидания. Альтернатива внешним очередям для простых случаев.

> [!question]- 4.1. SELECT FOR UPDATE — зачем нужен и как его готовить?
>
> ### Зачем нужен
>
> На READ COMMITTED (default) обычный `SELECT` видит snapshot, но **не блокирует строку**. Между `SELECT` и `UPDATE` другая транзакция может влезть → **lost update** или нарушение инварианта.
>
> `SELECT ... FOR UPDATE` решает три задачи:
> 1. **Защита read-modify-write** от гонок (списание баланса, резерв товара, инкремент счётчика).
> 2. **Enforce бизнес-инвариантов** которые нельзя выразить через UNIQUE/CHECK (например, "сумма дочерних ≤ лимит родителя").
> 3. **Материализация конфликта** — явно сериализовать транзакции на конкретной строке без перехода на SERIALIZABLE.
>
> ### Классический кейс: списание баланса
>
> ```sql
> BEGIN;
> -- блокируем строку до конца транзакции
> SELECT balance FROM accounts WHERE id = $1 FOR UPDATE;
> -- проверяем инвариант в коде: balance >= amount
> UPDATE accounts SET balance = balance - $2 WHERE id = $1;
> INSERT INTO ledger (account_id, delta) VALUES ($1, -$2);
> COMMIT;
> ```
>
> Без `FOR UPDATE` два параллельных списания прочитают 100, оба решат что хватает, и спишут дважды.
>
> ### Варианты (обязательно знать)
>
> | Вариант | Поведение | Когда использовать |
> |---|---|---|
> | `FOR UPDATE` | ждёт освобождения блокировки | default, транзакционная работа |
> | `FOR NO KEY UPDATE` | слабее, не блокирует FK-ссылки | UPDATE не-ключевых колонок |
> | `FOR SHARE` | shared, другие могут читать с FOR SHARE | "читаю + нужно что не изменится" |
> | `FOR KEY SHARE` | самый слабый | FK-checks (внутри PG) |
> | `FOR UPDATE NOWAIT` | моментальная ошибка вместо ожидания | UI / интерактив, где нельзя висеть |
> | `FOR UPDATE SKIP LOCKED` | пропускает залоченные строки | **job queue**, worker pool |
>
> ### Как готовить: правила
>
> 1. **Только внутри транзакции.** Вне `BEGIN...COMMIT` lock снимается сразу — бессмысленно.
> 2. **Блокируй в детерминированном порядке** (обычно `ORDER BY id`) — иначе deadlock при параллельных батчах.
> 3. **Держи транзакцию короткой.** Пока ты держишь FOR UPDATE, другие ждут → connection pool starvation.
> 4. **Никаких сетевых вызовов / долгих операций** под локом (HTTP, Kafka publish, external API).
> 5. **LIMIT + ORDER BY для очередей** всегда с `SKIP LOCKED`:
>    ```sql
>    SELECT id FROM jobs
>    WHERE status = 'pending'
>    ORDER BY created_at
>    LIMIT 10
>    FOR UPDATE SKIP LOCKED;
>    ```
> 6. **`NOWAIT` для интерактива** — юзер не должен висеть 30 секунд; лучше сразу "попробуйте ещё раз".
> 7. **FOR UPDATE на JOIN** блокирует строки **всех** таблиц из FROM. Чтобы лочить только одну — `FOR UPDATE OF table_name`.
>    ```sql
>    SELECT o.*, u.email FROM orders o JOIN users u ON u.id = o.user_id
>    WHERE o.id = $1 FOR UPDATE OF o;   -- лочим только orders
>    ```
>
> ### Типичные ошибки
>
> - **FOR UPDATE + агрегаты / GROUP BY / DISTINCT** — ошибка, нельзя.
> - **FOR UPDATE без WHERE** — лок на всю таблицу в виде row-locks. Плохо.
> - **Читать с FOR UPDATE, а потом долго думать в приложении.** Транзакция должна быть "взял → посчитал → записал → COMMIT".
> - **FOR UPDATE на REPEATABLE READ** не защищает от serialization failure — транзакция всё равно может откатиться с `could not serialize access`.
> - **Phantom rows**: `FOR UPDATE` лочит только **существующие** строки. Новую строку, вставленную параллельной транзакцией, он не заблокирует. Для защиты от phantoms → SERIALIZABLE или advisory lock на "ключ диапазона".
>
> ### Go-паттерн с retry на deadlock
>
> ```go
> func withdraw(ctx context.Context, db *sql.DB, accountID int64, amount int64) error {
>     return runInTx(ctx, db, func(tx *sql.Tx) error {
>         var balance int64
>         err := tx.QueryRowContext(ctx,
>             `SELECT balance FROM accounts WHERE id = $1 FOR UPDATE`,
>             accountID,
>         ).Scan(&balance)
>         if err != nil { return err }
>
>         if balance < amount {
>             return ErrInsufficientFunds
>         }
>         _, err = tx.ExecContext(ctx,
>             `UPDATE accounts SET balance = balance - $1 WHERE id = $2`,
>             amount, accountID)
>         return err
>     })
> }
> ```
>
> `runInTx` должен ловить `SQLSTATE 40P01` (deadlock_detected) и `40001` (serialization_failure) → retry с backoff.
>
> ### FOR UPDATE vs альтернативы
>
> - **vs optimistic (version column)** — FOR UPDATE лучше при high contention (не тратим работу на retry); optimistic лучше когда конфликты редки.
> - **vs advisory lock** — FOR UPDATE привязан к строке и автоматически снимается при COMMIT; advisory гибче (любой int64 как ключ), но не связан с данными.
> - **vs SERIALIZABLE** — FOR UPDATE точечный и дешёвый; SERIALIZABLE глобальный, даёт гарантию но требует retry везде.
>
> ### Связь с уровнями изоляции (частое заблуждение)
>
> FOR UPDATE работает на **любом** уровне, это не часть SERIALIZABLE. Наоборот — это **альтернатива** глобальной сериализации: остаёшься на дешёвом READ COMMITTED и точечно блокируешь нужные строки.
>
> | Уровень | Роль FOR UPDATE |
> |---|---|
> | **READ COMMITTED** (default) | основной use case: защита read-modify-write от lost update |
> | **REPEATABLE READ** | помогает избежать `could not serialize access` при конкурентном UPDATE |
> | **SERIALIZABLE** | почти не нужен — SSI сам ловит rw-конфликты; FOR UPDATE добавит overhead |
>
> Две философии конкурентности:
> - **Pessimistic** (FOR UPDATE на RC) — "я знаю где будет конфликт, блокирую точечно". Дёшево, предсказуемо.
> - **Optimistic + SERIALIZABLE** — "пусть БД всё отследит, если что — retry". Нужен retry-loop везде.

> [!question]- 4.2. FOR UPDATE и нагрузка: heavy-read vs heavy-write contention
>
> **Частое заблуждение:** "FOR UPDATE плох для heavy-read". На самом деле — нет, проблема в другом.
>
> ### Heavy-READ: FOR UPDATE НЕ мешает
>
> Благодаря MVCC обычный `SELECT` **не конфликтует** с `FOR UPDATE`. Читатели видят свой snapshot, писатель держит row-lock — они друг друга не блокируют.
>
> ```sql
> -- TX1: BEGIN; SELECT ... WHERE id=1 FOR UPDATE;  -- держит lock
> -- TX2..TX1000: SELECT ... WHERE id=1;            -- НЕ ждут, читают старую версию
> ```
>
> Тысячи параллельных читателей спокойно работают со строкой, которую один writer залочил. Это сила MVCC в PostgreSQL: **readers don't block writers, writers don't block readers**.
>
> ### Heavy-WRITE contention на одной строке — вот где больно
>
> FOR UPDATE становится бутылочным горлышком когда конкурирующие транзакции **пишут в ту же строку**:
>
> | Ситуация | Проблема |
> |---|---|
> | **Hot row** (глобальный счётчик, один баланс, лимит) | writers выстраиваются в очередь → throughput ≈ 1/latency_tx |
> | **Длинная транзакция под локом** | HTTP/Kafka/external API под FOR UPDATE — всё встало |
> | **Lock + connection pool** | ожидающие tx съедают коннекты → pool exhaustion → 503 наружу |
> | **Неправильный порядок блокировок** | deadlocks при параллельных батчах |
> | **FOR UPDATE без LIMIT на большой выборке** | залочили пол-таблицы, массовое ожидание |
>
> ### Как понять что это твой случай
>
> - В `pg_stat_activity` видишь `wait_event_type = Lock`, `wait_event = transactionid` → транзакции ждут row lock.
> - В `pg_locks` одна строка залочена, очередь ожидающих растёт.
> - Latency апдейтов прыгает с миллисекунд до секунд под нагрузкой.
> - Метрика `tup_updated` по таблице низкая при высоком количестве попыток.
>
> ### Контрмеры для hot row / heavy write
>
> 1. **Шардинг счётчика** — `N` строк вместо одной, randomize на вставке, sum на чтении:
>    ```sql
>    INSERT INTO counter_shards (shard, delta) VALUES (random_int(0, 15), 1);
>    -- чтение: SELECT sum(delta) FROM counter_shards WHERE metric_id = $1;
>    ```
> 2. **Append-only log + background aggregation** — пишем события, отдельный воркер агрегирует. Eventual consistency, но без contention.
> 3. **SKIP LOCKED** — если это очередь задач, каждый воркер берёт свою строку без ожидания.
> 4. **Advisory lock на бизнес-ключ** — иногда дешевле чем row lock, и можно блокировать "что-то чего ещё нет в БД".
> 5. **Optimistic locking** — если конфликты редки: version column + retry. Не платишь за lock когда всё ок.
> 6. **Batching** — группируй N апдейтов в одну транзакцию (амортизация стоимости lock acquire/release).
> 7. **Партиционирование горячей сущности** — user_balance → user_balance_shard_{0..15}.
> 8. **Убрать долгие операции из критической секции** — сначала всё что можно, потом короткая tx с FOR UPDATE и COMMIT.
>
> ### Диагностический чеклист
>
> ```sql
> -- Кто кого ждёт
> SELECT blocked.pid AS blocked_pid, blocking.pid AS blocking_pid,
>        blocked.query AS blocked_query, blocking.query AS blocking_query
> FROM pg_stat_activity blocked
> JOIN pg_stat_activity blocking ON blocking.pid = ANY(pg_blocking_pids(blocked.pid))
> WHERE blocked.wait_event_type = 'Lock';
>
> -- Долгоживущие транзакции (потенциальные держатели локов)
> SELECT pid, now() - xact_start AS duration, state, query
> FROM pg_stat_activity
> WHERE xact_start IS NOT NULL
> ORDER BY xact_start ASC;
> ```
>
> ### TL;DR
>
> - **Heavy read + FOR UPDATE = норм** (MVCC решает).
> - **Heavy write на одной строке = бутылочное горло** — никакой трюк с FOR UPDATE это не исправит, нужно менять схему (шардинг / append-only / очередь).
> - **Длинные транзакции под локом = катастрофа** — держи tx в миллисекундах, никаких внешних вызовов внутри.

> [!question]- 5. Как PostgreSQL обнаруживает и решает deadlocks?
>
> PostgreSQL запускает **deadlock detector** каждые `deadlock_timeout` (default 1s).
>
> **Алгоритм:**
> 1. Строит граф ожидания (wait-for graph)
> 2. Если находит цикл → deadlock
> 3. Выбирает одну транзакцию для отката (victim) — обычно ту что выполнила меньше работы
> 4. Victim получает: `ERROR: deadlock detected`
>
> **Как избежать deadlocks:**
> 1. **Всегда блокируй строки в одном порядке** (например, по ID ASC)
> 2. Короткие транзакции → меньше вероятность
> 3. Используй `NOWAIT` или `SKIP LOCKED` где возможно
> 4. Избегай explicit locks где MVCC достаточно

> [!example]- Код: классический deadlock
> ```sql
> -- TX1:
> BEGIN;
> UPDATE accounts SET balance = balance - 100 WHERE id = 1; -- lock row 1
> -- ждёт TX2...
> UPDATE accounts SET balance = balance + 100 WHERE id = 2; -- хочет lock row 2
>
> -- TX2 (одновременно):
> BEGIN;
> UPDATE accounts SET balance = balance - 50 WHERE id = 2;  -- lock row 2
> -- ждёт TX1...
> UPDATE accounts SET balance = balance + 50 WHERE id = 1;  -- хочет lock row 1
> -- DEADLOCK! Один из них будет aborted.
> ```

> [!question]- 6. Какие типы table-level locks есть в PostgreSQL?
>
> | Lock Mode | Конфликтует с | Типичная операция |
> |-----------|---------------|-------------------|
> | ACCESS SHARE | ACCESS EXCLUSIVE | `SELECT` |
> | ROW SHARE | EXCLUSIVE, ACCESS EXCLUSIVE | `SELECT FOR UPDATE/SHARE` |
> | ROW EXCLUSIVE | SHARE, SHARE ROW EXCLUSIVE, EXCLUSIVE, ACCESS EXCLUSIVE | `INSERT, UPDATE, DELETE` |
> | SHARE | ROW EXCLUSIVE, SHARE UPDATE EXCLUSIVE, EXCLUSIVE, ACCESS EXCLUSIVE | `CREATE INDEX (not CONCURRENTLY)` |
> | ACCESS EXCLUSIVE | все | `ALTER TABLE, DROP TABLE, VACUUM FULL, REINDEX` |
>
> **Ключевое**: обычный `SELECT` берёт ACCESS SHARE — не конфликтует ни с чем кроме ACCESS EXCLUSIVE.
> Поэтому `ALTER TABLE` на production = проблема (ждёт завершения всех транзакций).

> [!warning]- ALTER TABLE на production
> `ALTER TABLE ADD COLUMN` с default значением: до PostgreSQL 11 — перезаписывает всю таблицу (ACCESS EXCLUSIVE на время).
> С PostgreSQL 11+ — мгновенно (default хранится в pg_attribute).
> Но `ALTER TABLE ... SET NOT NULL` с CHECK — по-прежнему full scan.

> [!question]- 7. Что такое Advisory Locks? Когда их использовать?
>
> **Advisory locks** — application-level locks в PostgreSQL (не привязаны к строкам/таблицам).
>
> ```sql
> -- Session-level lock (держится до конца сессии или explicit unlock)
> SELECT pg_advisory_lock(12345);
> -- ... do work ...
> SELECT pg_advisory_unlock(12345);
>
> -- Transaction-level lock (освобождается при COMMIT/ROLLBACK)
> SELECT pg_advisory_xact_lock(12345);
>
> -- Non-blocking попытка
> SELECT pg_try_advisory_lock(12345); -- returns true/false
> ```
>
> **Use cases:**
> - Distributed locking (если все используют один PostgreSQL)
> - Предотвращение concurrent cron jobs
> - Application-level mutex
> - Leader election

> [!question]- 8. SERIALIZABLE isolation — как работает SSI?
>
> **SSI** (Serializable Snapshot Isolation) — механизм в PostgreSQL для SERIALIZABLE level.
>
> Работает поверх snapshot isolation, отслеживая **rw-dependencies** между транзакциями:
> - **rw-conflict**: TX1 читает данные, TX2 записывает в тот же набор
> - Если обнаружен **cycle** в dependency graph → одна транзакция отклоняется
>
> **Overhead**: отслеживание всех read/write sets → дополнительная память и CPU.
> **Ложные срабатывания**: возможны (SSI conservative) → retry нужен.
>
> **Когда использовать:**
> - Financial transactions с complex invariants
> - Когда application-level locking слишком сложен
> - Когда нужна **гарантия** что concurrent execution = serial execution

> [!question]- 9. Как работает VACUUM FREEZE? Зачем замораживать строки?
>
> **Freeze** заменяет `xmin` строки на специальное значение `FrozenTransactionId`.
> После freeze, строка считается видимой **всем** транзакциям — xmin больше не важен.
>
> **Зачем:** предотвращение Transaction ID Wraparound (см. [[02.1 - MVCC & Transactions]]).
>
> ```sql
> -- Параметры:
> vacuum_freeze_min_age = 50000000     -- мин. возраст XID для freeze
> vacuum_freeze_table_age = 150000000  -- trigger для aggressive vacuum
> autovacuum_freeze_max_age = 200000000 -- force vacuum (даже если autovacuum выключен)
> ```
>
> При `age(relfrozenxid)` приближается к `autovacuum_freeze_max_age`:
> autovacuum запускает **aggressive VACUUM** (сканирует все страницы, не только dirty).

> [!question]- 10. Как реализовать optimistic locking в PostgreSQL?
>
> **Optimistic locking** — не блокируем строку, а проверяем при UPDATE:
>
> ```sql
> -- Читаем с version
> SELECT id, data, version FROM records WHERE id = 1;
> -- version = 5
>
> -- Обновляем только если version не изменился
> UPDATE records
> SET data = 'new', version = version + 1
> WHERE id = 1 AND version = 5;
>
> -- Если 0 rows affected → кто-то изменил раньше → retry
> ```
>
> **В Go:**
> ```go
> result, err := db.ExecContext(ctx,
>     "UPDATE records SET data=$1, version=version+1 WHERE id=$2 AND version=$3",
>     newData, id, currentVersion)
> rows, _ := result.RowsAffected()
> if rows == 0 {
>     return ErrConflict // retry
> }
> ```
>
> **Vs Pessimistic** (SELECT FOR UPDATE): optimistic лучше при low contention, pessimistic — при high contention.
