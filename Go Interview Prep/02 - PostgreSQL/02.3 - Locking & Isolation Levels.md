# Locking & Isolation Levels

Связанные темы: [[02.1 - MVCC & Transactions]] | [[02.2 - Indexes Deep Dive]] | [[02.4 - Performance & Connection Pooling]]

---

> [!question]- 1. Какие уровни изоляции есть в PostgreSQL? Чем отличаются от стандарта SQL?
>
> | Уровень | Dirty Read | Non-Repeatable | Phantom Read | Serialization Anomaly |
> |---------|-----------|----------------|--------------|----------------------|
> | READ UNCOMMITTED | невозможен* | возможен | возможен | возможен |
> | **READ COMMITTED** | невозможен | возможен | возможен | возможен |
> | **REPEATABLE READ** | невозможен | невозможен | невозможен** | возможен |
> | **SERIALIZABLE** | невозможен | невозможен | невозможен | невозможен |
>
> **\*** PostgreSQL не реализует READ UNCOMMITTED — работает как READ COMMITTED.
> **\*\*** PostgreSQL REPEATABLE READ строже SQL стандарта: phantom reads тоже невозможны (snapshot isolation).
>
> **Default**: READ COMMITTED.

---

> [!question]- 2. Как работает READ COMMITTED? Какие проблемы может создать?
>
> Каждый **оператор** (не транзакция!) видит свой snapshot.
>
> ```sql
> BEGIN;
> SELECT * FROM accounts WHERE id = 1;  -- видит баланс 100
> -- другая транзакция: UPDATE accounts SET balance = 50 WHERE id = 1; COMMIT;
> SELECT * FROM accounts WHERE id = 1;  -- видит баланс 50 (новый snapshot!)
> COMMIT;
> ```
>
> **Проблемы:**
> - **Non-repeatable read**: повторный SELECT возвращает другие данные
> - **Lost update**: две транзакции читают, обе обновляют — одна перезаписывает другую
>
> **Lost update пример:**
> ```sql
> -- TX1: SELECT balance FROM accounts WHERE id=1; → 100
> -- TX2: SELECT balance FROM accounts WHERE id=1; → 100
> -- TX1: UPDATE accounts SET balance = 100 + 10 WHERE id=1; → 110
> -- TX2: UPDATE accounts SET balance = 100 + 20 WHERE id=1; → 120 (TX1 потеряна!)
> ```

---

> [!question]- 3. Как работает REPEATABLE READ? Когда получишь ошибку сериализации?
>
> Вся транзакция видит **один snapshot** (на момент первого non-transaction-control statement).
>
> Если две REPEATABLE READ транзакции изменяют **ту же строку**:
> - Первая коммитит успешно
> - Вторая получает: `ERROR: could not serialize access due to concurrent update`
>
> **Обработка:** retry loop:
> ```go
> for retries := 0; retries < 3; retries++ {
>     err := doTransaction(ctx)
>     if err == nil { return nil }
>     if isSerializationError(err) { continue }
>     return err
> }
> ```

> [!question]- 4. SELECT FOR UPDATE vs SELECT FOR SHARE — разница и use cases.
>
> **SELECT FOR UPDATE** — exclusive row lock:
> - Другие `FOR UPDATE` и `UPDATE/DELETE` **блокируются**
> - Другие обычные `SELECT` **не блокируются** (MVCC!)
> - Use case: "я собираюсь обновить эту строку"
>
> **SELECT FOR SHARE** — shared row lock:
> - Другие `FOR SHARE` **разрешены**
> - `FOR UPDATE` и `UPDATE/DELETE` **блокируются**
> - Use case: "убедиться что строка не изменится пока я читаю связанные данные"
>
> **Варианты:**
> - `FOR UPDATE NOWAIT` — ошибка вместо ожидания
> - `FOR UPDATE SKIP LOCKED` — пропустить заблокированные строки (очереди!)
>
> ```sql
> -- Job queue pattern:
> SELECT * FROM jobs WHERE status = 'pending'
> ORDER BY created_at
> LIMIT 1
> FOR UPDATE SKIP LOCKED;
> ```

> [!hint]- SKIP LOCKED — мощный паттерн
> Идеален для job queues в PostgreSQL. Каждый worker берёт незаблокированную задачу,
> без deadlock, без ожидания. Альтернатива внешним очередям для простых случаев.

> [!question]- 5. Как PostgreSQL обнаруживает и решает deadlocks?
>
> PostgreSQL запускает **deadlock detector** каждые `deadlock_timeout` (default 1s).
>
> **Алгоритм:**
> 1. Строит граф ожидания (wait-for graph)
> 2. Если находит цикл → deadlock
> 3. Выбирает одну транзакцию для отката (victim) — обычно ту что выполнила меньше работы
> 4. Victim получает: `ERROR: deadlock detected`
>
> **Как избежать deadlocks:**
> 1. **Всегда блокируй строки в одном порядке** (например, по ID ASC)
> 2. Короткие транзакции → меньше вероятность
> 3. Используй `NOWAIT` или `SKIP LOCKED` где возможно
> 4. Избегай explicit locks где MVCC достаточно

> [!example]- Код: классический deadlock
> ```sql
> -- TX1:
> BEGIN;
> UPDATE accounts SET balance = balance - 100 WHERE id = 1; -- lock row 1
> -- ждёт TX2...
> UPDATE accounts SET balance = balance + 100 WHERE id = 2; -- хочет lock row 2
>
> -- TX2 (одновременно):
> BEGIN;
> UPDATE accounts SET balance = balance - 50 WHERE id = 2;  -- lock row 2
> -- ждёт TX1...
> UPDATE accounts SET balance = balance + 50 WHERE id = 1;  -- хочет lock row 1
> -- DEADLOCK! Один из них будет aborted.
> ```

> [!question]- 6. Какие типы table-level locks есть в PostgreSQL?
>
> | Lock Mode | Конфликтует с | Типичная операция |
> |-----------|---------------|-------------------|
> | ACCESS SHARE | ACCESS EXCLUSIVE | `SELECT` |
> | ROW SHARE | EXCLUSIVE, ACCESS EXCLUSIVE | `SELECT FOR UPDATE/SHARE` |
> | ROW EXCLUSIVE | SHARE, SHARE ROW EXCLUSIVE, EXCLUSIVE, ACCESS EXCLUSIVE | `INSERT, UPDATE, DELETE` |
> | SHARE | ROW EXCLUSIVE, SHARE UPDATE EXCLUSIVE, EXCLUSIVE, ACCESS EXCLUSIVE | `CREATE INDEX (not CONCURRENTLY)` |
> | ACCESS EXCLUSIVE | все | `ALTER TABLE, DROP TABLE, VACUUM FULL, REINDEX` |
>
> **Ключевое**: обычный `SELECT` берёт ACCESS SHARE — не конфликтует ни с чем кроме ACCESS EXCLUSIVE.
> Поэтому `ALTER TABLE` на production = проблема (ждёт завершения всех транзакций).

> [!warning]- ALTER TABLE на production
> `ALTER TABLE ADD COLUMN` с default значением: до PostgreSQL 11 — перезаписывает всю таблицу (ACCESS EXCLUSIVE на время).
> С PostgreSQL 11+ — мгновенно (default хранится в pg_attribute).
> Но `ALTER TABLE ... SET NOT NULL` с CHECK — по-прежнему full scan.

> [!question]- 7. Что такое Advisory Locks? Когда их использовать?
>
> **Advisory locks** — application-level locks в PostgreSQL (не привязаны к строкам/таблицам).
>
> ```sql
> -- Session-level lock (держится до конца сессии или explicit unlock)
> SELECT pg_advisory_lock(12345);
> -- ... do work ...
> SELECT pg_advisory_unlock(12345);
>
> -- Transaction-level lock (освобождается при COMMIT/ROLLBACK)
> SELECT pg_advisory_xact_lock(12345);
>
> -- Non-blocking попытка
> SELECT pg_try_advisory_lock(12345); -- returns true/false
> ```
>
> **Use cases:**
> - Distributed locking (если все используют один PostgreSQL)
> - Предотвращение concurrent cron jobs
> - Application-level mutex
> - Leader election

> [!question]- 8. SERIALIZABLE isolation — как работает SSI?
>
> **SSI** (Serializable Snapshot Isolation) — механизм в PostgreSQL для SERIALIZABLE level.
>
> Работает поверх snapshot isolation, отслеживая **rw-dependencies** между транзакциями:
> - **rw-conflict**: TX1 читает данные, TX2 записывает в тот же набор
> - Если обнаружен **cycle** в dependency graph → одна транзакция отклоняется
>
> **Overhead**: отслеживание всех read/write sets → дополнительная память и CPU.
> **Ложные срабатывания**: возможны (SSI conservative) → retry нужен.
>
> **Когда использовать:**
> - Financial transactions с complex invariants
> - Когда application-level locking слишком сложен
> - Когда нужна **гарантия** что concurrent execution = serial execution

> [!question]- 9. Как работает VACUUM FREEZE? Зачем замораживать строки?
>
> **Freeze** заменяет `xmin` строки на специальное значение `FrozenTransactionId`.
> После freeze, строка считается видимой **всем** транзакциям — xmin больше не важен.
>
> **Зачем:** предотвращение Transaction ID Wraparound (см. [[02.1 - MVCC & Transactions]]).
>
> ```sql
> -- Параметры:
> vacuum_freeze_min_age = 50000000     -- мин. возраст XID для freeze
> vacuum_freeze_table_age = 150000000  -- trigger для aggressive vacuum
> autovacuum_freeze_max_age = 200000000 -- force vacuum (даже если autovacuum выключен)
> ```
>
> При `age(relfrozenxid)` приближается к `autovacuum_freeze_max_age`:
> autovacuum запускает **aggressive VACUUM** (сканирует все страницы, не только dirty).

> [!question]- 10. Как реализовать optimistic locking в PostgreSQL?
>
> **Optimistic locking** — не блокируем строку, а проверяем при UPDATE:
>
> ```sql
> -- Читаем с version
> SELECT id, data, version FROM records WHERE id = 1;
> -- version = 5
>
> -- Обновляем только если version не изменился
> UPDATE records
> SET data = 'new', version = version + 1
> WHERE id = 1 AND version = 5;
>
> -- Если 0 rows affected → кто-то изменил раньше → retry
> ```
>
> **В Go:**
> ```go
> result, err := db.ExecContext(ctx,
>     "UPDATE records SET data=$1, version=version+1 WHERE id=$2 AND version=$3",
>     newData, id, currentVersion)
> rows, _ := result.RowsAffected()
> if rows == 0 {
>     return ErrConflict // retry
> }
> ```
>
> **Vs Pessimistic** (SELECT FOR UPDATE): optimistic лучше при low contention, pessimistic — при high contention.
