# Performance & Connection Pooling

Связанные темы: [[02.1 - MVCC & Transactions]] | [[02.2 - Indexes Deep Dive]] | [[02.3 - Locking & Isolation Levels]]

---

> [!question]- 1. Как читать EXPLAIN ANALYZE? Какие red flags искать?
>
> ```sql
> EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT) SELECT * FROM orders WHERE user_id = 123;
> ```
>
> **Ключевые метрики:**
>
> - `actual time` — real execution time (startup..total) в ms
> - `rows` — actual rows vs estimated (большое расхождение = плохая статистика)
> - `Buffers: shared hit/read` — hit = из кэша, read = с диска
> - `Planning Time` / `Execution Time`
>
> **Red flags:**
> | Что видишь | Проблема |
> |------------|----------|
> | `Seq Scan` на большой таблице | Missing index |
> | `rows=1000000` estimated, `actual rows=10` | Stale statistics → `ANALYZE` |
> | `Sort Method: external merge` | Не хватает `work_mem` |
> | `Hash Join` с `Batches > 1` | Не хватает `work_mem` |
> | `Nested Loop` с большим outer | Может быть Hash Join лучше |
> | `Bitmap Heap Scan → Recheck Cond` с много lossy blocks | `work_mem` мал для bitmap |

> [!hint]- Совет
> Используй `EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON)` и вставь в explain.dalibo.com для визуализации.

> [!question]- 2. Что такое PgBouncer? Зачем нужен connection pooling?
>
> **Проблема:** каждое соединение PostgreSQL = отдельный backend process (~5-10MB RAM).
> 100 microservice pods × 10 connections = 1000 connections → 10GB RAM только на connections.
>
> **PgBouncer** — lightweight proxy перед PostgreSQL:
>
> - Клиенты подключаются к PgBouncer (тысячи)
> - PgBouncer раздаёт из пула реальных соединений (десятки)
>
> **Режимы:**
> | Режим | Когда отдаёт соединение обратно | Ограничения |
> |-------|-------------------------------|-------------|
> | **Transaction** | После COMMIT/ROLLBACK | Нет SET, LISTEN, prepared statements |
> | Session | При отключении клиента | Как без pooler, наименее эффективен |
> | Statement | После каждого statement | Нет multi-statement транзакций |
>
> **Рекомендация:** Transaction mode для 90% случаев.

> [!warning]- Gotcha с Transaction pooling
> В transaction mode **нельзя** использовать:
>
> - `SET` переменные (сбрасываются между транзакциями)
> - `LISTEN/NOTIFY` (привязаны к session)
> - Server-side prepared statements (пропадают)
> - `DISCARD ALL` не помогает — connection другая
>
> В Go: не используй `database/sql` driver features типа `Conn()` с pooler.

> [!question]- 3. Как оптимизировать slow query? Пошаговый подход.
>
> 1. **Найди slow queries:** `pg_stat_statements` → sort by `total_exec_time`
> 2. **EXPLAIN ANALYZE:** понять plan
> 3. **Check indexes:** есть ли подходящий индекс? Используется ли?
> 4. **Check statistics:** `ANALYZE table_name` если estimated rows расходятся с actual
> 5. **Check work_mem:** `Sort Method: external merge` → увеличь work_mem
> 6. **Rewrite query:** иногда CTE / subquery лучше чем join
> 7. **Partitioning:** если таблица > 100GB и scan только по date range
>
> **Quick wins:**
>
> ```sql
> -- Обновить статистику
> ANALYZE orders;
>
> -- Увеличить work_mem для конкретного запроса
> SET work_mem = '256MB';
> SELECT ... ; -- сложный запрос с сортировкой
> RESET work_mem;
> ```

> [!question]- 4. Partitioning в PostgreSQL — когда и какой тип?
>
> **Декларативное partitioning** (с PostgreSQL 10+):
>
> | Тип   | Критерий               | Use case                                   |
> | ----- | ---------------------- | ------------------------------------------ |
> | RANGE | `created_at` диапазоны | Time-series: logs, events, orders by month |
> | LIST  | `status`, `region`     | Enum values, geographic partitioning       |
> | HASH  | `user_id`              | Равномерное распределение                  |
>
> ```sql
> CREATE TABLE orders (
>     id bigserial,
>     created_at timestamptz,
>     data jsonb
> ) PARTITION BY RANGE (created_at);
>
> CREATE TABLE orders_2024_01 PARTITION OF orders
>     FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');
> ```
>
> **Когда partitioning нужен:**
>
> - Таблица > 100GB и запросы всегда фильтруют по partition key
> - Нужно быстрое удаление старых данных (DROP PARTITION vs DELETE)
> - Parallel query по партициям
>
> **Когда НЕ нужен:**
>
> - Таблица < 10GB — overhead partitioning > benefit
> - Запросы не фильтруют по partition key — все партиции scan

> [!question]- 5. shared_buffers, work_mem, effective_cache_size — как настроить?
>
> **`shared_buffers`** — кэш данных PostgreSQL в RAM:
>
> - Рекомендация: 25% от total RAM (но не более 8-16GB)
> - Больше не помогает — OS page cache эффективнее
>
> **`work_mem`** — память для одной sort/hash операции:
>
> - Default 4MB — очень мало для сложных запросов
> - Рекомендация: `RAM / max_connections / 4` (грубо)
> - Осторожно: каждый sort в query может использовать свою копию
> - 10 параллельных запросов × 5 sort operations × 256MB = 12.8GB
>
> **`effective_cache_size`** — оценка доступного кэша (shared_buffers + OS cache):
>
> - Рекомендация: 50-75% от total RAM
> - Влияет на планировщик: больше → более агрессивное использование индексов

> [!warning]- Частая ошибка с work_mem
> Установить `work_mem = 1GB` глобально.
> Результат: OOM killer при пике нагрузки (каждый sort allocation × connections).
> Лучше: маленький global, большой для конкретных запросов через `SET LOCAL`.

> [!question]- 6. Что такое pg_stat_user_tables? Какие метрики критичны?
>
> ```sql
> SELECT relname,
>        seq_scan, idx_scan,           -- seq vs index scans
>        n_tup_ins, n_tup_upd, n_tup_del, -- write activity
>        n_live_tup, n_dead_tup,       -- live vs dead tuples
>        last_vacuum, last_autovacuum, -- vacuum freshness
>        last_analyze, last_autoanalyze
> FROM pg_stat_user_tables
> ORDER BY seq_scan DESC;
> ```
>
> **Red flags:**
>
> - `seq_scan >> idx_scan` на большой таблице → missing index
> - `n_dead_tup / n_live_tup > 0.2` → autovacuum не справляется
> - `last_autovacuum` давно → проверь autovacuum настройки
> - Много `n_tup_upd` без HOT → добавь INCLUDE или уменьши индексы

> [!question]- 7. Prepared statements в Go — как правильно использовать с PostgreSQL?
>
> **`database/sql` Prepared Statements:**
>
> ```go
> stmt, err := db.PrepareContext(ctx, "SELECT * FROM users WHERE id = $1")
> defer stmt.Close()
> row := stmt.QueryRowContext(ctx, userID)
> ```
>
> **Проблемы с connection pooling:**
>
> - Prepared statement привязан к конкретному backend connection
> - С PgBouncer в transaction mode: prepared statement пропадает при смене connection
>
> **Решение для PgBouncer:**
>
> - Используй `pgx` driver с `PreferSimpleProtocol: true`
> - Или PgBouncer 1.21+ с prepared statement support
>
> **pgx native (без database/sql):**
>
> ```go
> pool, _ := pgxpool.New(ctx, connString)
> rows, _ := pool.Query(ctx, "SELECT * FROM users WHERE id = $1", userID)
> // pgx автоматически кэширует prepared statements per-connection
> ```

> [!question]- 8. Как мониторить PostgreSQL в production?
>
> **Обязательные метрики:**
>
> - Connection count: `SELECT count(*) FROM pg_stat_activity;`
> - Longest transaction: `SELECT max(now() - xact_start) FROM pg_stat_activity WHERE state != 'idle';`
> - Replication lag: `SELECT now() - pg_last_xact_replay_timestamp();`
> - Cache hit ratio: `SELECT sum(heap_blks_hit) / sum(heap_blks_hit + heap_blks_read) FROM pg_statio_user_tables;`
> - Dead tuples ratio per table
> - Lock waits: `SELECT * FROM pg_stat_activity WHERE wait_event_type = 'Lock';`
>
> **Инструменты:**
>
> - `pg_stat_statements` — query performance
> - `pg_stat_activity` — текущие connections и их состояние
> - `pgBadger` — log analyzer
> - Prometheus + `postgres_exporter` + Grafana

> [!question]- 9. CTE (WITH) vs Subquery — когда что?
>
> **С PostgreSQL 12+**: CTE по умолчанию **inlined** (оптимизатор может push down filters).
>
> ```sql
> -- CTE (может быть inlined):
> WITH active_users AS (
>     SELECT * FROM users WHERE status = 'active'
> )
> SELECT * FROM active_users WHERE created_at > '2024-01-01';
>
> -- Если нужно запретить inline (материализовать):
> WITH active_users AS MATERIALIZED (
>     SELECT * FROM users WHERE status = 'active'
> )
> SELECT * FROM active_users WHERE created_at > '2024-01-01';
> ```
>
> **MATERIALIZED полезно когда:**
>
> - CTE используется несколько раз (избежать повторного вычисления)
> - Planner делает плохой план с inline (fence для optimizer)
>
> **Recursive CTE** — для иерархических данных (деревья, графы):
>
> ```sql
> WITH RECURSIVE tree AS (
>     SELECT id, parent_id, name FROM categories WHERE parent_id IS NULL
>     UNION ALL
>     SELECT c.id, c.parent_id, c.name FROM categories c JOIN tree t ON c.parent_id = t.id
> )
> SELECT * FROM tree;
> ```

> [!question]- 10. Как обеспечить zero-downtime миграцию схемы?
>
> **Опасные операции:**
>
> - `ALTER TABLE ADD COLUMN ... DEFAULT` — безопасно с PG 11+
> - `ALTER TABLE ADD COLUMN ... NOT NULL` — нужен default или backfill + constraint
> - `CREATE INDEX` — блокирует writes → используй `CONCURRENTLY`
> - `ALTER TABLE ... SET NOT NULL` — full table scan → добавь CHECK constraint first
>
> **Безопасный паттерн:**
>
> ```sql
> -- 1. Add nullable column
> ALTER TABLE users ADD COLUMN email_verified boolean;
>
> -- 2. Backfill in batches
> UPDATE users SET email_verified = false WHERE email_verified IS NULL AND id BETWEEN 1 AND 10000;
>
> -- 3. Add NOT NULL via constraint (not SET NOT NULL)
> ALTER TABLE users ADD CONSTRAINT users_email_verified_nn
>     CHECK (email_verified IS NOT NULL) NOT VALID;
> ALTER TABLE users VALIDATE CONSTRAINT users_email_verified_nn;
>
> -- 4. Create index concurrently
> CREATE INDEX CONCURRENTLY idx_email_verified ON users (email_verified);
> ```

> [!question]- 11. Как убедиться, что индекс реально используется?
>
> **Способ 1: EXPLAIN — смотреть на тип scan в плане:**
>
> ```sql
> EXPLAIN SELECT * FROM orders WHERE user_id = 123;
> -- Index Scan using idx_orders_user_id on orders  ✓ индекс используется
> -- Seq Scan on orders                              ✗ индекс игнорируется
> ```
>
> **Способ 2: pg_stat_user_indexes — статистика обращений:**
>
> ```sql
> SELECT
>     indexname,
>     idx_scan,        -- сколько раз индекс был использован
>     idx_tup_read,    -- строк прочитано через индекс
>     idx_tup_fetch    -- строк fetch с heap через индекс
> FROM pg_stat_user_indexes
> WHERE tablename = 'orders'
> ORDER BY idx_scan;
> ```
>
> `idx_scan = 0` после долгой работы сервиса → индекс не используется, кандидат на удаление.
>
> **Почему планировщик может игнорировать индекс:**
>
> - Таблица маленькая → Seq Scan дешевле
> - Low selectivity: запрос возвращает > ~5-10% строк → Seq Scan выгоднее
> - Stale statistics → `ANALYZE tablename`
> - Функция в WHERE без function-based index:
>   ```sql
>   WHERE lower(email) = 'test@example.com'
>   -- нужен: CREATE INDEX ON users (lower(email))
>   ```

> [!question]- 12. Почему EXPLAIN ANALYZE на dev даёт другой план чем на prod?
>
> **Главная причина: разный объём данных и разная статистика.**
>
> PostgreSQL query planner принимает решения на основе **статистики** (pg_statistic):
>
> - Число строк в таблице
> - Распределение значений (гистограмма, most common values)
> - Доля NULL
>
> **Типичные отличия dev vs prod:**
>
> |                | Dev                              | Prod                   |
> | -------------- | -------------------------------- | ---------------------- |
> | Данных         | 1 000 строк                      | 10 000 000 строк       |
> | Индекс         | Seq Scan (мало данных — быстрее) | Index Scan             |
> | Join           | Nested Loop                      | Hash Join / Merge Join |
> | Реальное время | 1ms                              | 30s (тот же план!)     |
>
> **Следствие:** запрос, который быстр на dev, может быть катастрофически медленным на prod с тем же планом исполнения.
>
> **Как тестировать реалистичные планы:**
>
> ```sql
> -- Обновить статистику после загрузки данных:
> ANALYZE orders;
>
> -- Посмотреть текущую статистику по колонке:
> SELECT * FROM pg_stats
> WHERE tablename = 'orders' AND attname = 'user_id';
>
> -- Принудить к index scan (для теста):
> SET enable_seqscan = off;
> EXPLAIN SELECT * FROM orders WHERE user_id = 123;
> RESET enable_seqscan;
> ```
>
> **Best practice:** держи staging DB с prod-like данными (анонимизированными) для проверки query plans перед деплоем.

> [!hint]- Улучшение статистики для неравномерного распределения
>
> ```sql
> -- Увеличить детализацию гистограммы (default = 100):
> ALTER TABLE orders ALTER COLUMN status SET STATISTICS 500;
> ANALYZE orders;
> -- Помогает когда в колонке много разных значений или неравномерное распределение
> ```
