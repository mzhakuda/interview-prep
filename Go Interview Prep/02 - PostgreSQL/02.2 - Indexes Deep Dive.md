# Indexes Deep Dive

Связанные темы: [[02.1 - MVCC & Transactions]] | [[02.4 - Performance & Connection Pooling]] | [[02.3 - Locking & Isolation Levels]]

---

> [!question]- 1. Перечисли типы индексов в PostgreSQL. Когда какой использовать?
>
> | Тип | Операторы | Использование |
> |-----|-----------|---------------|
> | **B-tree** | `=, <, >, <=, >=, BETWEEN, IN, LIKE 'foo%'` | Default. 90% случаев |
> | **Hash** | `=` | Только equality. Чуть быстрее B-tree для `=` |
> | **GIN** | `@>, ?, ?&, @@` | Arrays, JSONB, Full-text search |
> | **GiST** | `&&, @>, <@, <<, >>` | Геометрия, range types, full-text |
> | **BRIN** | `<, <=, =, >=, >` | Очень большие таблицы с natural ordering (created_at) |
> | **SP-GiST** | space-partitioned operators | Trie, quadtree, non-balanced structures |
>
> **Правило выбора:**
> - Не знаешь какой → **B-tree**
> - JSONB queries → **GIN**
> - Таблица > 100M rows, ordered by date → **BRIN** (крошечный размер)
> - Full-text search → **GIN** (быстрый поиск) или **GiST** (быстрый update)

> [!question]- 2. Как работает B-tree индекс внутри?
>
> **B-tree** — сбалансированное дерево:
> - Каждый node = страница (8KB по default)
> - Leaf nodes содержат ключи + pointers на heap tuples
> - Internal nodes содержат ключи + pointers на child nodes
> - Высота дерева = `log(N)` → O(log N) поиск
>
> **Для таблицы в 10M строк**: ~3-4 уровня → 3-4 I/O операции для поиска.
>
> **Поддерживает:**
> - Range scans (ключи отсортированы в leaf nodes)
> - Sorted output (индекс уже отсортирован → no sort needed)
> - Index-only scans (если все нужные колонки в индексе + visibility map clean)
>
> **Не поддерживает:** `LIKE '%foo'` (не может использовать prefix), `!=`, `IS NULL` (в старых версиях)

> [!question]- 3. Что такое Partial Index? Когда он полезен?
>
> **Partial index** — индекс только на подмножество строк:
>
> ```sql
> CREATE INDEX idx_active_orders ON orders (created_at)
>     WHERE status = 'active';
> ```
>
> **Когда полезен:**
> - Запросы всегда фильтруют по `WHERE status = 'active'`
> - Только 5% строк active → индекс в 20x меньше full index
> - Быстрее updates (меньше строк для поддержки)
>
> **Важно:** PostgreSQL использует partial index только если WHERE в запросе
> **покрывается** условием индекса. Planner должен доказать что subset совпадает.
>
> ```sql
> -- Использует partial index:
> SELECT * FROM orders WHERE status = 'active' AND created_at > now() - interval '1 day';
>
> -- НЕ использует (другой status):
> SELECT * FROM orders WHERE status = 'pending' AND created_at > now() - interval '1 day';
> ```

> [!question]- 4. Что такое Covering Index (INCLUDE)? Чем отличается от составного?
>
> **Covering index** содержит дополнительные колонки для index-only scan:
>
> ```sql
> -- Составной индекс: обе колонки в B-tree (сортированы, searchable)
> CREATE INDEX idx_comp ON orders (user_id, status);
>
> -- Covering индекс: email хранится, но не searchable
> CREATE INDEX idx_cover ON orders (user_id) INCLUDE (email, name);
> ```
>
> **Разница:**
> - `INCLUDE` колонки **не** в B-tree ключе — не участвуют в поиске/сортировке
> - `INCLUDE` колонки хранятся только в leaf nodes
> - Позволяет **index-only scan** без обращения к heap
>
> **Когда использовать:**
> ```sql
> -- Этот запрос станет index-only scan:
> SELECT email, name FROM orders WHERE user_id = 123;
> ```

> [!question]- 5. GIN vs GiST для full-text search — как выбрать?
>
> | | GIN | GiST |
> |---|-----|------|
> | Размер | Больше (хранит все лексемы) | Меньше (lossy) |
> | Поиск | Быстрее (exact match) | Медленнее (может recheck) |
> | Insert/Update | Медленнее (дороже поддерживать) | Быстрее |
> | Crash recovery | Медленнее (больше WAL) | Быстрее |
>
> **Правило:**
> - Статические данные, много поиска → **GIN**
> - Частые updates, меньше поиска → **GiST**
> - Не уверен → **GIN** (чаще правильный выбор)
>
> ```sql
> -- Full-text search с GIN
> CREATE INDEX idx_fts ON articles USING GIN (to_tsvector('english', body));
> SELECT * FROM articles WHERE to_tsvector('english', body) @@ to_tsquery('golang & concurrency');
> ```

> [!question]- 6. Что такое BRIN индекс? Когда он в 1000x лучше B-tree?
>
> **BRIN** (Block Range Index) — хранит min/max значения для каждого range of pages.
>
> **Размер:** крошечный. Для таблицы в 10GB: B-tree ~200MB, BRIN ~100KB.
>
> **Работает хорошо когда:**
> - Данные **физически отсортированы** на диске (append-only таблицы)
> - Колонка монотонно растёт: `created_at`, `id`
> - Таблица очень большая (миллиарды строк)
>
> ```sql
> CREATE INDEX idx_brin ON logs USING BRIN (created_at) WITH (pages_per_range = 32);
> ```
>
> **Не работает когда:**
> - Данные перемешаны (random updates) — min/max каждого range покрывает всё
> - Нужна точная сортировка (BRIN не хранит отдельные значения)

> [!question]- 7. Как определить что индекс не используется? Когда удалять?
>
> ```sql
> -- Неиспользуемые индексы:
> SELECT schemaname, relname, indexrelname, idx_scan,
>        pg_size_pretty(pg_relation_size(indexrelid)) as size
> FROM pg_stat_user_indexes
> WHERE idx_scan = 0
>   AND indexrelname NOT LIKE '%_pkey'
> ORDER BY pg_relation_size(indexrelid) DESC;
> ```
>
> **Когда удалять:**
> - `idx_scan = 0` за длительный период (недели)
> - Размер значительный (MB/GB)
> - Не является UNIQUE constraint (бизнес-логика)
>
> **Осторожно:**
> - Reset статистики при `pg_stat_reset()` — проверь `stats_reset`
> - Replica может использовать индекс (проверь на реплике тоже)
> - Сезонные запросы — индекс может быть нужен раз в месяц

> [!question]- 8. Объясни multicolumn index. Важен ли порядок колонок?
>
> **Да, порядок критичен!**
>
> ```sql
> CREATE INDEX idx_multi ON orders (user_id, status, created_at);
> ```
>
> Индекс используется для:
> - `WHERE user_id = 1` (leftmost prefix)
> - `WHERE user_id = 1 AND status = 'active'`
> - `WHERE user_id = 1 AND status = 'active' AND created_at > '2024-01-01'`
>
> Индекс **НЕ** используется для:
> - `WHERE status = 'active'` (не leftmost prefix)
> - `WHERE created_at > '2024-01-01'` (не leftmost prefix)
>
> **Правило порядка:**
> 1. Equality условия первыми (`user_id =`)
> 2. Range условия последними (`created_at >`)
> 3. Высокая selectivity первой (больше уникальных значений)

> [!hint]- Мнемоника: ESR Rule
> **E**quality → **S**ort → **R**ange
> Порядок колонок в составном индексе для оптимальной производительности.

> [!question]- 9. Что такое Index-Only Scan? Какие условия нужны?
>
> **Index-Only Scan** — PostgreSQL отвечает на запрос **только из индекса**, не обращаясь к heap.
>
> **Условия:**
> 1. Все запрашиваемые колонки есть в индексе (или INCLUDE)
> 2. **Visibility map** говорит что pages "all-visible" (после VACUUM)
> 3. Planner решает что это выгоднее
>
> ```sql
> -- С covering index: CREATE INDEX idx ON orders (user_id) INCLUDE (total);
> EXPLAIN SELECT total FROM orders WHERE user_id = 123;
> -- "Index Only Scan using idx on orders"
> ```
>
> **Если visibility map не обновлён**: PostgreSQL всё равно обращается к heap
> для проверки visibility → план показывает "Heap Fetches: 1234".
>
> **VACUUM** обновляет visibility map → уменьшает heap fetches.

> [!question]- 10. Как expression index помогает? Приведи пример.
>
> **Expression index** — индекс на вычисляемое выражение:
>
> ```sql
> -- Индекс на lower(email) для case-insensitive search
> CREATE INDEX idx_email_lower ON users (lower(email));
>
> -- Запрос ДОЛЖЕН использовать то же выражение:
> SELECT * FROM users WHERE lower(email) = 'john@example.com'; -- использует индекс
> SELECT * FROM users WHERE email = 'john@example.com';         -- НЕ использует
> ```
>
> **Другие примеры:**
> ```sql
> -- Индекс на год из timestamp
> CREATE INDEX idx_year ON orders (extract(year FROM created_at));
>
> -- Индекс на JSONB поле
> CREATE INDEX idx_type ON events ((data->>'type'));
> ```
>
> **Важно:** expression в запросе должен **точно совпадать** с expression в индексе.
