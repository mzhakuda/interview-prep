## Задача 1: Система записи в автошколу

> **Тип:** Booking / Scheduling System
> **Паттерны:** Race condition, Notifications, Sharding, Reliability

---

## Условие

Спроектировать систему записи учеников к инструкторам автошколы.

**Масштаб:**
- Start: 10K студентов, 1K инструкторов, 1 город
- Scale: 100M пользователей, глобально

**Функционал:**
- Просмотр своих записей
- Создание записи к инструктору
- Перезапись (reschedule)
- Отмена записи

---

| Вопрос | Ответ |

| Вопрос | Ответ |
|--------|-------|

| Вопрос | Ответ |
|--------|-------|
| Double-booking | **Запрещен** |
| Отмена / перенос | Не позже чем за 24 часа |
| Уведомления | Подтверждение сразу + напоминание за 1 час |

#### NFR
| | |
|-|-|
| Availability | 99.9% |
| Latency | Расписание < 100ms, бронирование < 300ms |

## Оценки

```
100M users × 10% DAU = 10M DAU
10M × 3 запроса/день = 30M запросов/день
30M / 86400 ≈ 350 RPS средний
Пик (начало семестра) ≈ 1,500 RPS

Read / Write ≈ 90% / 10%

Storage:
  1 booking ≈ 500 bytes
  100M студентов × 10 записей = 1B строк ≈ 500 GB
```

## Схема БД

```sql
schools       (id, name, address, city)
classrooms    (id, school_id, room_number, capacity)
instructors   (id, name, email, phone, license_categories[], lesson_types[])
students      (id, name, email, phone, enrolled_categories[])

lessons       (id, type, instructor_id, classroom_id,
               scheduled_at, duration_min, max_students, status)

bookings      (id, lesson_id, student_id,
               status,        -- confirmed | cancelled | rescheduled
               created_at, cancelled_at)

notifications (id, booking_id, user_id, type,
               scheduled_at, status, sent_at)
```

**Ключевые решения:**
- `max_students` в lessons: практика=1, теория=30 → одна проверка на double-booking
- `status` в bookings: не удаляем отмены — нужна история
- `notifications` таблица: для scheduled reminders (cron job)

## Deep Dive: Booking Flow

**Проблема:** два студента одновременно видят свободный слот → race condition.

**Решение — SELECT FOR UPDATE:**

```sql
BEGIN;
SELECT id FROM lessons WHERE id = ? FOR UPDATE;
SELECT COUNT(*) FROM bookings
  WHERE lesson_id = ? AND status = 'confirmed';
-- если count < max_students:
INSERT INTO bookings (lesson_id, student_id, status)
VALUES (?, ?, 'confirmed');
COMMIT;
```

Обычные `SELECT` (чтение расписания) не блокируются — PostgreSQL MVCC.
FOR UPDATE плох только при массовом write contention на одной строке.

## Deep Dive: Notifications

```
Подтверждение (сразу):
  Start: goroutine (fire-and-forget)
  Scale: API → Kafka "booking-events" → Push/Email Service

Напоминание (за 1 час):
  Start: cron каждую минуту → SELECT notifications WHERE scheduled_at <= NOW()
  Scale: Redis Sorted Set → ZADD score=unix_ts, worker берет score <= now

Scheduler — отдельный сервис (не внутри API Server):
  K8s CronJob или 2 инстанции + Redis Lock (NX EX 70sec)
```

## Reliability

| Компонент | Решение |
|-----------|---------|
| Postgres Primary упал | Failover → реплика → Primary (Patroni) |
| Один шард недоступен | Partial outage, replica на каждый шард |
| Redis упал | Graceful degradation → запросы в Postgres |
| Kafka брокер упал | Replication Factor=3, другие брокеры берут трафик |
| API Server упал | Health checks → LB убирает из rotation |
| Scheduler упал | K8s перезапускает / Redis Lock → backup берет управление |

## Wrap-up

> _"Спроектировал систему записи в автошколу на PostgreSQL + Redis + Kafka.
> Старт — простой Postgres + API для 10K users.
> При росте до 100M: шардирование по hash(user\_id), Read Replicas ×3,
> Redis кэш расписания, Kafka для уведомлений, Scheduler как отдельный сервис.
>
> Главный trade-off: шардирование по user\_id удобно для студента,
> но запросы инструктора (его слоты) идут через несколько шардов.
>
> С большим временем: разобрал бы availability слотов инструктора
> и региональную репликацию данных."_
