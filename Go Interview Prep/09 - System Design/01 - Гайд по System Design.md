## 1. Сбор требований

### Функциональные (FR) vs Нефункциональные (NFR)

| | FR | NFR |
|-|----|-----|
| Вопрос | **Что** делает система? | **Как хорошо** это делает? |
| Примеры | Создать запись, отменить, посмотреть | Latency, Availability, Scale |

---

### Техника: Noun → Variations

На каждое **существительное** в задаче задаем вопрос:
> _"Какие типы/варианты этого бывают?"_

Приоритизируй по важности:

| Важность | Когда | Пример |
|----------|-------|--------|
| 🔴 Критично | Меняет схему БД или кардинальность связей | Урок: 1-to-1 vs many-to-1 |
| 🔴 Критично | Меняет какие данные вообще хранить | Офлайн/онлайн → адреса vs ссылки |
| 🟡 Полезно | Влияет на бизнес-логику | Типы студентов, типы записей |
| 🟢 Опционально | Важно при определенных условиях | Типы инструкторов |

### Три группы вопросов

**Акторы** — кто пользуется?
- Кто основные пользователи?
- Разные роли и права?
- Один аккаунт или разные? ← это требование, не архитектура

**Действия** — что делают?
- Основные операции каждого актора
- Ограничения? (отменить только за 24 часа)
- Нужны уведомления?

**Сущности** — Noun → Variations
- Какие типы/варианты бывают?
- Есть лимиты? (вместимость, длительность)
- Кто владелец?

### Как задавать вопросы правильно

Объясняй **зачем** — не спамь шаблоном:

```
❌ Плохо:
"Какие типы уроков?"
"Какие типы записей?"
"Какие типы инструкторов?"

✅ Хорошо:
"Урок — это всегда 1-to-1 или бывают групповые?
 Это влияет на схему БД."

→ [ответ]

"Уроки офлайн? Тогда нужно хранить адреса школ
 и считать вместимость аудиторий."
```

### NFR — три источника

| Источник | Что сюда входит | Как получить |
|----------|----------------|--------------|
| **Спрашиваешь** | Scale: DAU, RPS, регионы | Обязательно спросить |
| **Предлагаешь** | Latency, Availability | Говоришь сам, просишь подтвердить |
| **Выводишь** | Consistency, Durability | Очевидно из домена — говоришь как факт |

```
❌ "Какие у вас нефункциональные требования?"

✅ "По масштабу — сколько ожидаем пользователей?"
   [100M]
   "Тогда NFR: Availability 99.9%, latency расписания < 100ms,
   strong consistency для бронирования — double-booking недопустим.
   Что-то скорректировать?"
```

### Правило 5 минут

> Первые 5 минут — только вопросы. Не рисуй ничего.

Если торопят: _"Хочу потратить пару минут на требования, чтобы не проектировать не то."_

## 2. Оценки (Back-of-the-envelope)

### Алгоритм — 4 шага

```
1. DAU = Total Users × % активности  (обоснуй %)
2. Запросы/день = DAU × действий за сессию  (думай как юзер)
3. Avg RPS = запросы/день / 86400
4. Peak RPS = Avg RPS × пик  (назови конкретный бизнес-сценарий)
```

### Пиковый RPS — через бизнес-событие

```
❌ "Умножим на 3"
✅ "Пик — начало семестра, все записываются утром в понедельник → ×3"
✅ "Пик — дроп альбома Taylor Swift, DAU вырастет на 50%"
```

### Read/Write split

Всегда определяй — это влияет на архитектуру:
```
Read-heavy  → кэш (Redis) + Read Replicas + CDN
Write-heavy → шардирование + async processing
```

## 3. High-Level Design

### Главный принцип: Start Simple → Then Scale

```
❌ Антипаттерн: сразу Kafka + шардирование + 3 региона
   → не можешь объяснить зачем каждый компонент

✅ Правильно: простая схема → сам находишь узкие места → добавляешь точечно
   → каждый компонент = конкретная проблема
```

### Три слоя (появляются по ходу разговора)

| Слой | Что | Когда рисуешь |
|------|-----|---------------|
| HLD | Client, LB, API, DB, Cache | Сразу |
| Deep Dive | Booking flow, transaction logic | По запросу |
| Scaling | PgBouncer, Replicas, Shards, Partitioning | При обсуждении роста |

### Типичные компоненты и когда добавлять

| Компонент | Проблема которую решает |
|-----------|------------------------|
| Load Balancer | Одна точка отказа у API |
| Redis | Одни и те же данные читают тысячи раз |
| Read Replica | DB не тянет read RPS |
| PgBouncer | Postgres не тянет 10K+ connections (1 conn = 5-10MB RAM) |
| Шардирование | Один DB не тянет write |
| Партиционирование | Таблица слишком большая → медленные запросы |
| Kafka | Decoupling сервисов + надежная доставка |
| CDN | Статика / стриминг для глобальных юзеров |
| Региональный деплой | Latency 300ms+ для глобальных юзеров |

## 4. Выбор технологий

### Фреймворк выбора БД — 3 вопроса

**Q1: Какая форма данных?**
| Данные | БД |
|--------|----|
| Связанные сущности | PostgreSQL |
| Гибкая схема | MongoDB |
| Ключ → значение | Redis |
| Время-серия (метрики, GPS) | InfluxDB, TimescaleDB |

**Q2: Как читается и пишется?**
| Паттерн | БД |
|---------|----|
| Нужны транзакции | PostgreSQL (ACID) |
| Много записей, eventual consistency ok | Cassandra, DynamoDB |
| Горячий кэш | Redis |
| Полнотекстовый поиск | Elasticsearch |

**Q3: Consistency или Availability важнее?**
```
Деньги / бронирования / медицина → consistency → SQL + ACID
Лайки / просмотры / аналитика   → availability → NoSQL + eventual
```

### Структура ответа на интервью

```
Выбор → Почему (данные + паттерн) → Trade-off → Путь к масштабу
```

| Технология | vs | Когда что |
|-----------|-----|-----------|
| PostgreSQL (ACID) | Cassandra (scale) | Зависит от consistency требований |
| Redis | Memcached | Redis если нужны структуры данных |
| Kafka (replay, throughput) | RabbitMQ (routing, priority) | Kafka для event streaming |
| REST | gRPC | gRPC для внутренних сервисов |

## 5. Race Conditions (Concurrency)

Возникают когда два запроса одновременно читают и пишут одни данные.

### SELECT FOR UPDATE — базовое решение

```sql
BEGIN;
SELECT id FROM lessons WHERE id = ? FOR UPDATE; -- lock на строку
SELECT COUNT(*) FROM bookings WHERE lesson_id = ? AND status = 'confirmed';
-- если count < max_students:
INSERT INTO bookings (...) VALUES (...);
COMMIT;
```

> **MVCC в PostgreSQL:** обычный `SELECT` читает snapshot и **никогда не блокируется**
> row lock'ом от FOR UPDATE. Тысячи читателей работают параллельно с писателем.
> FOR UPDATE плох не при heavy-read, а при **heavy write contention** на одной строке.

### Optimistic Locking — при высоком write contention

```sql
-- Читаем без блокировки:
SELECT id, booked_count, version FROM lessons WHERE id = ?

-- Пишем с проверкой версии:
UPDATE lessons
SET booked_count = booked_count + 1, version = version + 1
WHERE id = ? AND version = ? AND booked_count < max_students
-- affected rows = 0 → retry
```

### Distributed Lock (Redis) — для flow через несколько сервисов

```
SET lock:resource:{id} "taken" NX EX 5
NX = только если не существует
EX = TTL защита от deadlock
```

### Когда что

| Ситуация | Решение |
|----------|---------|
| Low contention (обычное бронирование) | SELECT FOR UPDATE |
| High contention (билеты на концерт) | Optimistic Locking |
| Flow через несколько сервисов | Distributed Lock (Redis) |

## 6. Notifications

### Два типа — разные решения

```
Immediate (сейчас)   → горутина → Kafka при росте
Scheduled (в X часов) → DB polling → Redis Sorted Set при росте
```

### Immediate — эволюция

```
Маленький масштаб:  API → goroutine (fire-and-forget, ненадежно)
При росте:          API → kafka.Publish → Notification Service
```

Kafka гарантирует at-least-once: если сервис упал — сообщение ждет.
Бонус: один event → несколько consumers (Push + Email + SMS).

### Scheduled — три подхода

| Подход | Как | Когда |
|--------|-----|-------|
| DB polling | cron каждую минуту: `WHERE scheduled_at <= NOW()` | Малый масштаб |
| Redis Sorted Set | `ZADD score=timestamp`, worker берет `score <= now` | Высокий масштаб |
| Delayed messages | RabbitMQ plugin с delay | Если уже есть RMQ |

> **Kafka не заменяет планировщик** — не умеет "доставь через X часов".
> Scheduler находит что пора → публикует в Kafka → Notification Service отправляет.

### Scheduler — отдельный сервис

```
❌ Внутри API Server: 5 инстансов → 5 cron job → 5 дублей уведомлений

✅ Отдельный Scheduler Service (×1):
   Вариант A: Kubernetes CronJob (K8s перезапустит при падении)
   Вариант B: 2 инстанции + Redis Lock (NX EX 70sec)
```

## 7. Reliability

### Паттерны

| Паттерн | Что делает |
|---------|-----------|
| **Failover** | При падении Primary → реплика автоматически становится Primary (Patroni) |
| **Graceful Degradation** | Система работает медленнее, но не падает (Redis упал → запросы в DB) |
| **Partial Degradation** | Один шард недоступен → затронуты только его пользователи |
| **Circuit Breaker** | Прекращает запросы к упавшему сервису, защищает от cascade failure |
| **Health Checks** | Load Balancer убирает из rotation нездоровые инстансы |
| **DLQ** | Dead Letter Queue — сообщения после N неудачных попыток → отдельный топик |

### Что происходит при падении компонентов

| Компонент | Что происходит | Решение |
|-----------|---------------|---------|
| Postgres Primary | writes недоступны ~30 сек | Failover → реплика → Primary (Patroni) |
| Один шард | Partial outage только для этого шарда | Replica на каждый шард |
| Redis | Cache miss → запросы в Postgres | Graceful Degradation |
| Kafka брокер | Другие брокеры берут трафик | Replication Factor = 3 |
| Kafka consumer | Сообщения ждут в Kafka | At-least-once + DLQ |
| API Server | LB убирает из rotation | Health checks + stateless |
| Scheduler | Backup берет управление | K8s CronJob или Redis Lock |

### Cascade Failure — как решать

```
Проблема: сервис A → сервис B → сервис C
          C упал → B висит → A висит → всё упало

Решение 1: Event-driven (async)
  Сервисы не зависят друг от друга напрямую → через очередь

Решение 2: Timeouts
  Каждый вызов имеет timeout → не висит вечно

Решение 3: Circuit Breaker
  После N ошибок → перестаем слать запросы → даем системе восстановиться
```

## 8. Deep Dive

**Что выбрать:**
| Кандидат | Выбирай если |
|----------|-------------|
| Транзакционный flow | Риск race condition |
| Scheduling / slots | Сложная логика времени |
| Notifications | Нужна надежная доставка |
| Поиск | High read load, сложные запросы |
