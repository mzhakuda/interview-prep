# Interfaces & Type System

Связанные темы: [[07.1 - GC & Memory Model]] | [[05.1 - Clean Architecture]]

---

> [!question]- 1. Как интерфейсы реализованы внутри? iface vs eface.
>
> **`eface`** — empty interface (`interface{}` / `any`):
> ```go
> type eface struct {
>     _type *_type  // pointer to type info
>     data  unsafe.Pointer  // pointer to data
> }
> ```
>
> **`iface`** — non-empty interface:
> ```go
> type iface struct {
>     tab  *itab           // pointer to interface table (type + method pointers)
>     data unsafe.Pointer  // pointer to data
> }
> ```
>
> **`itab`** содержит:
> - Тип интерфейса
> - Конкретный тип
> - Массив указателей на методы (для dispatch)
>
> **Размер:** 16 байт (2 pointers на 64-bit).
> **Dispatch:** indirect call через function pointer в itab (~2ns overhead vs direct call).

> [!question]- 2. Когда `interface{}` содержащий nil != nil?
>
> ```go
> var p *int = nil
> var i interface{} = p
>
> fmt.Println(i == nil)  // false!
> fmt.Println(p == nil)  // true
> ```
>
> **Почему?** Interface — это (type, value) пара.
> - `i` содержит: `(*int, nil)` — type не nil!
> - Interface == nil только когда **оба** type и value = nil
>
> **Частая ошибка с error:**
> ```go
> func getError() error {
>     var err *MyError = nil
>     return err // interface{(*MyError, nil)} != nil!
> }
>
> if err := getError(); err != nil {
>     // Сюда ЗАЙДЁТ! err != nil потому что type != nil
> }
>
> // Правильно:
> func getError() error {
>     return nil // interface{(nil, nil)} == nil
> }
> ```

> [!warning]- Это один из самых частых подвохов на Go интервью
> Всегда возвращай `nil` напрямую, не через типизированный nil pointer.

> [!question]- 3. Value receiver vs Pointer receiver — полные правила.
>
> | | Value receiver `(v T)` | Pointer receiver `(p *T)` |
> |---|----------------------|--------------------------|
> | Может вызвать на value? | Да | Да (auto `&v`) |
> | Может вызвать на pointer? | Да (auto `*p`) | Да |
> | Модифицирует оригинал? | Нет (копия) | Да |
> | Реализует interface? | T и *T | Только *T |
>
> **Ключевое правило для interfaces:**
> ```go
> type Stringer interface { String() string }
>
> type MyType struct{ name string }
> func (m *MyType) String() string { return m.name } // pointer receiver
>
> var s Stringer = MyType{}  // COMPILE ERROR: MyType не реализует Stringer
> var s Stringer = &MyType{} // OK: *MyType реализует Stringer
> ```
>
> **Когда pointer receiver:**
> - Метод модифицирует struct
> - Struct большой (avoid copy overhead)
> - Consistency: если хотя бы один метод = pointer, все должны быть pointer
> - Struct содержит sync.Mutex (нельзя копировать)

> [!question]- 4. Type assertion vs Type switch — когда что?
>
> **Type assertion** — конкретный тип:
> ```go
> val, ok := i.(string)
> if !ok {
>     // i не содержит string
> }
> ```
>
> **Type switch** — множество типов:
> ```go
> switch v := i.(type) {
> case string:
>     fmt.Println("string:", v)
> case int:
>     fmt.Println("int:", v)
> case nil:
>     fmt.Println("nil")
> default:
>     fmt.Printf("unknown: %T\n", v)
> }
> ```
>
> **Performance:** type switch быстрее серии if-else type assertions.
> Go compiler оптимизирует type switch в jump table.

> [!question]- 5. Embedding в Go — как работает и зачем?
>
> **Struct embedding** — composition вместо inheritance:
> ```go
> type Logger struct{}
> func (l Logger) Log(msg string) { fmt.Println(msg) }
>
> type Server struct {
>     Logger // embedded — Server "наследует" Log()
>     addr string
> }
>
> s := Server{addr: ":8080"}
> s.Log("hello") // promoted method
> ```
>
> **Interface embedding:**
> ```go
> type ReadWriter interface {
>     io.Reader
>     io.Writer
> }
> ```
>
> **Ключевое:** это не наследование!
> - Нет polymorphism (Server не является Logger)
> - Method promotion (Server.Log = Server.Logger.Log)
> - Можно override (определить свой Log на Server)

> [!question]- 6. Generics в Go — синтаксис, constraints, когда использовать?
>
> ```go
> // Обобщённая функция
> func Map[T, U any](s []T, fn func(T) U) []U {
>     result := make([]U, len(s))
>     for i, v := range s {
>         result[i] = fn(v)
>     }
>     return result
> }
>
> // Constraint
> type Number interface {
>     ~int | ~float64 | ~int64
> }
>
> func Sum[T Number](nums []T) T {
>     var sum T
>     for _, n := range nums {
>         sum += n
>     }
>     return sum
> }
>
> // Обобщённый тип
> type Stack[T any] struct {
>     items []T
> }
>
> func (s *Stack[T]) Push(item T) { s.items = append(s.items, item) }
> func (s *Stack[T]) Pop() (T, bool) {
>     if len(s.items) == 0 {
>         var zero T
>         return zero, false
>     }
>     last := s.items[len(s.items)-1]
>     s.items = s.items[:len(s.items)-1]
>     return last, true
> }
> ```
>
> **Когда использовать generics:**
> - Collections (Stack, Queue, Set)
> - Utility functions (Map, Filter, Reduce)
> - Type-safe containers
>
> **Когда НЕ использовать:**
> - Когда `interface{}` / `any` достаточно
> - Для polymorphism (используй interfaces)

> [!question]- 7. Что такое `comparable` constraint?
>
> `comparable` — built-in constraint для типов поддерживающих `==` и `!=`:
>
> ```go
> func Contains[T comparable](s []T, target T) bool {
>     for _, v := range s {
>         if v == target { return true }
>     }
>     return false
> }
>
> // Можно использовать как map key:
> func UniqueKeys[K comparable, V any](m map[K]V) []K {
>     keys := make([]K, 0, len(m))
>     for k := range m {
>         keys = append(keys, k)
>     }
>     return keys
> }
> ```
>
> **`comparable` включает:** все basic types, structs из comparable полей, arrays, pointers, interfaces, channels.
> **Не включает:** slices, maps, functions.

> [!question]- 8. Error handling patterns в Go — wrapping, sentinel, custom types.
>
> **Sentinel errors:**
> ```go
> var ErrNotFound = errors.New("not found")
> if errors.Is(err, ErrNotFound) { ... }
> ```
>
> **Custom error types:**
> ```go
> type ValidationError struct {
>     Field   string
>     Message string
> }
> func (e *ValidationError) Error() string { return e.Field + ": " + e.Message }
>
> var valErr *ValidationError
> if errors.As(err, &valErr) { ... }
> ```
>
> **Wrapping (Go 1.13+):**
> ```go
> return fmt.Errorf("fetch user %d: %w", id, err) // %w wraps
>
> // Unwrapping chain:
> errors.Is(err, ErrNotFound)    // checks anywhere in chain
> errors.As(err, &target)        // checks anywhere in chain
> errors.Unwrap(err)             // one level up
> ```
>
> **Best practice:** wrap с контекстом на каждом уровне, check с `Is`/`As` на верхнем.

> [!question]- 9. Slices internals — header, capacity, gotchas.
>
> **Slice header:** `struct { pointer, length, capacity }` (24 байта)
>
> ```go
> s := make([]int, 3, 5)
> // pointer → [0, 0, 0, _, _]
> // length = 3, capacity = 5
> ```
>
> **Append и grow:**
> - `len < cap`: append в существующий array
> - `len == cap`: новый array (2x или 1.25x для больших), копирование
>
> **Gotcha: shared underlying array:**
> ```go
> a := []int{1, 2, 3, 4, 5}
> b := a[1:3]  // b = [2, 3], shares underlying array
> b[0] = 99    // a = [1, 99, 3, 4, 5]!
>
> // Безопасная копия:
> b := make([]int, 2)
> copy(b, a[1:3])
>
> // Или (Go 1.21+):
> b := slices.Clone(a[1:3])
> ```

> [!question]- 10. `init()` функции — порядок выполнения, когда использовать?
>
> **Порядок:**
> 1. Imported packages init() (рекурсивно, в порядке импорта)
> 2. Package-level variable declarations
> 3. `init()` функции текущего пакета
> 4. Может быть несколько init() в одном файле и пакете
>
> **Когда использовать:**
> - Регистрация database drivers: `import _ "github.com/lib/pq"`
> - Инициализация global state (если нельзя через var)
> - Проверка environment при старте
>
> **Когда НЕ использовать:**
> - Complex initialization (используй explicit New() функции)
> - Side effects (загрузка файлов, network calls) — делает testing сложным
> - Всё что можно сделать в `main()` — делай в main

> [!question]- 11. Что такое `panic` и `recover`? Как правильно использовать?
>
> **`panic`** — остановить нормальное выполнение горутины:
> - Начинает раскрутку стека (stack unwinding)
> - Выполняет все pending `defer` по LIFO
> - Если не перехвачен → crash с goroutine dump
>
> **`recover`** — перехватить panic (только внутри `defer`):
> ```go
> func safeDiv(a, b int) (result int, err error) {
>     defer func() {
>         if r := recover(); r != nil {
>             err = fmt.Errorf("recovered from panic: %v", r)
>         }
>     }()
>     return a / b, nil // panic если b == 0
> }
> ```
>
> **Правила:**
> - `recover()` работает **только** непосредственно в `defer` функции
> - `recover()` вне `defer` всегда возвращает nil
> - После `recover` выполнение продолжается **после** вызова panic-функции, не внутри неё
>
> **Когда использовать:**
> - HTTP server middleware — перехватить панику обработчика
> - Library/plugin boundary — не давать панике пробиться наружу
>
> **Когда НЕ использовать:**
> - Как замену error handling — используй явные errors
> - В business logic

> [!warning]- Антипаттерн: молчаливое подавление паник
> ```go
> defer func() {
>     recover() // ПЛОХО: подавляет все паники без логирования
> }()
> ```
> Всегда логируй причину и возвращай ошибку. Если не понимаешь панику — re-panic.

> [!question]- 12. Array vs Slice — ключевые отличия.
>
> | | Array | Slice |
> |---|-------|-------|
> | Размер | Фиксирован (часть типа) | Динамический |
> | Тип | `[5]int` ≠ `[6]int` | `[]int` |
> | Передача | По значению (полная копия) | По значению (только header) |
> | Comparable | Да (`==`) | Нет (только с nil) |
> | Stack/Heap | Stack если мал | Underlying array на heap |
>
> ```go
> // Array — копируется полностью при присвоении
> a := [3]int{1, 2, 3}
> b := a           // полная копия
> b[0] = 99        // a[0] == 1, b[0] == 99
>
> // Slice — header (ptr+len+cap), общий underlying array
> s := []int{1, 2, 3}
> t := s           // копия header, тот же массив!
> t[0] = 99        // s[0] == 99 тоже!
>
> // Размер массива — часть типа
> func f(a [3]int) {} // принимает только [3]int, не [4]int
> func g(s []int)  {} // принимает любой slice
> ```
>
> **Когда использовать array:**
> - Фиксированный размер известен compile-time: `[32]byte` для SHA256
> - Нужна comparable для map key: `type Point [2]float64`
> - Маленький буфер, важна стек-аллокация

> [!question]- 13. Как переопределить один метод интерфейса? Embedding pattern.
>
> В Go нет наследования, но embedding позволяет переопределить только нужные методы:
>
> ```go
> // Декоратор: логируем Write, делегируем Read
> type LoggingWriter struct {
>     io.ReadWriter // embed interface — все методы делегируются
> }
>
> func (lw LoggingWriter) Write(p []byte) (n int, err error) {
>     log.Printf("writing %d bytes", len(p))
>     return lw.ReadWriter.Write(p) // делегируем оригиналу
> }
> // Read не переопределён — вызывается через embedded interface
> ```
>
> **Практический пример — http.ResponseWriter wrapper:**
> ```go
> type statusRecorder struct {
>     http.ResponseWriter       // embed → все методы делегируются
>     status int
> }
>
> func (r *statusRecorder) WriteHeader(status int) {
>     r.status = status                    // перехватываем для логирования
>     r.ResponseWriter.WriteHeader(status) // делегируем оригиналу
> }
>
> // statusRecorder реализует http.ResponseWriter
> // но WriteHeader перехвачен — можно читать r.status после запроса
> ```
>
> **Правило:** embed interface + override нужных методов = decorator pattern без кодогенерации.

> [!question]- 14. Сколько времени занимает доступ к 7-му элементу массива из 10?
>
> **O(1) — константное время**, независимо от размера массива.
>
> Массив — непрерывный блок памяти. Адрес элемента:
> ```
> address = base_address + index × element_size
> ```
>
> Для `arr[6]` (7-й элемент, 0-indexed):
> ```
> address = base + 6 × sizeof(int)
> ```
> Одна арифметическая операция + одно обращение к памяти. Не важно 7-й или 700-й — время одинаковое.
>
> **Сравнение структур данных:**
> | Структура | Доступ по индексу |
> |-----------|------------------|
> | Array / Slice | O(1) |
> | Linked List | O(n) |
> | HashMap | O(1) amortized |
> | Binary search (sorted array) | O(log n) |
>
> Это одна из причин, почему array/slice предпочтительны для random access — предсказуемая O(1) сложность плюс cache-friendly (данные в памяти непрерывно).
