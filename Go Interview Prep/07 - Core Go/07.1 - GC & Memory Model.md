# GC & Memory Model

Связанные темы: [[07.2 - Interfaces & Type System]] | [[01.1 - GMP Model & Scheduler]]

---

> [!question]- 1. Как работает Garbage Collector в Go? Опиши алгоритм.
>
> Go использует **tri-color mark-and-sweep** concurrent GC:
>
> **Три цвета:**
> - **White** — не посещён (кандидат на удаление)
> - **Gray** — посещён, но потомки не проверены
> - **Black** — посещён, все потомки проверены (live)
>
> **Фазы:**
> 1. **Mark Setup** (STW ~10-30μs): включить write barrier
> 2. **Marking** (concurrent): обход графа объектов, раскрашивание
> 3. **Mark Termination** (STW ~10-30μs): финализация, выключить write barrier
> 4. **Sweep** (concurrent): освобождение white объектов
>
> **Write Barrier:** отслеживает новые ссылки во время concurrent marking.
> Если goroutine создаёт новую ссылку black → white, barrier окрашивает white → gray.
>
> **Pacer:** GC запускается когда heap вырос на `GOGC`% от live heap (default 100% = удвоение).

> [!question]- 2. Что такое GOGC и GOMEMLIMIT? Как настроить?
>
> **`GOGC`** (default: 100) — процент роста heap для trigger GC:
> - `GOGC=100`: GC при 2x live heap (100% рост)
> - `GOGC=50`: GC чаще (при 1.5x) → меньше memory, больше CPU
> - `GOGC=200`: GC реже (при 3x) → больше memory, меньше CPU
> - `GOGC=off`: GC выключен (для benchmarks)
>
> **`GOMEMLIMIT`** (Go 1.19+) — soft memory limit:
> - GC становится агрессивнее при приближении к лимиту
> - Не hard limit — может превысить при необходимости
> - Полезно в контейнерах: `GOMEMLIMIT=1GiB`
>
> **Рекомендация для контейнеров:**
> ```
> GOMEMLIMIT = container_memory_limit × 0.9  # 90%, оставить буфер для OS/stack
> GOGC = 100  # или даже off с GOMEMLIMIT
> ```

> [!question]- 3. Что такое Escape Analysis? Как влияет на производительность?
>
> **Escape analysis** — compile-time анализ: аллоцировать на stack или heap?
>
> **Stack** — быстро (автоматически, при выходе из функции)
> **Heap** — медленно (GC должен собрать)
>
> **Когда escapes на heap:**
> - Возврат pointer из функции: `return &obj`
> - Передача в interface: `fmt.Println(x)` (x escapes)
> - Closure captures: переменная используется в goroutine
> - Слишком большой объект для stack
> - Размер неизвестен compile-time: `make([]byte, n)` где n — variable
>
> **Проверка:**
> ```bash
> go build -gcflags="-m" ./...
> # ./main.go:10:6: moved to heap: result
> ```
>
> **Оптимизация:** передавай by value (не pointer) когда объект маленький и не нужен shared access.

> [!question]- 4. Stack vs Heap allocation в Go. Что быстрее и почему?
>
> **Stack:**
> - Allocation: просто сдвинуть stack pointer (~1ns)
> - Deallocation: автоматически при return (stack frame pop)
> - Locality: данные рядом в memory → cache-friendly
>
> **Heap:**
> - Allocation: найти свободный block в heap (~30-100ns)
> - Deallocation: GC (periodic, adds latency)
> - Locality: данные разбросаны → cache misses
>
> **Stack быстрее в ~30-100x для allocation.**
> Поэтому escape analysis важен: меньше heap allocations = быстрее + меньше GC pressure.

> [!question]- 5. Как профилировать Go приложение? pprof, trace.
>
> **CPU profiling:**
> ```go
> import _ "net/http/pprof"
> go http.ListenAndServe(":6060", nil)
> // go tool pprof http://localhost:6060/debug/pprof/profile?seconds=30
> ```
>
> **Memory profiling:**
> ```bash
> go tool pprof http://localhost:6060/debug/pprof/heap
> # (pprof) top10
> # (pprof) list functionName
> # (pprof) web  # визуализация в браузере
> ```
>
> **Goroutine profiling:**
> ```bash
> go tool pprof http://localhost:6060/debug/pprof/goroutine
> # Найти goroutine leaks
> ```
>
> **Trace (полная трассировка):**
> ```bash
> curl -o trace.out http://localhost:6060/debug/pprof/trace?seconds=5
> go tool trace trace.out
> # Визуализация: goroutine scheduling, GC pauses, blocking
> ```
>
> **Benchmark с memory:**
> ```bash
> go test -bench=. -benchmem -memprofile=mem.prof
> go tool pprof mem.prof
> ```

> [!question]- 6. Что такое memory alignment? Как влияет на размер struct?
>
> CPU читает память по **aligned addresses** (кратно размеру типа).
>
> ```go
> // Плохо: 24 байта (с padding)
> type Bad struct {
>     a bool   // 1 byte + 7 padding
>     b int64  // 8 bytes
>     c bool   // 1 byte + 7 padding
> }
>
> // Хорошо: 16 байт (без лишнего padding)
> type Good struct {
>     b int64  // 8 bytes
>     a bool   // 1 byte
>     c bool   // 1 byte + 6 padding
> }
> ```
>
> **Правило:** располагай поля от большего к меньшему.
> **Инструмент:** `go vet -fieldalignment` или `golang.org/x/tools/go/analysis/passes/fieldalignment`.

> [!question]- 7. Как работает `defer`? Порядок, аргументы, return value.
>
> **Порядок:** LIFO (Last In, First Out)
> ```go
> defer fmt.Println("first")
> defer fmt.Println("second")
> defer fmt.Println("third")
> // Output: third, second, first
> ```
>
> **Аргументы вычисляются сразу:**
> ```go
> x := 1
> defer fmt.Println(x) // напечатает 1
> x = 2
> ```
>
> **Можно модифицировать named return values:**
> ```go
> func f() (result int) {
>     defer func() { result++ }()
>     return 10
> }
> // Возвращает 11 (defer модифицирует result после return)
> ```
>
> **Performance:** с Go 1.14 open-coded defer — почти нулевой overhead.

> [!question]- 8. `new()` vs `make()` — разница?
>
> | | `new(T)` | `make(T, args...)` |
> |---|---------|-------------------|
> | Возвращает | `*T` (pointer) | `T` (value) |
> | Инициализация | Zero value | Initialized |
> | Типы | Любой тип | Только slice, map, channel |
>
> ```go
> p := new(int)           // *int, *p == 0
> s := make([]int, 5, 10) // []int, len=5, cap=10
> m := make(map[string]int) // map, initialized (не nil)
> ch := make(chan int, 3)    // buffered channel
> ```
>
> **На практике:** `new()` используется редко. Предпочитают `&MyStruct{}` (composite literal).

> [!question]- 9. Что такое unsafe.Pointer? Когда допустимо использовать?
>
> `unsafe.Pointer` обходит type safety Go. Позволяет:
> - Конвертировать между incompatible pointer types
> - Pointer arithmetic
> - Доступ к unexported полям через reflection
>
> **Допустимо:**
> - Performance-critical code (avoiding copies)
> - Interfacing with C (cgo)
> - Implementing low-level libraries (sync, reflect)
>
> **Недопустимо:**
> - Application code (почти всегда)
> - Когда можно решить через interfaces/generics
>
> **Правила:** `go vet` проверяет корректность unsafe patterns.

> [!question]- 10. Strings в Go — как устроены? Immutable?
>
> **String = struct { pointer, length }** (16 байт на 64-bit):
> - `pointer` → underlying byte array (UTF-8 encoded)
> - `length` — количество байт (не рун!)
>
> **Immutable:** строку нельзя изменить после создания.
> `s[0] = 'a'` — compile error.
>
> **Substring:** `s[1:5]` **не копирует** данные — разделяет underlying array.
> Это значит: большая строка удерживается в памяти если есть ссылка на substring.
>
> **Rune vs Byte:**
> ```go
> s := "Привет"
> len(s)         // 12 (байт, UTF-8)
> len([]rune(s)) // 6 (рун, символов)
> ```
