# Allocations Deep Dive

Связанные темы: [[07.1 - GC & Memory Model]] | [[07.2 - Interfaces & Type System]] | [[01.1 - GMP Model & Scheduler]]

---

> [!question]- 1. Как устроен аллокатор Go? Опиши трёхуровневую архитектуру.
>
> Go runtime использует **трёхуровневый аллокатор** (вдохновлён TCMalloc от Google):
>
> ```
> Горутина → P → mcache (per-P, без лока)
>                   ↓ (если пусто)
>               mcentral (per-size-class, с локом)
>                   ↓ (если пусто)
>               mheap (глобальный, с локом)
>                   ↓ (если пусто)
>               OS (mmap / sbrk)
> ```
>
> **mcache** (per-P, lock-free):
> - У каждого P свой mcache — **никаких локов** при аллокации
> - Содержит набор **mspan** по одному на каждый size class
> - Горутина аллоцирует из mcache своего текущего P → **O(1), ~10-25ns**
> - Так как горутина привязана к P, а mcache привязан к P — concurrency решается без lock
>
> **mcentral** (per-size-class):
> - Глобальный пул spans для каждого size class
> - Когда mcache исчерпал span → берёт новый из mcentral (нужен **лок**, но короткий)
> - Содержит два списка: spans со свободными объектами и полные spans
> - Один mcentral на каждый из 67 size classes → lock contention минимален
>
> **mheap** (глобальный):
> - Управляет всей виртуальной памятью процесса
> - Аллоцирует большие куски (spans, arena) у ОС через `mmap`
> - Когда mcentral нужен новый span → mheap выделяет страницы
> - Один глобальный лок — но обращения редки (bulk allocation)

> [!question]- 2. Что такое size classes? Зачем 67 классов?
>
> Go делит heap-объекты на **67 фиксированных size classes** (от 8 до 32768 байт):
>
> ```
> 8, 16, 24, 32, 48, 64, 80, 96, 112, 128, 144, 160, ...
> ..., 4096, 5376, 6144, 6784, 8192, ...
> ..., 24576, 27264, 28672, 32768
> ```
>
> **Зачем:**
> - Объект запрашивает N байт → округляется вверх до ближайшего size class
> - Объект 20 байт → класс 24 байта (4 байта waste — **internal fragmentation**)
> - Каждый mspan хранит объекты **только одного size class** → нет external fragmentation
> - Аллокация = найти свободный слот в span нужного класса → **O(1)** через bitmap
> - Средний waste — ~12%, но это окупается скоростью и отсутствием фрагментации
>
> **Почему не произвольные размеры:**
> - Произвольный malloc → фрагментация, сложный free list, slow allocation
> - Фиксированные классы → bitmap allocator, нет фрагментации, fast path без лока

> [!question]- 3. Опиши три пути аллокации: tiny, small, large.
>
> ```
>                         size?
>                      /    |     \
>                ≤16B    17B-32KB   >32KB
>                 |         |        |
>           Tiny alloc   Small     Large
>           (без ptr)    mcache    mheap напрямую
> ```
>
> **1. Tiny allocator (≤ 16 байт, без указателей):**
> - Упаковывает несколько мелких объектов в **один 16-байтовый блок**
> - Только для объектов **без указателей** (иначе GC не сможет корректно сканировать)
> - `bool`, `byte`, мелкие `int`, `rune` → часто попадают сюда
> - Экономит ~12% аллокаций в типичной программе
> - Внутри mcache хранится текущий tiny block и offset в нём
>
> ```go
> // Эти 3 переменные могут оказаться в одном 16-байтовом блоке:
> a := byte(1)   // 1 байт
> b := int8(2)   // 1 байт
> c := uint16(3) // 2 байта
> // Итого 4 байта → всё в одном tiny block
> ```
>
> **2. Small allocations (17 байт — 32 KB):**
> - Определяется size class → берётся свободный слот из mspan в mcache
> - Если mspan пуст → refill из mcentral → если mcentral пуст → из mheap
> - **Основной путь** — большинство аллокаций идут сюда
> - Lock-free на уровне mcache (per-P) → быстро
>
> **3. Large allocations (> 32 KB):**
> - Напрямую из **mheap** → аллоцируется нужное количество страниц (8KB per page)
> - Нужен глобальный лок (но large allocs редки)
> - Получает собственный mspan с `sizeClass=0`
> - Примеры: большие slice, большие буферы, image data

> [!question]- 4. Что такое mspan? Как устроен?
>
> **mspan** — единица управления памятью. Непрерывный блок страниц, разбитый на объекты одного size class.
>
> ```
> mspan:
> ┌──────────────────────────────────────────────┐
> │ startAddr  uintptr  — начало блока памяти    │
> │ npages     uintptr  — кол-во страниц (8KB)   │
> │ sizeClass  uint8    — класс размера           │
> │ freeIndex  uintptr  — следующий свободный слот│
> │ allocBits  *bitmap  — какие слоты заняты      │
> │ gcmarkBits *bitmap  — какие помечены GC       │
> │ nelems     uintptr  — всего объектов в span   │
> └──────────────────────────────────────────────┘
> ```
>
> **Пример:** sizeClass=5 (48 байт), npages=1 (8KB):
> ```
> 8192 / 48 = 170 объектов в одном span
>
> [ obj0 | obj1 | obj2 | ... | obj169 ]
>   48B    48B    48B           48B
> ```
>
> **Как работает аллокация внутри mspan:**
> 1. Смотрим `allocBits` начиная с `freeIndex`
> 2. Находим первый нулевой бит → свободный слот
> 3. Ставим бит → объект аллоцирован
> 4. **O(1)** амортизированно (bitmap scan)
>
> **Sweep:** при GC `gcmarkBits` показывает живые объекты. После sweep:
> - `allocBits = gcmarkBits` (живые остаются allocated)
> - `gcmarkBits` обнуляется для следующего цикла
> - Неотмеченные слоты → свободны для новых аллокаций

> [!question]- 5. Детали escape analysis — все причины escape на heap.
>
> Компилятор решает stack vs heap на этапе компиляции. **Если компилятор не может доказать, что объект не переживёт функцию → heap.**
>
> **Причины escape:**
>
> ```go
> // 1. Указатель возвращается из функции
> func new() *T { t := T{}; return &t }  // &t escapes
>
> // 2. Присвоение в поле, которое живёт дольше
> type S struct { p *int }
> func f() S { x := 42; return S{p: &x} }  // &x escapes
>
> // 3. Передача в interface (boxing)
> fmt.Println(42)  // 42 боксируется: interface{_type, data→heap}
>
> // 4. Горутина захватывает переменную
> func f() { x := 0; go func() { x++ }() }  // x escapes
>
> // 5. Слишком большой объект для stack frame
> func f() { _ = make([]byte, 1<<20) }  // 1MB
>
> // 6. Размер неизвестен compile-time
> func f(n int) { _ = make([]byte, n) }  // n — variable
>
> // 7. Slice append может вырастить backing array
> s := make([]int, 0)
> s = append(s, 1)  // может аллоцировать на heap
>
> // 8. Запись указателя в heap-объект
> var global *int
> func f() { x := 42; global = &x }  // x escapes
>
> // 9. Reflection (reflect.ValueOf и т.д.)
> reflect.ValueOf(myStruct)  // escapes
> ```
>
> **Что НЕ escapes (остаётся на stack):**
> - Локальные переменные, используемые только внутри функции
> - Передача by value в другие функции (копия)
> - Мелкие структуры, возвращаемые by value
> - `make([]byte, 64)` с константным размером (может остаться на стеке)
>
> ```bash
> # Два уровня детализации:
> go build -gcflags="-m" ./...     # базовый
> go build -gcflags="-m -m" ./...  # подробный (показывает причину escape)
> ```

> [!question]- 6. Как связан аллокатор с GC?
>
> **Аллокатор и GC тесно интегрированы:**
>
> **1. GC trigger через аллокатор:**
> - Аллокатор отслеживает `heap_live` (объём живых данных на heap)
> - Когда `heap_live` вырос на `GOGC`% → запускается GC
> - `GOMEMLIMIT` добавляет soft cap — GC агрессивнее при приближении
>
> **2. Sweep интегрирован в аллокацию:**
> - Sweep lazy — происходит **в момент аллокации**, а не отдельной фазой
> - Когда горутина просит mspan, она сначала sweep-ит его (если нужно)
> - Это распределяет sweep cost по аллокациям, а не создаёт паузу
>
> **3. Bitmap связь:**
> - `allocBits` — что занято (аллокатор)
> - `gcmarkBits` — что live (GC)
> - После mark phase: `allocBits = gcmarkBits` → освобождение без обхода
>
> **4. Write barrier:**
> - Во время marking, аллокатор **помечает новые объекты как black**
> - Иначе GC мог бы собрать только что аллоцированный объект
>
> **5. mcache ↔ GC:**
> - При STW фазе GC нужно "увидеть" все mcache
> - mcache per-P → GC итерирует по всем P для учёта

> [!question]- 7. Как снизить аллокации на практике? Приёмы оптимизации.
>
> **1. Pre-allocate slices/maps:**
> ```go
> // Плохо — 10+ аллокаций при росте
> s := []int{}
> for i := 0; i < 10000; i++ { s = append(s, i) }
>
> // Хорошо — 1 аллокация
> s := make([]int, 0, 10000)
> for i := 0; i < 10000; i++ { s = append(s, i) }
>
> // Maps: m := make(map[K]V, expectedSize)
> ```
>
> **2. sync.Pool для горячих объектов:**
> ```go
> var bufPool = sync.Pool{
>     New: func() any { return new(bytes.Buffer) },
> }
> func process() {
>     buf := bufPool.Get().(*bytes.Buffer)
>     buf.Reset()
>     defer bufPool.Put(buf)
> }
> ```
>
> **3. Избегать interface boxing в hot path:**
> ```go
> // Плохо — boxing int → interface → heap alloc
> fmt.Println(counter)
> // Лучше
> fmt.Fprintf(os.Stdout, "%d\n", counter)
> ```
>
> **4. Value receivers для мелких структур (≤ 2-3 слова):**
> ```go
> type Point struct { X, Y float64 }
> func (p Point) Distance() float64 { ... }  // копия на стеке, нет escape
> ```
>
> **5. strings.Builder вместо конкатенации:**
> ```go
> var b strings.Builder
> b.Grow(estimatedSize)
> for _, v := range items { b.WriteString(v.String()) }
> ```
>
> **6. Reuse buffers / byte slices:**
> ```go
> buf := make([]byte, 0, 4096)
> for _, req := range requests {
>     buf = buf[:0]  // reset без аллокации
>     buf = append(buf, req.Data...)
>     process(buf)
> }
> ```
>
> **7. Avoid unnecessary pointers:**
> ```go
> // Плохо — каждый элемент может быть на heap отдельно
> type Items struct { list []*Item }
>
> // Лучше — один contiguous block
> type Items struct { list []Item }
> ```

> [!question]- 8. Как профилировать аллокации?
>
> **Benchmark с подсчётом:**
> ```bash
> go test -bench=. -benchmem
> # BenchmarkFoo  1000000  1024 ns/op  256 B/op  3 allocs/op
> #                                     ^^^^^^^^  ^^^^^^^^^^^^
> #                                     байт/оп   аллокаций/оп
> ```
>
> **pprof — heap profile:**
> ```bash
> go test -bench=. -memprofile=mem.prof
> go tool pprof -alloc_objects mem.prof  # по кол-ву аллокаций
> go tool pprof -alloc_space mem.prof    # по объёму памяти
> go tool pprof -inuse_space mem.prof    # что живёт прямо сейчас
>
> # В pprof:
> # (pprof) top10          — топ функций
> # (pprof) list funcName  — построчно
> # (pprof) web            — граф в браузере
> ```
>
> **Runtime в коде:**
> ```go
> var m runtime.MemStats
> runtime.ReadMemStats(&m)
> fmt.Printf("HeapAlloc: %d MB\n", m.HeapAlloc/1024/1024)
> fmt.Printf("TotalAlloc: %d MB\n", m.TotalAlloc/1024/1024)  // cumulative
> fmt.Printf("NumGC: %d\n", m.NumGC)
> fmt.Printf("Mallocs: %d, Frees: %d\n", m.Mallocs, m.Frees)
> // Mallocs - Frees = live objects
> ```
>
> **Escape analysis (compile-time):**
> ```bash
> go build -gcflags="-m -m" ./...
> ```
>
> **Execution tracer (runtime):**
> ```bash
> curl -o trace.out http://localhost:6060/debug/pprof/trace?seconds=5
> go tool trace trace.out
> # Показывает: аллокации, GC паузы, goroutine scheduling
> ```

> [!question]- 9. Что такое arena allocator? (Go 1.20 experimental)
>
> `arena` — экспериментальный пакет (GOEXPERIMENT=arenas) для **ручного управления памятью**:
>
> - Аллоцируешь объекты в arena → все освобождаются **разом** при `arena.Free()`
> - Не ждём GC — мгновенное освобождение
> - Полезно для request-scoped данных: protobuf decode, JSON parse
>
> ```go
> import "arena"
>
> a := arena.NewArena()
> defer a.Free()  // всё в arena освобождается разом
>
> // Аллокация в arena
> slice := arena.MakeSlice[MyStruct](a, 0, 1000)
> obj := arena.New[MyStruct](a)
> ```
>
> **Плюсы:** нет GC pressure, мгновенный free, locality
> **Минусы:** experimental, ручной lifetime management, use-after-free возможен (panic at runtime)
>
> **Статус:** experimental, API может измениться. Не для production без явной необходимости.

> [!question]- 10. Как аллокатор Go сравнивается с malloc/jemalloc/TCMalloc?
>
> | | Go allocator | glibc malloc | TCMalloc | jemalloc |
> |--|-------------|-------------|----------|----------|
> | Thread-local cache | per-P mcache | нет (global lock) | per-thread cache | per-thread cache |
> | Size classes | 67 | bins | ~88 | ~200 |
> | Small alloc lock | **нет** (per-P) | global/arena lock | per-thread, нет | per-thread, нет |
> | GC integration | встроен | нет (manual free) | нет | нет |
> | Fragmentation | low (size classes) | medium | low | low |
> | Tiny optimization | ≤16B packing | нет | нет | нет |
>
> **Ключевое преимущество Go:** аллокатор и GC — одна система. `allocBits`/`gcmarkBits` в mspan делают sweep O(1) per span. В C/C++ аллокатор и memory management — разные системы.
>
> **Ключевой недостаток:** нет ручного free → зависимость от GC latency.
