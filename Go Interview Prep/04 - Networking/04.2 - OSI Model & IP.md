> [!question]- 2. IPv4 vs IPv6: сколько байт весит каждый, почему и как ты это понял?
>
> **IPv4: 32 бита = 4 байта**
> - Формат: 4 октета по 8 бит: `192.168.1.1`
> - Каждый октет: 0-255 (2^8 = 256 значений)
> - Всего: 2^32 ≈ 4.3 млрд адресов — **исчерпано в 2011 году**
>
> **IPv6: 128 бит = 16 байт** (в 4 раза больше)
> - Формат: 8 групп по 16 бит (4 hex символа): `2001:0db8::8a2e:0370:7334`
> - Всего: 2^128 ≈ 340 ундециллионов адресов
> - `::` = один или несколько блоков нулей (сокращение)
>
> **Мнемоника:**
> - IPv4 = 4 числа × 8 бит = 32 бит = **4 байта**
> - IPv6 = 8 групп × 16 бит = 128 бит = **16 байт**
>
> **Почему 128 бит?** Не 64, потому что нужно пространство для:
> - 64 бита = Network prefix (routing)
> - 64 бита = Interface ID (host)

> [!warning]- Типичная ошибка
> "IPv6 в 4 раза больше адресов чем IPv4"
> **Неправильно!** 2^128 / 2^32 = 2^96 ≈ 7.9 × 10^28 раз больше.
> Не "в 4 раза больше", а в **79 октиллионов** раз больше.

> [!question]- 3. TCP vs UDP — когда что?
>
> | | TCP | UDP |
> |---|-----|-----|
> | Connection | Connection-oriented (handshake) | Connectionless |
> | Reliability | Guaranteed delivery, ordering | No guarantees |
> | Flow control | Yes (sliding window) | No |
> | Overhead | Higher (headers, ACKs, retransmit) | Lower |
> | Speed | Slower (reliability overhead) | Faster |
>
> **TCP когда:** HTTP, database connections, file transfer, email — нужна надёжность.
> **UDP когда:** DNS, video streaming, gaming, VoIP — нужна скорость, потеря ok.
> **QUIC:** UDP + reliability + encryption (HTTP/3) — best of both worlds.

> [!question]- 4. Как работает NAT? Зачем нужен?
>
> **NAT** (Network Address Translation) — преобразование IP адресов на маршрутизаторе.
>
> **Зачем:** IPv4 адресов не хватает. NAT позволяет множеству устройств в LAN использовать один public IP.
>
> **Как работает:**
> ```
> Private (192.168.1.10:5000) → Router NAT → Public (203.0.113.1:40000) → Internet
>
> Router хранит mapping:
> 192.168.1.10:5000 ↔ 203.0.113.1:40000
> 192.168.1.11:5001 ↔ 203.0.113.1:40001
> ```
>
> **Проблемы NAT:**
> - Peer-to-peer затруднён (оба за NAT)
> - Некоторые протоколы ломаются (FTP active mode)
> - Решения: STUN, TURN, ICE (WebRTC)

> [!question]- 5. Что такое TCP congestion control? Объясни основные алгоритмы.
>
> **Цель:** не перегрузить сеть отправкой слишком большого количества данных.
>
> **Congestion Window (cwnd):** максимум данных "в полёте" (отправлено но не подтверждено).
>
> **Алгоритмы:**
> 1. **Slow Start:** cwnd начинает с 1 MSS, удваивается каждый RTT
> 2. **Congestion Avoidance:** после threshold, cwnd растёт линейно (+1 MSS за RTT)
> 3. **Fast Retransmit:** 3 duplicate ACKs → retransmit без ожидания timeout
> 4. **Fast Recovery:** не сбрасывать cwnd до 1, а halve it
>
> **Современные:** CUBIC (Linux default), BBR (Google — bandwidth-based, не loss-based).

> [!question]- 6. WebSocket vs SSE vs Long Polling — когда что?
>
> | | WebSocket | SSE | Long Polling |
> |---|-----------|-----|-------------|
> | Direction | Bidirectional | Server → Client | Client pulls |
> | Protocol | ws:// (upgrade from HTTP) | HTTP | HTTP |
> | Complexity | Medium | Low | Low |
> | Reconnect | Manual | Automatic | Built-in |
> | Binary data | Yes | No (text only) | Yes |
>
> **WebSocket:** chat, gaming, real-time collaboration
> **SSE:** live feeds, notifications, stock prices (server-push only)
> **Long Polling:** fallback when others unavailable, simple notifications

> [!question]- 7. Что такое CORS? Зачем нужен и как настроить?
>
> **CORS** (Cross-Origin Resource Sharing) — механизм безопасности браузера.
>
> **Same-origin policy:** `https://app.com` не может делать запросы к `https://api.com` (другой origin).
>
> **CORS headers позволяют серверу разрешить cross-origin:**
> ```
> Access-Control-Allow-Origin: https://app.com
> Access-Control-Allow-Methods: GET, POST, PUT
> Access-Control-Allow-Headers: Authorization, Content-Type
> Access-Control-Allow-Credentials: true
> Access-Control-Max-Age: 3600
> ```
>
> **Preflight request (OPTIONS):** браузер отправляет перед "сложными" запросами (не-GET, custom headers).
>
> ```go
> func corsMiddleware(next http.Handler) http.Handler {
>     return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
>         w.Header().Set("Access-Control-Allow-Origin", "https://app.com")
>         w.Header().Set("Access-Control-Allow-Methods", "GET, POST, PUT, DELETE")
>         w.Header().Set("Access-Control-Allow-Headers", "Authorization, Content-Type")
>         if r.Method == "OPTIONS" {
>             w.WriteHeader(http.StatusOK)
>             return
>         }
>         next.ServeHTTP(w, r)
>     })
> }
> ```

> [!question]- 8. Как работает Keep-Alive в HTTP? Зачем?
>
> **Без Keep-Alive (HTTP/1.0):** каждый запрос = новое TCP соединение (3-way handshake).
>
> **С Keep-Alive (HTTP/1.1 default):** одно TCP соединение для множества запросов:
> ```
> Connection: keep-alive
> Keep-Alive: timeout=5, max=100
> ```
>
> **Преимущества:**
> - Экономия на TCP handshake (~1 RTT per connection)
> - Экономия на TLS handshake (~1-2 RTT per connection)
> - TCP congestion window уже "прогрет"
>
> **В Go `http.Client`:** Keep-Alive включён по умолчанию.
> ```go
> client := &http.Client{
>     Transport: &http.Transport{
>         MaxIdleConns:        100,
>         MaxIdleConnsPerHost: 10,
>         IdleConnTimeout:     90 * time.Second,
>     },
> }
> ```

> [!warning]- Частая ошибка в Go
> ```go
> // ПЛОХО: body не закрыт → connection не возвращается в pool
> resp, _ := http.Get(url)
> // используем resp.Body но не закрываем
>
> // ПРАВИЛЬНО:
> resp, err := http.Get(url)
> if err != nil { return err }
> defer resp.Body.Close()
> io.ReadAll(resp.Body) // или io.Copy(io.Discard, resp.Body)
> ```
