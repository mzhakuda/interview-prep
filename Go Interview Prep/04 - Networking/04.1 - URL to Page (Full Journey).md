# URL to Page (Full Journey)

Связанные темы: [[04.2 - OSI Model & IP]] | [[03.1 - API Gateway Patterns]]

---

> [!question]- 1. Опиши полный путь от ввода youtube.com в браузере до отображения страницы. Укажи уровень OSI для каждого шага.
>
> **Шаг 1: Browser Cache Check** (Application)
> - Браузер проверяет свой кэш: есть ли IP для youtube.com?
> - Если да и TTL не истёк → сразу к шагу 3
>
> **Шаг 2: DNS Resolution** (Layer 7 → Layer 3)
> ```
> Browser → OS resolver cache → /etc/hosts
>        → Recursive DNS resolver (ISP / 8.8.8.8)
>          → Root nameserver (.) → "go ask .com TLD"
>            → .com TLD nameserver → "go ask ns1.google.com"
>              → Authoritative nameserver (ns1.google.com) → IP: 142.250.x.x
> ```
> - Результат кэшируется на TTL (обычно 300s)
> - DNS использует UDP port 53 (TCP для большИх ответов / zone transfers)
>
> **Шаг 3: TCP Handshake** (Layer 4 — Transport)
> ```
> Client → SYN (seq=x)            → Server
> Client ← SYN-ACK (seq=y, ack=x+1) ← Server
> Client → ACK (ack=y+1)           → Server
> ```
> - 1 RTT (round-trip time) для установки соединения
> - TCP socket: (src_ip, src_port, dst_ip, dst_port)
>
> **Шаг 4: TLS Handshake** (Layer 5-6 — Session/Presentation)
> ```
> Client → ClientHello (supported ciphers, TLS version, random)
> Server → ServerHello (chosen cipher) + Certificate + ServerKeyExchange
> Client → Verify certificate chain (CA trust)
>        → ClientKeyExchange (pre-master secret encrypted with server's public key)
> Both   → Derive session keys from pre-master secret
> Client → Finished (encrypted)
> Server → Finished (encrypted)
> ```
> - TLS 1.3: 1 RTT (vs 2 RTT в TLS 1.2)
> - TLS 1.3 0-RTT: resumption при повторном подключении
>
> **Шаг 5: HTTP/2 Request** (Layer 7 — Application)
> ```
> GET / HTTP/2
> Host: youtube.com
> Accept: text/html
> Accept-Encoding: gzip, br
> Cookie: session_id=abc123
> ```
> - HTTP/2: multiplexing (множество requests по одному TCP соединению)
> - HPACK header compression
>
> **Шаг 6: Load Balancer** (Layer 4/7)
> - Распределяет запрос по backend серверам
> - Может быть L4 (TCP) или L7 (HTTP) балансировщик
> - Алгоритмы: round-robin, least-connections, IP-hash
>
> **Шаг 7: API Gateway / Web Server**
> - Authentication (JWT/session validation)
> - Rate limiting check
> - Routing к нужному микросервису
>
> **Шаг 8: Application Server → Response**
> - Backend обрабатывает запрос
> - Query к database (PostgreSQL, Spanner)
> - Формирует HTML response
> - HTTP 200 OK + HTML body (gzip compressed)
>
> **Шаг 9: Browser Rendering** (Client)
> ```
> HTML bytes → Characters → Tokens → Nodes → DOM tree
> CSS bytes  → Characters → Tokens → Nodes → CSSOM tree
>
> DOM + CSSOM → Render Tree → Layout (positions) → Paint (pixels) → Composite
> ```
> - JavaScript может блокировать parsing (defer/async атрибуты)
> - Critical rendering path: HTML → CSS → first paint
> - Lazy loading для images/iframes ниже fold

> [!hint]- Ключевые цифры для интервью
> - DNS lookup: ~20-120ms
> - TCP handshake: ~1 RTT (~20-100ms)
> - TLS handshake: ~1-2 RTT (~50-200ms)
> - HTTP request + response: ~100-500ms
> - Total: ~200-1000ms для первого визита

> [!question]- 2. Что такое HTTPS? Как TLS обеспечивает безопасность?
>
> HTTPS = HTTP over TLS. Обеспечивает:
>
> **1. Confidentiality** — шифрование данных:
> - Symmetric encryption (AES-256-GCM) для данных
> - Asymmetric encryption (ECDHE) для обмена ключами
>
> **2. Integrity** — данные не модифицированы:
> - HMAC (Hash-based Message Authentication Code)
> - Каждый пакет содержит MAC
>
> **3. Authentication** — сервер тот, за кого себя выдаёт:
> - X.509 сертификат подписан Certificate Authority
> - Browser проверяет chain of trust до root CA

> [!question]- 3. HTTP/1.1 vs HTTP/2 vs HTTP/3 — ключевые отличия.
>
> | | HTTP/1.1 | HTTP/2 | HTTP/3 |
> |---|---------|--------|--------|
> | Transport | TCP | TCP | **QUIC (UDP)** |
> | Multiplexing | Нет (1 req/conn) | Да (streams) | Да (streams) |
> | Head-of-line blocking | TCP + HTTP | TCP only | **Нет** |
> | Header compression | Нет | HPACK | QPACK |
> | Server Push | Нет | Да | Нет (deprecated) |
> | Connection setup | TCP + TLS = 3 RTT | TCP + TLS = 2 RTT | **1 RTT** (0-RTT resume) |
>
> **HTTP/3 + QUIC:**
> - Встроенное шифрование (TLS 1.3 в protocol)
> - Connection migration (смена IP/порта без reconnect — мобильные сети)
> - Потеря пакета в одном stream не блокирует другие

> [!question]- 4. Что такое CDN? Как работает?
>
> **CDN** (Content Delivery Network) — сеть edge серверов по всему миру.
>
> **Принцип:**
> 1. Пользователь запрашивает `youtube.com/video.mp4`
> 2. DNS возвращает IP ближайшего edge сервера
> 3. Edge сервер проверяет cache
>    - Hit → отдаёт сразу (минимальная latency)
>    - Miss → запрашивает origin → кэширует → отдаёт
>
> **Что кэшируется:**
> - Static: images, CSS, JS, fonts, videos
> - Dynamic: API responses (с short TTL)
>
> **Cache invalidation:**
> - TTL expiry (Cache-Control: max-age=3600)
> - Purge by URL / tag
> - Versioned URLs: `style.v2.css`

> [!question]- 5. DNS записи — какие бывают?
>
> | Тип | Назначение | Пример |
> |-----|-----------|--------|
> | **A** | Domain → IPv4 | `example.com → 93.184.216.34` |
> | **AAAA** | Domain → IPv6 | `example.com → 2606:2800:220:1::` |
> | **CNAME** | Alias → другой domain | `www.example.com → example.com` |
> | **MX** | Mail server | `example.com → mail.example.com` |
> | **TXT** | Текстовые записи | SPF, DKIM, domain verification |
> | **NS** | Nameserver | `example.com → ns1.example.com` |
> | **SRV** | Service discovery | `_http._tcp.example.com → ...` |
>
> **DNS propagation:** изменения распространяются по сети за минуты-часы (зависит от TTL).
