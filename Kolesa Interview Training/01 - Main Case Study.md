# Главный кейс — PDF streaming в BCC Business

> Используется, когда просят "расскажи про самый результативный проект / самое сложное техническое решение / гордость".
>
> Все метрики — твои личные, не команда. Это было **моё** решение, **я** профилировал, **я** переписывал протокол.

---

## Короткая версия (30 секунд — elevator pitch)

> В BCC Business у нас был PDF Processor — gRPC-микросервис для генерации документов escrow-сделок. Он грузил весь PDF в память целиком, и под нагрузкой поды уходили в OOM. Я профилировал через pprof, переписал протокол на gRPC bidirectional streaming с zero-copy через io.Pipe — чанки по 64KB. В итоге потребление памяти упало с 72MB до 16MB на запрос, пиковая аллокация — с 61MB до 3.5MB. Сервис перестал падать, и этот подход переиспользовали в других сервисах.

---

## Длинная версия (2-3 минуты — STAR)

### Situation
BCC Business — корпоративный сегмент Bank CenterCredit, это SMB-экосистема внутри одного из крупнейших банков Казахстана (3M+ клиентов всего). Я бэкенд-разработчик в команде из 4 человек, владею разработкой в **7 микросервисах**: escrow, депозиты, документооборот, инкассация, контроль доступа, PDF processing, platform boilerplate.

PDF Processor — отдельный **gRPC-микросервис**, который собирает документы для escrow-сделок: договоры, акты, спецификации. Обслуживает **5,000+ бизнес-клиентов**. Реализация: клиент (escrow-сервис) делает unary-запрос, сервер генерирует PDF полностью в `bytes.Buffer`, возвращает целым blob.

### Task
Проблема в том, что один escrow-запрос триггерит **параллельную генерацию нескольких PDF** (иногда 4-5 штук: договор + приложения). Под нагрузкой сервис начал регулярно OOM-иться. Лимит памяти на под в Kubernetes — жёсткий, увеличивать нельзя (SRE команда политически заблокировала). Нужно было уменьшить потребление памяти без изменения интерфейса для всех потребителей.

### Action
**1. Профилирование.** Снял heap profile через `pprof` — увидел что основной аллокатор это копирование в `bytes.Buffer` при сборке финального PDF. То есть файл проходил через память **минимум дважды**: собирался в память, потом копировался в gRPC response.

**2. Redesign протокола.** Переделал unary RPC → bidirectional streaming:
```proto
rpc GeneratePDF(stream PDFRequest) returns (stream PDFChunk);
message PDFChunk {
  oneof payload {
    FileMetadata metadata = 1;  // имя, размер, MIME
    bytes data = 2;             // чанк 64KB
    Trailer trailer = 3;        // checksum
  }
}
```
Один stream может передавать несколько файлов подряд — с metadata-header'ом и trailer-ом между ними.

**3. Zero-copy через io.Pipe.** На сервере:
- В goroutine запускаю PDF generator → пишет в `io.PipeWriter`
- Основная goroutine читает из `io.PipeReader` чанками по 64KB и шлёт в gRPC stream
- **Весь файл никогда не лежит в памяти целиком** — только один чанк в buffer'е pipe

**4. Адаптер на клиенте.** Чтобы не ломать существующий код, обернул gRPC stream в `io.Reader`:
```go
type streamReader struct { stream pb.PDF_GenerateClient; buf []byte }
func (r *streamReader) Read(p []byte) (int, error) { /* подтягивает следующий chunk */ }
```
Клиентский код продолжал работать с обычным `io.Reader`, не знал про streaming.

**5. Backpressure.** gRPC bidi stream сам даёт flow control — если клиент медленный, send блокируется, генератор перестаёт писать в pipe. Никаких accumulating buffers.

### Result
Сняли heap profile после релиза:
- **Аллокации на запрос:** 72MB → 16MB (−78%)
- **Пиковая аллокация:** 61.55MB → 3.55MB (−94%)
- **OOM-ы ушли в ноль** — за 3 месяца ни одного инцидента
- Подход **переиспользовали** в deposit-сервисе для скачивания выписок (раньше base64 в памяти → теперь streaming)
- Kubernetes request/limit по памяти оставили прежними, не пришлось торговаться с SRE

## Заготовки под follow-up вопросы

**"А почему именно gRPC streaming, а не S3/presigned URL?"**
> Подумали и про это. Для генерируемых документов presigned URL потребовал бы промежуточного S3 upload, а у нас SLA на latency жёсткий — документ должен быть клиенту через ~1-2 секунды. Плюс документы разовые, их не нужно хранить долго. gRPC streaming даёт лучший latency и меньше двигаемых частей.

**"А если клиент отвалится посередине стрима?"**
> Контекст gRPC cancel'ится, pipe closes, генератор получает `io.ErrClosedPipe` на первом Write и останавливается. Нет утечек.

**"Как тестировал?"**
> Unit-тесты на адаптер `io.Reader` через `bufconn` (in-memory gRPC). Плюс нагрузочный тест через `ghz` — 100 параллельных стримов, смотрел pprof heap в k6-style прогоне. Для самой генерации — table-driven с golden files.

**"Почему chunk 64KB?"**
> Пробовали 16/32/64/128KB. 64KB — sweet spot: gRPC frame overhead размазывается, но мы не забиваем CPU на syscalls. Это ~ размер страницы × 16, ложится в L2 cache.

**"Что бы сделал иначе?"**
> Добавил бы retry с resumable upload по offset'у (сейчас при падении — полный re-generate). И метрику chunk throughput, чтобы видеть хвост латенси.
